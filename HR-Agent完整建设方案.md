## 一、战略定位

### 1.1 核心结论

> **HR Agent 不是替代 HR Portal 的新产品，而是 HR Portal 的"智能交互层"。用户可以通过两种方式与 Agent 交互：① 飞书机器人（轻量触达，随时随地）② Web 工作台（深度分析，完整体验）。两者共享同一套能力底座和数据，通过"飞书预览 + 链接跳转"实现无缝协同。**

> **最关键的设计原则：权限管理贯穿始终，从第一个版本开始就是最高优先级。没有权限管控，Agent 再聪明也不能上线。**

> **数据安全是底线：原始数据不出 HR Portal 服务器，AI 只接收经过权限过滤和脱敏后的数据摘要。**

> **Skill 编排是 Agent 能力的核心载体：原子 Skill 是"乐高积木"，工作流编排是"设计图"。从第一天就预留编排能力，具体 Skill 内容逐步填充。**

### 1.2 双入口协同战略

| 入口 | 定位 | 适用场景 | 核心能力 | 输出形式 |
|:---|:---|:---|:---|:---|
| **飞书机器人** | 轻量触达入口 | 快速问答、即时查询、接收预警、消息通知 | 指标查询、简单分析、摘要推送 + 超链接 | 文本 + 表格 + 图表预览 + 跳转链接 |
| **Web 工作台** | 深度消费入口 | 复杂分析、图表交互、归因报告、下钻探索 | 全量能力（图表、下钻、报告、编排） | 完整图表 + 交互操作 + 报告 + 协作分享 |

### 1.3 数据安全的核心地位

> **在 HR 系统中，数据安全不是"功能"，而是"底线"。原始数据绝对不能暴露给 AI 模型本身。**

| 安全维度 | 说明 | 在 Agent 中的体现 |
|:---|:---|:---|
| **数据不出域** | 原始数据不离开 HR Portal 服务器 | 所有数据查询、过滤、脱敏在 HR Portal 后端完成 |
| **脱敏处理** | 发送给 AI 的数据自动脱敏 | 姓名→工号、薪酬→等级范围，AI 看不到具体敏感值 |
| **权限强制隔离** | 行/列权限在数据查询层强制执行 | AI 无法获取用户无权访问的数据 |
| **不用于训练** | Claude API 明确承诺不用于训练 | 合同保障 |
| **全链路审计** | 所有数据访问和 AI 调用记录在案 | 可追溯、可审计 |

### 1.4 权限管理的核心地位

> **在 HR 系统中，权限不是"功能"，而是"底线"。没有权限管控，数据安全、合规审计、隐私保护全部无从谈起。**

| 权限维度 | 说明 | 在 Agent 中的体现 |
|:---|:---|:---|
| **身份认证** | 确认"你是谁" | 飞书用户与 HR Portal 用户身份映射 |
| **行级权限** | 控制"能看哪些行" | 不同部门的 HRBP 看到不同部门的数据 |
| **列级权限** | 控制"能看哪些列" | 普通 HR 看不到薪酬列，HRVP 可以看到 |
| **功能权限** | 控制"能做什么操作" | 普通用户只能查询，管理员可以修改指标口径 |
| **数据脱敏** | 敏感数据展示控制 | 飞书消息中敏感数据自动脱敏 |

### 1.5 飞书 + Web 协同模式

> **飞书做"通知+预览"，Web 做"深度交互"**

```text
飞书机器人                              Web 工作台
┌─────────────────────┐                ┌─────────────────────┐
│ ① 快速问答           │                │ ⑤ 完整上下文恢复     │
│   文本 + 表格 + 预览 │                │   图表 + 下钻 + 编辑 │
│ ↓                   │                │ ↑                   │
│ ② 图表预览卡片       │ ── 点击链接 ─▶ │ ⑥ 深度交互分析       │
│   "查看完整图表"     │                │   归因报告 + 下载    │
│ ↓                   │                │ ↑                   │
│ ③ 分析摘要推送       │ ── 点击链接 ─▶ │ ⑦ 完整报告查看       │
│   "查看完整报告"     │                │   编辑 + 分享        │
│ ↓                   │                │ ↑                   │
│ ④ 预警消息推送       │ ── 点击链接 ─▶ │ ⑧ 预警详情查看       │
│   "查看详情"         │                │   趋势分析 + 建议    │
└─────────────────────┘                └─────────────────────┘
```

### 1.6 与通用 Agent 的本质区别

| 对比维度 | 通用 Agent（Cursor/Claude Code） | HR Agent（你的） |
|:---|:---|:---|
| **核心任务** | 写代码、处理通用任务 | 查询 HR 数据、分析离职率、生成报告 |
| **数据来源** | 公开数据 + 用户提供 | **HR Portal 七层数仓（可信数据）** |
| **知识基础** | 通用知识 | **知识库 + 指标口径 + 分析方法论** |
| **权限边界** | 无 | **行/列权限贯穿始终，飞书用户身份自动映射** |
| **数据安全** | 数据可能被用于训练 | **数据不出域 + 脱敏处理，不用于训练** |
| **入口方式** | 单一工作台 | **飞书机器人 + Web 工作台双入口协同** |
| **图表能力** | 无 | **自动生成可交互图表 + 多格式下载 + 下钻** |
| **能力组织** | 零散工具 | **Skill 编排 + DAG 工作流，可复用可组合** |
| **进化方式** | 模型升级 | **从第一天开始的反馈闭环 + 知识沉淀 + Skill 沉淀** |

### 1.7 与 AI 行业趋势的呼应

头部 AI Agent 产品（Cursor、Claude Code、Codex）不约而同采用三栏布局作为标准前端架构，这已成为行业共识。同时，Agent 的核心数据模型（Thread/Run/Artifact/Tool Invocation）和 Skill 编排机制已成为业界标准架构，这些理念将贯穿整个 HR Agent 设计方案。

## 二、核心设计原则

### 2.1 十大原则

| 原则 | 说明 |
|:---|:---|
| **权限贯穿始终** | 从第一个版本开始，每个迭代都必须包含权限管理内容。没有权限，不上线 |
| **数据不出域** | 原始数据不离开 HR Portal 服务器，AI 只接收脱敏后的数据摘要 |
| **底座先行** | Agent 的智能程度，取决于底座的完善程度。数仓、指标、权限是 Agent 能正确工作的前提 |
| **双入口协同** | 飞书机器人和 Web 工作台同步建设，共享同一套后端能力，通过超链接实现无缝协同 |
| **Skill 驱动** | 能力封装为 Skill，原子 Skill 可组合为工作流，从第一天预留编排能力 |
| **极小迭代** | 每个版本 1-2 天，快速验证，用户可感知 |
| **资产持续复用** | 早期建设的东西在后期继续使用，不推翻重来 |
| **渐进式增强** | 功能从简单到复杂，自然演进（原子 Skill → 组合 → 工作流） |
| **底层支撑远期** | 早期架构预留扩展点（Skill 注册表、DAG 引擎），但不提前实现复杂编排 |
| **自进化从第一天启动** | 反馈闭环不是三期才做的事，而是从第一个用户开始 |

### 2.2 演进哲学：从 A 到 B

| 场景 | MVP（A） | 后期（B） | 演进方式 |
|:---|:---|:---|:---|
| 知识库 | 口径配置表 | RAG + 知识图谱 | 配置表数据作为种子导入 |
| 反馈机制 | 点赞/点踩 | 完整自循环闭环 | 反馈数据持续积累 |
| 成功案例 | 简单 JSON 存储 | Few-shot 示例库 + 向量索引 | 案例数据迁移升级 |
| 图表 | ECharts 基础渲染 | 图表模板学习 + 智能推荐 | 用户偏好持续积累 |
| 飞书交互 | 文本问答 | 卡片交互 + 快捷按钮 + 超链接跳转 | 逐步丰富交互形态 |
| 权限管理 | 基础用户身份映射 + 行级权限 | 完整 RBAC + 列级权限 + 审计日志 | 权限体系持续增强 |
| 数据安全 | 基础脱敏 + 权限过滤 | 完整数据不出域 + 全链路审计 | 安全体系持续增强 |
| Skill | 原子 Skill（单一任务） | 组合 Skill + DAG 工作流编排 | 从简单到复杂逐步演进 |

### 4.1 核心原则：数据不出域

> **原始数据不离开 HR Portal 服务器，AI 只接收经过权限过滤和脱敏后的数据摘要。**

```text
用户请求 → HR Portal 后端（数据查询 + 权限过滤 + 脱敏）→ 发送给 AI 的仅是脱敏摘要 → AI 返回分析 → 后端合并原始数据 → 返回用户
```

### 4.6 与通用 Agent 的数据安全对比

| 对比维度 | 通用 Agent（Cursor/Claude Code） | 你的 HR Agent |
|:---|:---|:---|
| 数据访问模式 | Agent 直接读取代码/文件 | Agent 通过 API 间接访问，数据经权限过滤 |
| 数据敏感性 | 代码文件（通常不涉及个人隐私） | HR 数据（高敏感） |
| 数据处理位置 | 在 Agent 运行环境处理 | 在 HR Portal 后端处理，数据不出域 |
| 数据留存 | 可能有本地缓存 | 无缓存，每次实时查询 |
| 是否用于训练 | 可能被用于模型训练 | Claude API 承诺不用于训练 |
| 权限控制 | 依赖操作系统权限 | 行/列级权限精细控制 |
| 数据脱敏 | 无 | 敏感字段自动脱敏 |
| 审计能力 | 有限 | 完整审计日志 |

### 5.2 权限上下文数据结构

```python
# 权限上下文（每次 Agent 调用时注入）
class PermissionContext:
    def __init__(self, user_id: str, department_id: str, role: str):
        self.user_id = user_id
        self.department_id = department_id
        self.role = role
        self.department_ids = self._get_subordinate_departments()  # 下属部门列表
        self.visible_columns = self._get_visible_columns()          # 可见列列表
        self.can_manage_metrics = self.role in ['HRVP', 'ADMIN']    # 功能权限
        self.can_export_data = self.role in ['HRVP', 'SSC', 'ANALYST']
        self.can_execute_actions = self.role in ['HRVP', 'SSC']     # 能否执行跨系统操作
        self.sensitive_masks = self._get_sensitive_masks()          # 敏感字段脱敏规则
```

### 5.3 行级权限实现

```sql
-- 查询时自动注入部门过滤
SELECT * FROM dws.monthly_dimission 
WHERE dept_id IN (${user_dept_ids})  -- 自动注入用户可见部门列表
  AND month = '2026-06';
```

### 5.4 列级权限实现

```python
# 返回数据时自动过滤不可见列
def filter_columns(data: dict, permission_context: PermissionContext) -> dict:
    """根据用户权限过滤返回字段"""
    visible = permission_context.visible_columns
    if visible is None:  # 全权限
        return data
    return {k: v for k, v in data.items() if k in visible}
```

### 5.5 权限演进路径

| 阶段 | 权限能力 | 说明 |

## 十四、超链接协同技术规范

### 14.1 链接格式

```text
https://{portal_domain}/agent/redirect?thread_id={thread_id}&chart_id={chart_id}&report_id={report_id}&alert_id={alert_id}&skill_code={skill_code}
```

### 14.2 权限继承机制 + 安全上下文传递

```text
用户点击链接
        │
        ▼
Web 端解析 URL 参数 → 获取 thread_id / skill_code
        │
        ▼
从 Session/Token 获取当前用户身份
        │
        ▼
验证用户是否有权限访问该 thread 的数据 / 执行该 Skill
        │
        ▼
加载用户的脱敏规则和安全配置
        │
   ┌────┴────┐
   │         │
   ▼         ▼
 有权限     无权限
   │         │
   ▼         ▼
恢复上下文   展示权限不足页面
展示内容    引导申请权限
执行 Skill（自动应用脱敏）
```

### 14.3 参数说明

| 参数 | 说明 | 必填 | 示例 |
|:---|:---|:---|:---|
| `thread_id` | 会话 ID，用于恢复对话上下文 | 是 | `thread_20260714_001` |
| `chart_id` | 图表 ID，用于定位具体图表 | 否 | `chart_20260714_003` |
| `report_id` | 报告 ID，用于定位具体报告 | 否 | `report_20260714_001` |
| `alert_id` | 预警 ID，用于定位具体预警 | 否 | `alert_20260714_002` |
| `skill_code` | Skill 编码，用于直接触发 Skill | 否 | `dimission_analysis` |

## 十八、总结

| 问题 | 答案 |
|:---|:---|
| 权限管理什么时候开始？ | 迭代 1.3（第 2-3 天）——身份映射 + 行级权限，从最早可用版本就位 |
| 数据安全什么时候开始？ | 迭代 1.3（第 2-3 天）——敏感字段声明；迭代 1.6（第 4-5 天）——基础脱敏 + AI审计 |
| Skill 框架什么时候开始？ | 迭代 1.2（第 2 天）——Skill 注册表 + 第一个原子 Skill，从第一天就预留编排能力 |
| 数据会暴露给 AI 吗？ | 不会。原始数据不出 HR Portal 服务器，AI 仅接收脱敏后的数据摘要 |
| AI 能看到什么？ | 仅看到经过权限过滤和脱敏后的数据摘要（如"部门编号 101，离职率 20%"） |
| 原始数据在哪处理？ | 在 HR Portal 后端完成查询、过滤、脱敏、聚合 |
| AI 用数据训练吗？ | 不会。Claude API 承诺不用于训练 |
| 如果 AI 厂商违约呢？ | 你的数据经过脱敏，即使泄露也无法还原为原始个人信息 |
| 我能审计吗？ | ✅ 所有数据访问、AI 交互、Skill 执行记录在审计日志中 |
| 为什么 Skill 要早于编排？ | Skill 注册表是编排的基础，先有 Skill 才能编排；先有原子才能组合 |

> **一句话总结：权限管理贯穿始终，从迭代 1.3 开始每个版本都包含权限内容。数据安全从第一天就位——敏感字段声明、基础脱敏、AI 审计日志同步上线。Skill 框架从迭代 1.2 开始预留，从原子 Skill 逐步演进到复合 Skill 再到 DAG 工作流编排。没有权限，不上线；没有数据安全，不扩展；没有 Skill，不编排。**
