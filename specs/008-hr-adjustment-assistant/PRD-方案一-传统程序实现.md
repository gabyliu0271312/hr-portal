## 一、项目背景

### 1.1 现状描述

公司目前已在 HR 数据中台（HR Portal）中接入了北森人力资源系统的人员名单数据和组织架构数据，并具备基础的报表展示能力。然而，在每月例行的人力资源调整工作中，存在以下痛点：

**业务现状：**

- 每月都需要进行组织调整（裁撤、改名、新增）和人员调整（跨组织调动、直接上级变更）
- 当前流程依赖线下沟通：HRBP 通过邮件/微信/口头方式将调整需求告知 SSC 团队
- SSC 团队人工整理调整信息，逐条在北森系统中进行手动操作
- 调整记录散落在邮件和聊天工具中，事后追溯困难

**系统现状：**

- 北森 SaaS 系统在权限管控和数据集成上有一定局限性
- 已有数据中台具备人员和组织的数据拉取能力（通过 `datasources` 模块 + `beisen_client.py` 对接北森 OpenAPI 的 GridHeader/GridData 读取接口）
- 已具备基础报表接口（`reports` 模块）、定时调度引擎（`scheduler` 模块，基于 APScheduler）
- 已搭建 AI 原生底座（`app/ai` 模块），包含 Capability Registry、Orchestrator、Policy Guard、Schema Validator、Audit 等核心能力
- 已接入 Claude Opus 等大模型能力（通过 OpenAI 兼容接口，配置于 `ai_provider_configs`）
- **飞书 SDK 尚未实现**（仅有飞书 SSO 登录能力，无消息发送/机器人/文档接口）
- **北森写操作 API 尚未确认**（当前仅对接报表读取接口）

### 1.2 核心问题

| 问题 | 描述 |
|------|------|
| 信息采集混乱 | 调整需求分散在各个 HRBP 的邮件和聊天记录中，无统一入口 |
| 容易遗漏出错 | 人工整理容易出现遗漏、理解偏差、执行错误 |
| 缺乏时效性 | 每月调整有明确截止日期，但缺乏机制保障 10 号前完成收集 |
| 无据可查 | 调整指令没有统一记录，事后无法追溯"谁说的、什么时候说的" |
| 沟通成本高 | 存在重名、组织名称不标准等歧义时，来回确认耗时耗力 |
| 执行效率低 | SSC 需要在北森系统中逐条手工操作，效率低下 |

## 二、目标用户与用户角色

### 2.1 用户角色定义

| 角色 | 定义 | 核心诉求 |
|------|------|------|
| HRBP | 各业务部门的人力资源业务伙伴 | 快速提交调整需求；需求能被准确理解和执行 |
| SSC 管理员 | 共享服务中心的操作人员 | 减少手工录入；调整指令清晰、准确、结构化 |
| HRVP/HR 负责人 | 人力资源最高负责人 | 可追溯的完整调整记录 |
| 系统管理员 | HR Portal 系统管理员 | 系统易扩展、可配置；组件可复用；故障易排查 |

### 2.2 用户画像

**HRBP（主要提报人）：**

- 每月都会提交调整需求（频率高）
- 不太关心底层数据逻辑，只关心"说出来，系统能懂"
- 希望流程简单，不需要学新工具
- 重名时能快速被引导选择正确的人

**SSC 管理员（主要执行人）：**

- 每月固定时间集中处理调整
- 关心数据是否准确，执行后是否有确认
- 不希望 AI 自动执行，需要自己确认后再操作
- 希望有清晰的"待办清单"和"执行记录"

### 3.2 用户场景与验收测试（User Scenarios & Testing）

> **格式说明：** 每个用户场景包含优先级、独立测试方式、以及 Given/When/Then 验收场景。

---

**US-ADJ-001：HRBP 通过自然语言提交调整需求** `P0`

**为什么是这个优先级：** 这是整个系统的入口。没有需求采集，后续所有流程（校验、确认、执行、审计）都无法启动。

**独立测试：** 在 Portal Web 端打开调整助手对话页，输入一段自然语言调整描述，验证系统能正确解析出调整类型、涉及员工、目标组织、新上级。

**验收场景：**

1. **Given** HRBP 张三已登录 Portal，打开了"月度调整"对话页，
   **When** 他输入"把赵六从研发二部调到研发一部，上级改成钱七，7 月 1 日生效"，
   **Then** 系统正确识别：动作=人员调动、员工=赵六、原组织=研发二部、目标组织=研发一部、新上级=钱七、生效日期=2026-07-01，并展示确认卡片供张三核对。

2. **Given** HRBP 李四已登录 Portal，打开对话页，
   **When** 他输入"销售部改名为华东销售部，同时裁撤旧产品线部"，
   **Then** 系统正确识别两条指令：① 组织更名（销售部→华东销售部）② 组织裁撤（旧产品线部），并按 DAG 规则自动排序。

3. **Given** HRBP 王五已登录 Portal，打开对话页，
   **When** 他输入一段含糊表述"把小刘调一下"（未指定原组织、目标组织、新上级、生效日期），
   **Then** 系统反问"请确认：您要将哪位'刘'员工调动？目标组织是？"，并将不完整的字段标记为待补充，不生成确认卡片。

---

**US-ADJ-002：系统自动查表校验实体信息** `P0`

**为什么是这个优先级：** 用户直接用自然语言说名字，系统必须能查到准确的工号和组织编码，否则生成的都是无效数据。

**独立测试：** 提交包含已知员工姓名和组织名称的调整需求，验证系统查询 `emp_realtime_roster` 和 `org_tree` 后返回的员工 ID 和组织 ID 是否正确。

**验收场景：**

1. **Given** 系统中存在唯一员工"赵六"（工号 10086，研发二部），
   **When** HRBP 输入"把赵六调到研发一部"，
   **Then** 系统查询 `emp_realtime_roster` 后确认赵六工号为 10086，查询 `org_tree` 后确认研发一部编码为 D-101，并在确认卡片中展示完整的"赵六(10086) 研发二部(D-102) → 研发一部(D-101)"。

2. **Given** 系统中不存在员工"王十八"，
   **When** HRBP 输入"把王十八调到市场部"，
   **Then** 系统回复"未找到员工'王十八'，请检查姓名是否正确"，不生成任何调整条目。

3. **Given** 系统中有运营部的近似名称"运营管理部"和"运营支持部"，
   **When** HRBP 输入"把张三调到运营部"，
   **Then** 系统返回两个候选组织供用户选择，不自动选取任何一个。

---

**US-ADJ-003：重名/歧义时主动反问确认** `P0`

**为什么是这个优先级：** 用户明确要求"涉及到重复和疑问的时候，你主动响应，让他进行确认"（U3）。默认猜测造成的错误不可接受。

**独立测试：** 提交包含重名员工的调整需求，验证系统发起反问、列出所有候选、用户选择后正确解析。

**验收场景：**

1. **Given** 系统中存在两位"钱七"：研发一部-钱七(10011) 和 产品部-钱七(10022)，
   **When** HRBP 输入"把赵六的上级改成钱七"，
   **Then** 系统反问"查到两位'钱七'：① 研发一部-钱七(10011) ② 产品部-钱七(10022)。请确认是哪一位？"，并将该条目状态设为 `pending_confirmation`，不默认选取任何候选。

2. **Given** HRBP 对重名反问回复"第一个"，
   **When** 系统收到回复，
   **Then** 系统将钱七(10011)作为最终选择，更新 new_manager_id=10011，并将歧义消解记录写入 `hr_adjustment_ambiguities`。

3. **Given** HRBP 对重名反问超过 48 小时未回复，
   **When** 定时扫描任务触发，
   **Then** 系统自动生成待办条目（`hr_adjustment_todos`），类型=ambiguity_timeout，不默认选取第一个候选。

---

**US-ADJ-004：每月 10 号自动分月规则** `P0`

**为什么是这个优先级：** 用户明确要求"每个月调整是 10 号，10 号之前当月，10 号之后下月"（U4）。分月错误会导致调整生效到错误的月份。

**独立测试：** 在不同日期提交调整需求，验证系统按 10 号分界规则正确分配归属月份。覆盖跨年边界。

**验收场景：**

1. **Given** 当前日期为 2026-06-08（10 号之前），
   **When** HRBP 提交一条调整需求，
   **Then** 系统自动将 target_month 设为"2026-06"，确认卡片上提示"本次调整归属：2026 年 6 月"。

2. **Given** 当前日期为 2026-06-12（10 号之后），
   **When** HRBP 提交一条调整需求，
   **Then** 系统自动将 target_month 设为"2026-07"，确认卡片上提示"本次调整归属：2026 年 7 月（因提交时间在 6 月 10 日之后）"。

3. **Given** 当前日期为 2026-06-12，
   **When** HRBP 提交调整需求时明确说"这个调整记到 6 月份"，(U5 显式覆盖)
   **Then** 系统将 target_month 设为"2026-06"，确认卡片上提示"本次调整归属：2026 年 6 月（用户显式指定）"。

4. **Given** 当前日期为 2026-12-28（跨年边界），
   **When** HRBP 提交一条调整需求，
   **Then** 系统正确计算 target_month 为"2027-01"。

**US-ADJ-005：个人确认卡片（含生效日期必填）** `P0`

**为什么是这个优先级：** 用户要求"根据每个人对话整理成他此时调整的内容，发给他进行确认"（U7）。没有确认环节就无法保证数据准确性。

**独立测试：** 系统为某 HRBP 生成确认卡片，逐条验证卡片内容完整、生效日期字段为必填、确认/修改/取消按钮功能正常。

**验收场景：**

1. **Given** HRBP 张三提交了 2 条调整需求，均已完成实体校验，
   **When** 系统生成确认卡片，
   **Then** 卡片展示 2 条调整的完整信息（员工+工号、原/目标组织+编码、新上级、生效日期输入框），每条有✅确认/✏️修改/❌取消按钮，底部提示"未填写生效日期的条目将进入待办"。

2. **Given** 确认卡片中某条调整的生效日期为空，
   **When** HRBP 点击该条的"✅确认"按钮，
   **Then** 系统拒绝确认，提示"生效日期为必填项"，该条目不进入 confirmed 状态。

3. **Given** HRBP 点击某条调整的"❌取消"按钮，
   **When** 系统收到取消操作，
   **Then** 该条目状态变为 `cancelled`，不出现在调整清单的确认区域，但仍保留在对话流水记录中。

**US-ADJ-006：调整清单汇总展示与确认执行** `P0`

**为什么是这个优先级：** 用户要求"记录在 HR Portal 里面，在这里面进行方案设计实施和 API 打接对通"（U16）。Portal 是 SSC 管理员的操作中心。

**独立测试：** 模拟多条已确认的调整条目，验证调整清单表格正确展示、筛选/排序功能正常、"确认执行"按钮触发正确的后端流程。

**验收场景：**

1. **Given** 本月已有 5 条调整条目（3 条 confirmed、1 条 pending、1 条 cancelled），
   **When** SSC 管理员打开 Portal 调整清单页面，
   **Then** 表格展示所有条目，可按状态筛选（全部/待确认/已确认/已执行），默认按提交时间倒序排列。

2. **Given** 5 条调整条目中的 3 条为 confirmed 状态，
   **When** SSC 管理员点击"确认执行"按钮，
   **Then** 系统弹出二次确认弹窗，列出即将执行的 3 条条目详情，要求输入执行原因备注，然后调用执行流程。

3. **Given** 执行过程中某条北森 API 调用失败，
   **When** 系统检测到失败，
   **Then** 该条目状态标记为 `failed` 并高亮显示，其他未执行的条目暂停，SSC 管理员看到失败原因后决定重试或跳过。

**US-ADJ-007：待办管理（超时未回复转待办）** `P0`

**为什么是这个优先级：** 用户要求"如果反问迟迟没有回复，你要写成待办，不能默认为第一条"（U8）。没有待办机制就无法保障所有条目都被处理。

**独立测试：** 模拟反问超时场景，验证系统自动生成待办、定期提醒、待办可被处理和关闭。

**验收场景：**

1. **Given** 某条调整的重名反问已超 48 小时未回复，
   **When** 定时扫描任务触发，
   **Then** 系统自动在 `hr_adjustment_todos` 中创建一条待办，类型=ambiguity_timeout，指定提报人为待办人，截止日期=当前月 10 号。

2. **Given** 某条调整已确认但缺少生效日期，
   **When** 系统检测到缺失有效日期字段，
   **Then** 自动生成待办，类型=missing_effective_date，截止日期=当月 10 号。

3. **Given** 待办看板中有 3 条待办（1 条逾期、1 条今天截止、1 条正常），
   **When** SSC 管理员打开待办看板，
   **Then** 逾期条目红色高亮，今天截止条目黄色标记，正常条目白色显示，按截止日期紧急程度排序。

4. **Given** SSC 管理员处理某条待办，补充了缺失信息并保存，
   **When** 待办关联的调整条目信息被更新，
   **Then** 待办状态变为 `resolved`，关联条目推进到下一步状态。

---

**US-ADJ-008：审计追溯（对话流水 + 执行日志）** `P0`

**为什么是这个优先级：** 用户要求"有据可查，按月把每个人在对话中说的需要调整什么都记录下来"（U10）。合规性是最基本要求。

**独立测试：** 完成一次完整的"对话→确认→执行"流程后，验证审计日志记录包含原始对话、歧义消解链、最终决议、执行日志的完整链路。

**验收场景：**

1. **Given** 用户完成了一条调整的全流程（提交→反问→确认→执行），
   **When** 审计员按 conversation_id 查询审计日志，
   **Then** 完整展示：原始文本、每轮歧义消解（问题/候选/用户选择）、最终结构化数据、北森 API 请求/响应快照、执行时间、执行人。所有步骤通过 trace_id 串联。

2. **Given** 本月有 10 条调整记录，分别来自 3 位 HRBP，
   **When** 审计员按 target_month="2026-07" 查询，
   **Then** 返回 7 月份所有调整记录，可按提报人、调整类型、执行状态筛选，支持导出为 Excel。

---

**US-ADJ-009：定时提醒触发（每月 8 号）** `P1`（Phase B 依赖飞书）

**为什么是这个优先级：** 用户要求"定期触发提醒，发到群里"（U6），但此功能依赖飞书 SDK（Phase B），且 Phase A 的 Portal 内已有待办提醒可部分替代。

**独立测试：** 配置每月 8 号 10:00 的定时任务，验证任务正确触发、工作日计算正确（8 号为周末则顺延）。

**验收场景：**

1. **Given** 当前日期为 2026-07-08（周三），时间为 10:00，
   **When** APScheduler 触发定时任务，
   **Then** 飞书机器人向指定 HR 群发送提醒消息"📢 本月组织与人员调整需求收集已开始，请在 7 月 10 日前 @HR中台 提交调整需求"，Portal 内同步推送系统通知。

2. **Given** 当前日期为 2026-08-08（周六），
   **When** 定时任务检查日期，
   **Then** 自动顺延至 8 月 10 日（周一）10:00 触发，发送群提醒。

### 3.3 边界情况（Edge Cases）

| # | 场景 | 处理规则 |
|---|------|----------|
| E1 | 提交时北森数据未同步（员工表无此人） | 先查本地 `emp_realtime_roster`，若无则提示"员工数据暂未同步，请确认员工姓名或联系管理员"，生成待办 |
| E2 | 同时提交多条调整，部分成功部分失败 | 按 DAG 顺序执行，失败条目标记 `failed` 并阻塞后续，已成功的不回滚（确认北森事务能力前不下结论） |
| E3 | 组织裁撤时仍有在职员工 | 系统检测后反问"该组织下仍有 X 名在职员工，请先处理员工调动"，阻止裁撤操作 |
| E4 | 生效日期为过去日期 | 允许设定过去日期（补录历史调整），但弹窗二次确认"该生效日期已过，是否确认？" |
| E5 | 同一员工在同月被两次提交调整 | 后提交的条目自动标记为 `duplicate`，提示"该员工本月已有一条调整记录（编号 XXX），是否覆盖？" |
| E6 | 确认卡片中取消了全部条目 | 会话状态设为 `cancelled`，保留完整对话流水，但不生成任何 confirmed 条目 |
| E7 | 用户在对话中修正前一条指令 | 如"不对，赵六不是调到研发一部，是研发三部"，系统更新该条目的目标组织，不创建新条目 |
| E8 | 月结日（10 号）当天提交 | 精确到小时：10 号 00:00-23:59 之间提交的全部归入当月 |
| E9 | 多人同时提交相同员工调整 | 按提交时间戳排序，先进先出；后者自动检测为 E5 情况 |
| E10 | 对话过程中用户掉线/刷新页面 | 会话上下文由 Redis 保存（复用现有 AI 对话管理），刷新后自动恢复 |
| E11 | 北森 API 批量调用中部分超时 | 超时的条目标记 `failed`，已成功的正常更新状态，不整体回滚 |
| E12 | 权限变更（HRBP 离职/调岗） | 已提交但未确认的条目不自动取消；白名单更新后新提交有效，旧会话只读 |
| E13 | 组织操作涉及非当前生效组织 | 北森 API 仅支持"当前生效"组织变更/停用。系统检测到目标组织非当前生效时，提示"北森 API 不支持对非当前生效组织执行此操作"，转为待办，待组织状态变更后处理 |
| E14 | 人员调动对象为离职/外部人员 | 北森 API 仅支持已入职正式员工/实习生。执行前反查员工状态，若为离职/外部人员则阻止执行 → 生成待办，标注"员工状态不符" |

## 四、业务目标与成功标准

### 4.1 业务目标

| 目标 | 说明 |
|------|------|
| 效率提升 | 将 HRBP 提交调整需求的时间从"写邮件/发消息"缩短到 Portal 内对话式录入 |
| 准确性保障 | 通过自动查表和主动反问，消除因重名、组织名称不标准带来的理解偏差 |
| 时效性保障 | 通过定时提醒和月度分界规则，确保每月调整在 10 号前完成收集 |
| 全程可追溯 | 所有对话流水和最终决议完整保存，任何调整都能追溯到原始指令 |
| 平台化可扩展 | 所有能力注册为 AI Capability，后续其他 HR 场景可复用编排 |

### 4.2 功能性需求（Functional Requirements）

> **标注说明：** MUST = 必须实现，阻塞上线；SHOULD = 应该实现，可降级；MAY = 可选增强。

**FR-ADJ-001：对话采集入口** `MUST`

系统 MUST 在全局 AI 侧边栏（`GlobalAiAssistant.vue`）提供月度调整助手入口，用户切换 capability 至 `hr.adjustment.collect` 后可用自然语言描述人员调动（transfer，含汇报关系变更）、组织更名（rename）、组织新增（create）、组织停用（disable）四种调整类型。后端统一通过 `POST /api/v1/ai/chat`（capability=hr.adjustment.collect）接收请求。调整清单和待办看板以独立页面形式在 Portal 中提供（路由 `/hr-adjustment/list`、`/hr-adjustment/todos`）。

**FR-ADJ-002：员工信息查表校验** `MUST`

收到含员工姓名的调整需求时，系统 MUST 查询 `emp_realtime_roster` 表进行精确匹配（员工姓名→工号）。唯一匹配则直接采用；多个匹配则发起反问；零匹配则提示"未找到该员工"。MUST NOT 允许 LLM 根据训练数据猜测员工身份。

**FR-ADJ-003：组织信息查表校验** `MUST`

收到含组织名称的调整需求时，系统 MUST 查询 `org_tree` 表进行精确/模糊匹配。精确匹配则直接采用；模糊匹配则列出候选反问；零匹配则提示"未找到该组织"。

**FR-ADJ-004：重名/歧义主动反问** `MUST`

系统 MUST 在以下情况下主动发起反问：① 员工重名（>1 候选）；② 组织名模糊匹配（相似度 < 1.0）；③ 汇报关系存在循环依赖。反问 MUST 列出所有候选供用户选择，MUST NOT 默认选取任何候选。

**FR-ADJ-005：10 号分月规则** `MUST`

系统 MUST 按每月 10 号为分界点分配 target_month：10 号前提交归当月，10 号及之后提交归下月。用户显式声明"记到下个月"等关键词时 MUST 覆盖默认规则。跨年边界（12 月→次年 1 月）MUST 正确计算。

**FR-ADJ-006：多指令 DAG 排序** `MUST`

当一次对话包含 ≥2 条调整指令时，系统 MUST 按确定性 DAG 规则排序：① 新增组织 → ② 组织更名（变更接口）→ ③ 人员调动（含汇报关系变更）→ ④ 组织停用（变更接口）。排序逻辑 MUST 由后端代码实现（`utils/dag_sorter.py`），不依赖 LLM。支持 4 种 action_type：transfer / rename / create / disable。

**FR-ADJ-007：确认卡片（含生效日期必填）** `MUST`

系统 MUST 为每位提报人生成个人确认卡片，逐条展示调整详情。生效日期字段 MUST 为必填；未填写时 MUST 拒绝确认。卡片 MUST 提供确认/修改/取消三种操作。

**FR-ADJ-008：超时转待办** `MUST`

反问发起后超过 48 小时未回复，系统 MUST 自动生成待办条目（`hr_adjustment_todos`），类型=ambiguity_timeout。MUST NOT 默认选取第一个候选或自动推进流程。

**FR-ADJ-009：待办管理** `MUST`

系统 MUST 提供待办看板，支持按状态（open/resolved/expired）、待办人、类型筛选，按截止日期紧急程度排序。逾期条目 MUST 红色高亮。系统 MUST 支持待办处理（补充信息→推进条目状态→关闭待办）。

**FR-ADJ-010：调整清单汇总展示** `MUST`

系统 MUST 在 Portal 提供月度调整清单页面，以表格形式展示所有调整条目，含序号、调整类型、对象（姓名+工号）、原组织、目标组织、新上级、生效日期、提报人、状态。MUST 支持按状态筛选和按提交时间排序。

**FR-ADJ-011：确认执行机制** `MUST`

SSC 管理员 MUST 能在 Portal 中点击"确认执行"按钮触发执行流程。系统 MUST 在点击后弹出二次确认弹窗，列出即将执行的全部条目。执行 MUST 由后端确定性代码触发（不经过 LLM），MUST 记录完整的执行日志。

**FR-ADJ-012：审计日志全链路记录** `MUST`

系统 MUST 记录完整的操作链路：原始对话文本 → 每轮歧义消解（问题/候选/选择）→ 最终结构化数据 → 执行请求/响应。所有步骤 MUST 通过 conversation_id 和 trace_id 关联。日志 MUST 支持按月份、提报人、调整类型筛选和导出。

**FR-ADJ-013：权限三层控制** `MUST`

系统 MUST 实施三层权限：① 入口权限（用户能否使用调整助手，通过菜单权限 code 控制）；② Capability 权限（能否调用 hr.adjustment.* 能力）；③ 数据权限（员工/组织查询走 scope_filter + masker）。

**FR-ADJ-014：AI 不直接写入业务数据** `MUST`

AI Capability（hr.adjustment.collect）MUST 只能生成 draft 状态条目。confirmed 状态 MUST 由用户确认操作触发。executed 状态 MUST 由"确认执行"按钮触发。LLM MUST NOT 拥有调用北森写 API 或直接修改业务数据的能力。

**FR-ADJ-015：会话上下文持久化** `MUST`

对话过程中用户刷新页面或断线重连，系统 MUST 能从 Redis 恢复会话上下文。已生成的草稿条目 MUST NOT 丢失。会话超时时间 MUST 可配置（默认 72 小时）。

**FR-ADJ-016：定时提醒** `SHOULD`（Phase B 飞书依赖）

系统 SHOULD 在每月 8 号 10:00 触发提醒（APScheduler），通知 HR 群提交当月调整需求。若 8 号为周末则顺延至下周一。Phase A SHOULD 在 Portal 内推送系统通知作为替代。

**FR-ADJ-017：飞书私聊确认卡片** `SHOULD`（Phase B 飞书依赖）

Phase B 阶段 SHOULD 通过飞书私聊发送确认卡片（复用 Web 端同一套数据结构和验证逻辑）。

**FR-ADJ-018：飞书文档写入** `SHOULD`（Phase B 飞书依赖）

Phase B 阶段 SHOULD 将 confirmed 条目自动追加到飞书主文档（表格格式），对话流水写入飞书附录文档。

**FR-ADJ-019：北森 API 执行前实时反查** `SHOULD`（Phase C 依赖）

Phase C 阶段执行前 SHOULD 实时查询北森最新状态（目标组织是否存在、员工是否在原岗位、新上级是否在职），全部通过方允许执行。

**FR-ADJ-020：北森 API 批量执行** `SHOULD`（Phase C 依赖）

Phase C 阶段 SHOULD 支持批量调用北森 API，含频率控制（≤50 次/秒）、超时重试（默认 3 次，指数退避）、防重保障（新增/调动接口 3 秒内相同参数拦截）。

**FR-ADJ-021：纯汇报关系变更走人员调动** `MUST`（Phase C 依赖）

北森不存在独立的"汇报关系变更"API。纯换上级不调部门时，系统 MUST 调用人员调动 API，`oIdDepartment` 传入员工当前部门，`pOIdEmpAdmin` 传入新上级。MUST NOT 假设存在独立 manager change 端点。

**FR-ADJ-022：组织操作 OriginalId 管理** `MUST`（Phase C 依赖）

新增组织时 MUST 传入 `originalId`（内部组织编码，建立北森外部 ID 映射）。后续变更/停用时 MUST 通过 `originalId` 定位组织，不依赖北森 OId。`hr_adjustment_items` MUST 增加 `beisen_org_original_id` 字段记录。

## 六、系统方案设计

### 6.1 平台定位与设计理念

HR Portal 定位于企业 HR 基础设施平台，采用 **Capability 驱动的 AI 原生架构**。所有业务能力注册为标准化 Capability，通过 AI Orchestrator 进行意图识别、能力路由和受控编排。

核心原则（来自 ADR-AI-001 ~ ADR-AI-010）：

```
LLM 负责理解、生成草稿、解释结果。
```

**2.2 实体解析与查表校验**

复用 `trees/router.py`（组织树查询）和 `data/router.py`（员工数据查询）：

| 校验环节 | 查询来源 | 异常处理 |
|----------|----------|----------|
| 员工查重 | `emp_realtime_roster` 表（`data` 模块） | 重名 → AI 列出选项反问；查无 → 提示"未找到该员工" |
| 组织查重 | `org_tree` 表（`trees` 模块） | 近似名称 → AI 反问确认；查无 → 提示"未找到该组织" |

| 权限校验 | `auth` 模块 RBAC + `permissions` 模块 | 无权限 → 提示"请联系 HRVP 授权" |

**2.3 多指令拓扑排序（DAG）**

当一次对话包含多条调整指令时，系统按确定性规则排序（代码逻辑，不依赖 LLM）：

```
① 新增组织 → ② 组织更名（变更接口） → ③ 人员调动（含汇报关系变更） → ④ 组织停用（变更接口）
```

**排序由后端确定性代码实现**（`utils/dag_sorter.py`），LLM 仅负责从自然语言中识别出多条指令。

> **⚠️ 基于实际 API 的修正：** 原 PRD 假设"组织裁撤"使用 DELETE 接口，实际北森使用 PUT 变更接口实现停用。排序优先级不变（④ 仍为最后），但实现方式修正。

示例：用户说"把销售部改名为华东销售部，把赵六从研发部调到销售部，停用旧产品线部"

系统排序为：① 组织更名 → ② 人员调动 → ③ 组织停用

**2.4 日程判断与生效日期管理**

| 规则 | 说明 |
|------|------|
| 默认规则 | 每月 10 号为分界点，10 号前提交归当月，10 号后归下月 |
| 用户显式覆盖 | 若用户明确说"记到下个月"，则归入下月 |
| 生效日期确认 | 确认卡片中强制要求填写具体生效日期（YYYY-MM-DD） |

**2.5 歧义处理 → 待办（不默认不猜测）**

| 场景 | 处理方式 |
|------|----------|
| 重名有多个候选 | AI 列出选项反问，等待用户选择 |
| 反问超时未回复 | 自动生成待办，不默认选第一个 |
| 组织名模糊匹配 | AI 反问确认最匹配的组织 |
| 信息不完整 | 标记缺失字段，生成待办 |

#### 6.5.2 `hr.adjustment.collect` 详细设计

```python
@dataclass(frozen=True)
class HrAdjustmentCollectCapability:
    capability_id = "hr.adjustment.collect"
    name = "月度调整需求采集"
    module = "hr_adjustment"
    type = "chat"
    risk_level = "medium"
    confirmation = "none"
    tools = [
        "tool:employee.search",        # 查 emp_realtime_roster
        "tool:organization.search",    # 查 org_tree
        "tool:dag.sort",               # DAG 排序
        "tool:schedule.check",         # 日程判断（10 号分界）
    ]
    policy_profile = {
        "max_context_rows": 10,
        "allow_sensitive_context": False,
    }
    model_profile = "reasoning"
    audit_enabled = True
```

**关键约束：**

- 此 Capability 只能生成草稿（draft），不能直接写入业务数据
- 遇到重名/歧义时强制反问，不允许 LLM 猜测
- DAG 排序由确定性代码执行，LLM 只负责识别多条指令
- 员工/组织查询必须复用现有 `data` + `trees` 模块，不能直接拼 SQL

#### 6.5.3 `hr.adjustment.confirm` 详细设计

```python
@dataclass(frozen=True)
class HrAdjustmentConfirmCapability:
    capability_id = "hr.adjustment.confirm"
    name = "调整确认执行"
    module = "hr_adjustment"
    type = "write"
    risk_level = "high"
    confirmation = "required"        # ★ 高风险：必须用户确认
    side_effect_tags = [
        "writes_data",               # 写入业务数据
        "batch_operation",            # 批量操作
    ]
    tools = []
    model_profile = "none"           # 确定性操作，不需要 LLM
    audit_enabled = True
```

**关键约束：**

- 用户确认执行前，不调用任何北森写 API
- 确认通过后，前端触发 `/api/v1/hr-adjustments/{id}/execute`
- 后端执行时强制走权限、数据范围、脱敏
- 完整操作链路记录到 `system_logs`

### 6.7 北森 API 网关设计（基于实际 API 文档重写）

#### 6.7.1 文档依据

以下设计基于已获取的北森 OpenAPI 文档（2026-06-26 获取）：

| 接口 | 文档文件 |
|------|----------|
| 新增组织单元 | `北森新增组织单元接口.md` |
| 变更组织单元（含更名） | `北森变更组织单元接口.md` |
| 停用组织单元 | `北森停用组织单元接口.md` |
| 人员调动（含汇报关系变更） | `北森人员调动接口.md` |

#### 6.7.2 实际 API 清单（已确认）

| 调整类型 | 北森 API | HTTP 方法 | 关键必填参数 | 幂等性/防重 |
|----------|----------|-----------|--------------|-------------|
| 组织新增 | 新增组织单元 | POST | `data.name`、`data.code`、`data.establishDate`、`pOIdOrgAdmin`（上级组织OId） | **3 秒防重**（相同参数 3 秒内仅一次） |
| 组织更名 | 变更组织单元 | PUT | `oId` / `originalId`（二选一）、`data.name`（新名称）、`data.startDate`（生效日期） | 支持部分变更，仅传需更新字段 |
| 组织停用 | 停用组织单元 | PUT | 同变更接口参数 | 同变更接口 |
| 人员调动 | 人员调动 | POST | `employmentRecord.startDate`（生效日期）、`employmentRecord.oIdDepartment`（目标部门OId）、`userId` / `originalId`（二选一） | **3 秒防重** |
| 汇报关系变更 | **合并到人员调动** | — | `employmentRecord.pOIdEmpAdmin`（直线经理UserID，可选） | **无独立接口**，作为人员调动的可选字段 |

> **⚠️ 关键发现：**
>
> 1. **组织更名和停用是同一接口的不同调用**——都是"变更组织单元"PUT 接口。更名传 `data.name`，停用传对应的状态字段。不存在独立的"组织更名"或"组织裁撤"端点。
> 2. **汇报关系变更没有独立接口**——直线经理变更（`pOIdEmpAdmin`）是人员调动接口的可选字段。如果只需要换上级不调部门，仍需调用人员调动接口并传入当前部门作为 `oIdDepartment`。
> 3. **仅支持"当前生效"的组织**——变更/停用只能操作当前生效的组织，不支持修改未来或过去的组织记录。
> 4. **人员调动仅支持已入职正式员工/实习生**——外部人员、已离职人员不支持通过 API 调动。
> 5. **人员调动含组织编制超编校验**——北森内部自动校验，超编时 API 返回错误。

#### 6.7.3 接口关键参数详情

**组织新增（POST）— 必填项：**

| 参数 | 类型 | 说明 |
|------|------|------|
| `data.name` | string | 组织名称，如"技术部" |
| `data.code` | string | 组织编码，如"BP123" |
| `data.establishDate` | date-time | 设立日期，如"2021-01-01" |
| `pOIdOrgAdmin` 或 `pOIdOrgAdminOriginalId` | integer/string | 行政维度上级组织，二选一必填 |
| `data.originalId` | string（可选） | 外部 ID，用于建立第三方系统 ID 映射关系，**后续变更/停用等操作的前置依赖** |

**组织变更/停用（PUT）— 关键特性：**

- 支持部分变更（只传需变更字段）
- 通过 `emptyFields` 数组支持清空字段
- `originalId` 必须在新增时已关联，否则报错
- 十级组织（FirstLevel-TenthLevel）自动计算，无需手工传入

**人员调动（POST）— 必填项：**

| 参数 | 类型 | 说明 |
|------|------|------|
| `employmentRecord.startDate` | date-time | 生效日期，必须 ≥ 1900-01-01 |
| `employmentRecord.oIdDepartment` | integer | 调动后部门 OId（必填） |
| `userId` 或 `originalId` | integer/string | 员工标识，二选一 |
| `employmentRecord.pOIdEmpAdmin` | integer（可选） | 直线经理 UserID，如需同时变更上级则传入 |

#### 6.7.4 全局约束

| 约束项 | 值 |
|--------|-----|
| 频率限制 | 50 次/秒/企业，1500 次/分钟/企业 |
| 组织操作限制 | ① 行政维度上级不能为空且必须存在当前生效记录且不能为默认部门 ② 仅支持当前生效组织（不支持过去/未来组织） |
| 人员调动限制 | ① 仅支持已入职正式员工/实习生 ② 不支持外部人员和已离职人员 ③ 含组织编制超编校验 |
| 防重机制 | 新增组织 + 人员调动：相同参数 3 秒内仅能请求一次 |
| OriginalId 机制 | 新增时创建映射 → 后续变更/停用时通过 OriginalId 定位，前提是新增时已关联 |
| OriginalId 规则 | 任何 `XXXOriginalId` 字段（如 `HRBPOriginalId`）的前提是该外部 ID 已在之前的操作中关联，否则报错 |

#### 6.7.5 对方案设计的影响

| # | 影响 | 对应的方案调整 |
|----|------|---------------|
| ① | 不存在独立的 DELETE 端点 | DAG 排序中"组织裁撤"修正为"组织停用（变更操作）"。执行顺序不变（④ 仍为最后），但实现方式改为 PUT 而非 DELETE |
| ② | 汇报关系变更无独立 API | 原 PRD 中 `manager_change` 动作类型合并到 `transfer`。新增 FR-ADJ-021：纯汇报变更=调用人员调动 API 且目标部门=当前部门 |
| ③ | 仅支持当前生效组织 | 新增 Edge Case E13：若 HRBP 提交的调整涉及未来/过去的组织变更，系统需在 Portal 内提示"北森 API 不支持非当前生效组织操作"，转为待办 |
| ④ | OriginalId 机制 | 新增组织时 MUST 传入 `originalId`（我们的内部组织编码），后续变更/停用通过 `originalId` 定位，不依赖北森 OId。`hr_adjustment_items` 需增加 `beisen_org_original_id` 字段 |
| ⑤ | 人员调动限制 | 执行前需反查员工状态（是否在职、是否正式/实习），非合规员工阻止执行并生成待办 |

### 6.8 安全与权限设计（复用现有体系）

#### 6.8.1 三层权限模型

```
第一层：入口权限
  用户能否进入调整助手页面和使用对应 AI Capability。
  → 通过现有菜单权限体系控制（menu code: tools.hr_adjustment）

第二层：Capability 权限
  用户能否调用 hr.adjustment.collect / confirm 等 AI 能力。
  → 通过 capabilities.py 的 required_permission 控制

第三层：业务数据权限
  员工查询、组织查询均走现有 scope_filter + masker。
  → 复用 permissions/scope_filter.py
```

#### 6.8.2 执行阶段安全约束

- AI 只能生成草稿，不能直接写入 `hr_adjustment_items`（状态: draft）
- 用户确认后，后端确定性代码写入（状态: confirmed）
- 北森 API 执行由 Portal 前端的"确认执行"按钮触发（状态: executed），LLM 不能触发
- 所有写入操作必须通过 Policy Guard 校验
- 审计日志记录完整操作链路

## 九、核心设计原则总结

| # | 原则 | 来源 |
|---|------|------|
| ① | **AI 做草稿，人做决策** | ADR-AI-002：AI 输出默认为草稿 Artifact |
| ② | **Capability 注册驱动** | ADR-AI-004：代码白名单注册，数据库启停展示 |
| ③ | **对话时校验，执行时二次反查** | 确保数据时效性 |
| ④ | **DAG 确定性排序，AI 只做识别** | 杜绝拓扑混乱 |
| ⑤ | **歧义转待办，不默认不猜测** | 宁可慢，不可错 |
| ⑥ | **Portal 确认 + API 执行，全程可追溯** | 所有操作在 Portal 内完成 |
| ⑦ | **三层权限模型** | 入口权限 + Capability 权限 + 业务数据权限 |
| ⑧ | **渐进交付，不耦合飞书** | Phase A Web 端先行，Phase B 再加飞书 |
| ⑨ | **北森 API 已确认，方案已基于实况修正** | 4 个写 API 已获取，不存在独立 DELETE 和 manager change 端点，方案已适配 |
| ⑩ | **OriginalId 映射机制** | 新增组织时分配 OriginalId → 后续操作通过 OriginalId 定位，不依赖北森 OId |
| ⑪ | **汇报关系变更合并到人员调动** | 纯换上级场景仍调用人员调动 API，oIdDepartment 传入当前部门 |

### 12.2 架构矛盾处理策略

#### 12.2.1 飞书机器人时机矛盾

**矛盾：** ADR-AI-007 明确将飞书机器人排在 Phase 6（最远期），但原始需求以飞书群聊 @ 机器人为核心入口（U1-U12 均依赖飞书能力）。

**处理策略：**

```
Phase A（Web 端先行）
  ├─ Portal 内对话式采集（复用 /api/v1/ai/chat）
  ├─ Web 端确认卡片（Portal 页面，非飞书私聊）
  ├─ Portal 内待办提醒 + 系统通知（替代飞书群提醒）
  └─ ✅ 不依赖飞书，可独立上线形成业务闭环

Phase B（飞书集成，Phase A 稳定后）
  ├─ 飞书群聊 @ 机器人采集
  ├─ 飞书私聊确认卡片（复用 Web 端数据结构）
  ├─ 飞书定时群提醒（每月 8 号）
  └─ 飞书文档写入（主文档 + 附录）

关键原则：不调整 ADR-AI-007，不提前飞书机器人优先级。
通过分阶段交付化解矛盾——Web 端先行验证业务闭环，飞书集成作为体验增强。
```

#### 12.2.2 Agent SDK vs Capability Registry 架构选择

**矛盾：** 方案二提出"Agent SDK + Skill 技能包"路线，但现有架构强烈主张 Capability Registry + Orchestrator 模式。

**处理策略：**

| 维度 | Capability Registry（当前路线） | Agent SDK（方案二） |
|------|-------------------------------|---------------------|
| LLM 角色 | 意图识别 + 草稿生成 | 全链路推理编排 |
| 后端角色 | 权限 + 确定性执行 + 审计 | 简化为原子 API |
| 审计粒度 | 每个 Capability 独立审计 | Skill 级别（需额外拆分） |
| Token 成本 | 低（仅对话解析用 LLM） | 高（每次对话多轮推理） |
| 团队学习成本 | 低（已有 14 个 Capability 实践） | 高（需 Agent SDK 选型 + POC） |
| 与现有架构兼容 | ✅ 完全兼容 | ⚠️ 需大幅调整 |

**结论：** 明确选择 Capability Registry 路线。不引入 Agent SDK。早期用方案一的确定性逻辑上线，验证业务闭环后，可逐步引入 Agent 增强（但不作为主力架构）。

#### 12.2.3 审计粒度要求

评审指出方案二将 DAG 排序、校验逻辑全部放入 Skill 脚本中，缺少 Capability 粒度的审计。当前方案一的设计已满足此要求：

- 每个操作步骤记录独立的 capability_id
- DAG 排序由 `utils/dag_sorter.py` 确定性代码执行（不经过 LLM，有完整日志）
- 歧义消解链记录在 `hr_adjustment_ambiguities` 表
- 执行日志记录在 `system_logs` + `hr_adjustment_items.beisen_request/beisen_response`
