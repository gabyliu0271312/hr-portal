### 2.3 PRD 与架构决策的矛盾点

#### 矛盾1：飞书机器人的优先级

> **ADR-AI-007**: "飞书机器人延后到 Web 工作台稳定之后"  
> **路线图 Phase 6**: 飞书机器人属于最远期的阶段

PRD将飞书群聊@机器人作为核心入口（Step 1/2/3/4全部依赖飞书），但架构决策明确将其排在 Phase 6（当前处于 Phase 0/1）。这是最严重的矛盾。

**建议**: 或调整架构路线图将飞书机器人前移，或在PRD中增加"Web端对话工作台"作为过渡方案。

#### 矛盾2：Agent SDK 方案的架构方向

> **架构评审 §2.3**: "把'AI 原生'理解成'所有功能优先调用 LLM'...这个方向如果直接实施，会把当前已经稳定的权限、数据范围、报表执行推到 LLM 外围"  
> **架构评审 §4.1**: "第一期 AI 平台应内嵌在 HR Portal 后端"

方案二提出的"Agent SDK + Skill技能包，后端仅保留原子能力层"本质上与现有架构设计方向**存在张力**。现有架构强烈主张：

- LLM负责理解和草稿，后端负责权限、数据查询、确定性计算
- Capability Registry + Orchestrator 而不是独立 Agent 框架
- 反对将业务逻辑全部上移到 Skill/Agent

这不是说方案二不可行，而是需要明确选择：**要么调整架构方向接纳 Agent SDK，要么方案二需要大幅修改以符合现有架构设计**。

#### 矛盾3：向量检索组件

PRD两方案都在组件清单中列出了"向量检索组件"，但架构评审明确指出"一期不做向量库 RAG"，当前实施蓝图中也是"不做事项"之一。

## 四、两方案深度对比（结合代码实际）

### 4.1 方案一（传统程序实现）评审

#### 优势（与代码契合度高的部分）

1. **技术栈一致**: Spring Boot → 但现有后端是 **FastAPI (Python)**，不是 Spring Boot。技术选型需修正。
2. **组件化理念**: 与现有架构的"能力注册 + 编排器"模式兼容
3. **权限模型**: 可以直接复用现有 L1-L4 四层权限
4. **AI集成方式**: "Claude API直接调用，不涉及Agent" 与现有 `model_provider.py` 方式一致
5. **数据存储**: MySQL → 但现有是 **PostgreSQL**，需修正

#### 问题与风险

| 问题 | 严重程度 | 建议 |
|---|---|---|
| 技术栈标注错误（Spring Boot / MySQL / RocketMQ） | 🔴 高 | 应修正为 FastAPI + PostgreSQL + APScheduler（项目已使用） |
| DAG排序引擎从零构建 | 🟡 中 | 如非复杂场景，可用简单拓扑排序替代，不必引入Activiti/Temporal |
| 工作流引擎选择过重 | 🟡 中 | 项目无工作流需求先例，Activiti/Temporal对月度调整场景过于重型 |
| 组件状态标注不准（见§2.1） | 🔴 高 | 需逐项重新核实 |
| 消息队列引入 | 🟡 中 | 项目目前无MQ，24个组件中仅此1个依赖MQ，可考虑APScheduler替代 |
| 飞书机器人未实现但PRD以之为核心 | 🔴 高 | 需先实现飞书集成基础层 |
| 北森API写操作接口未确认 | 🔴 高 | 当前仅对接了北森Report读取接口（GridHeader/GridData），**人员调动、组织更名等写操作API是否存在、是否开放完全未知** |

#### 工作量重评估（建议修正）

| 原评估 | 调整建议 | 理由 |
|---|---|---|
| 飞书事件订阅 5天 | **15天** | 当前飞书模块完全空白，需从feishu_client → notification_service → 交互卡片全部从零搭建 |
| AI解析服务 5天 | **8天** | 需新建Capability注册 + prompt模板 + 结构化输出校验 |
| 校验引擎 8天 | **10天** | 新增org_unit表后需适配，权限校验需对接现有scope_filter体系 |
| 北森API网关 5天 | **10天+** | 写操作API完全未知，可能需要与北森供应商沟通确认接口能力 |
| 飞书文档组件 3天 | **8天** | 飞书文档API封装 + 模板系统 + 权限控制 |
| 待办管理+日程 5天 | **8天** | 全新模块，含CRUD、状态机、逾期提醒 |
| 集成测试 10天 | **15天** | 飞书+北森两端联调，不可控因素多 |
| **合计** | **约64→约90人天** | 交付周期约 **14-16周**（3人并行） |

### 4.2 方案二（Agent Skill实现）评审

#### 与现有架构的真实关系

方案二的核心理念"Agent做编排，后端做执行"与架构评审文档**部分一致但存在张力**：

**一致的部分**：
- "LLM负责理解和编排，后端负责权限和执行" — 与架构评审一致
- "人机回环，Agent不能直调北森API" — 与现有 draft → confirm → execute 链路一致
- "工具级权限控制" — 与 Tool Wrapper + Policy Guard 设计一致

**存在张力的部分**：
- 架构评审明确反对"把所有业务逻辑封装为黑盒 Skill"，要求每一步保留 capability_id 和独立审计
- 方案二将 DAG 排序、校验逻辑全部放入 Skill 的 Python 脚本中，缺少 Capability 粒度的审计
- 架构路线图 Phase 5 的 "工作流编排"是多个原子 Capability 受控编排，而非方案二的 monolithic Skill

#### 问题与风险

| 问题 | 严重程度 | 建议 |
|---|---|---|
| Agent SDK选型未定且团队无经验 | 🔴 高 | Claude Agent SDK / Dify / Coze Agent 三选一，需POC验证 |
| 后端简化为"原子API"，与现有微服务架构冲突 | 🟡 中 | 现有后端已经有完整的业务逻辑层，推到重来成本可能高于新建 |
| Token成本不可控 | 🟡 中 | 每次对话都需Agent推理链路（Thought→Action→Observation），Token消耗远高于方案一 |
| Skill的审计粒度问题 | 🟡 中 | 建议按架构评审要求，每个Skill执行步骤记录独立的capability调用审计 |
| 工作量偏乐观（46人天） | 🔴 高 | 见下方重评估 |
| 飞书/北森底层依赖与方案一完全相同 | 🔴 高 | Agent再智能也无法替代飞书API和北森API的缺失 |

#### 工作量重评估（建议修正）

| 原评估 | 调整建议 | 理由 |
|---|---|---|
| 后端原子API 5天 | **10天** | 不是"精简"而是"重构"，现有业务逻辑需要梳理哪些保留哪些废弃 |
| Agent SDK选型 3天 | **8天** | 需POC对比 + 安全评估 + 集成测试 |
| 8个Skill编写 20天 | **25天** | 每个SKILL.md + Tool函数 + 端到端测试，不是简单写prompt |
| Portal前端改造 5天 | **10天** | 需集成Agent对话界面，相比方案一的表单式交互更复杂 |
| 数据库调整 3天 | **5天** | 新增agent_traces表的审计链路集成 |
| 集成测试 10天 | **15天** | Agent行为不确定性的测试成本更高 |
| **新增** Agent安全审计 | **8天** | 工具白名单、权限校验、越狱防护、推理链路审计 |
| **合计** | **46→约81人天** | 交付周期约 **10-12周**（2人并行+AI辅助） |

**关键发现**: 修正后两方案的工作量差距大幅缩小（90 vs 81 人天），方案二的"节省28%"优势不再明显。真正的差异在于**维护成本**和**迭代灵活性**，而非开发工作量。

## 六、具体修正建议

### 6.1 两方案共同需修正

1. **技术栈统一**: 
   - 后端: Spring Boot → **FastAPI (Python 3.13)**
   - 数据库: MySQL → **PostgreSQL 15**
   - 消息队列: RocketMQ/RabbitMQ → **APScheduler（先不引入MQ）**
   - 缓存: Redis → **保持Redis，但需扩展缓存策略**

2. **组件状态重新标注**（见§2.1详细对照）:
   - "飞书消息组件" → ❌ 待开发（非已有）
   - "向量检索组件" → 删除（一期不做）
   - "Webhook网关组件" → 删除或延后（非必需）

3. **新增前置依赖项**:
   ```
   🔴 P0: 飞书通知基础组件（feishu_client + notification_service）→ 阻塞所有飞书集成
   🔴 P0: 飞书文档API封装 → 阻塞U11/U12
   🔴 P0: 北森写操作API确认 → 阻塞U13/U14，可能改变整个方案
   🟡 P1: org_unit权威建树改造（spec 007）→ 阻塞组织查重
   🟡 P1: L2数据范围全量覆盖（spec 005 G1-G5）→ 阻塞数据权限
   ```

### 6.2 方案一专属修正

1. **去除不必要的重型组件**:
   - 工作流引擎 Activiti/Temporal → 简单状态机（pending→confirmed→executed）
   - 消息队列 RocketMQ/RabbitMQ → 同步调用 + APScheduler 异步任务
   - Elasticsearch → 暂不需要，PostgreSQL JSON查询足够

2. **DAG排序**: 6.4 Step 2中的DAG排序逻辑可简化为：
   ```python
   # 固定优先级排序，不需要完整DAG引擎
   PRIORITY = {"create": 1, "rename": 2, "transfer": 3, "delete": 4}
   sorted_actions = sorted(actions, key=lambda a: PRIORITY[a.type])
   # + 依赖检查：调动依赖目标组织存在
   ```

3. **AI解析组件**: 应复用现有 `ai/capabilities.py` 注册机制，新增 `hr_adjustment.parse` capability，而非另起独立的AI解析服务。

### 6.3 方案二专属修正

1. **Skill细粒度拆分**: 按架构评审要求，将每个Skill的执行步骤拆分为独立的Capability调用，保留独立审计：
   ```
   hr-coordinator-skill
     ├── hr_adjustment.intent_classify（意图识别）
     ├── hr_adjustment.entity_extract（实体抽取）
     ├── hr_adjustment.validate_employee（员工查重）
     ├── hr_adjustment.validate_org（组织查重）
     ├── hr_adjustment.sort_actions（DAG排序）
     └── hr_adjustment.create_todo（生成待办）
   ```

2. **Agent SDK选型建议**:
   - 优先考虑与现有 FastAPI 可集成的方案（如 Dify API 模式或自建 Agent 编排）
   - 避免引入独立部署的 Agent 平台增加运维复杂度
   - 考虑直接基于现有 `orchestrator.py` + `capabilities.py` 扩展为 Agent 能力，而非引入全新框架

3. **安全增强**: 方案二的安全模型需要更强的"人机回环"保障：
   - Agent 的工具列表应通过代码白名单 + 数据库启停双重控制
   - 每次 Tool 调用必须记录完整参数和返回值
   - 增加 "Agent行为边界测试用例" 覆盖 Prompt注入、越权调用、重复确认绕过等场景

## 七、推荐实施方案（混合路线）

结合代码现状和两份PRD，建议采取**渐进式混合路线**：

### Phase A：补齐基础依赖（3-4周）
```
1. 飞书通知基础组件（feishu_client + notification_service）
2. 飞书文档API封装
3. 北森API写操作能力确认 → 若不可行，转人工北森操作+RPA辅助
4. 待办管理模块（通用设计，后续复用）
5. org_unit权威建树改造
```

### Phase B：核心能力开发（5-6周）
```
6. AI Capability注册：hr_adjustment.* 系列
7. 对话管理 + 校验引擎（复用现有AI orchestration）
8. 确认卡片 + 待办流转
9. HR Portal调整清单页面
10. 北森API网关 + 执行引擎
```

### Phase C：飞书群聊集成（3-4周）
```
11. 飞书事件订阅（@机器人消息回调）
12. 定时提醒（复用APScheduler）
13. 飞书文档主文档自动写入
14. 端到端联调测试
```

**关键决策点**: Phase A 完成后，根据北森API确认结果和飞书集成进展，再决定是否引入Agent Skill模式（方案二的能力）。早期先用方案一的确定性逻辑上线，验证业务闭环后再逐步引入Agent增强。

## 八、总结

### PRD的质量亮点
- 需求梳理完整，22条U需求逐条映射到方案
- 流程设计清晰，从群聊采集到Portal确认到API执行形成了完整闭环

- 安全考量充分，"人机回环"、权限分层、二次反查等设计原则扎实

### 必须修正的关键问题
1. **技术栈错误**: Spring Boot/MySQL/RocketMQ → FastAPI/PostgreSQL/APScheduler
2. **组件状态不准确**: 至少5个组件标注与代码实际状态不符
3. **北森API依赖未确认**: 写操作API是否存在是整个方案的生死线
4. **飞书机器人时机矛盾**: ADR明确延后但PRD以之为核心入口
5. **工作量严重低估**: 两方案均需上调30-40%

### 一句话建议

> 先确认北森API和补齐飞书基础能力，用方案一的确定性逻辑快速上线业务闭环（Web端对话工作台先行），再逐步引入方案二的Agent增强能力。不要在基础依赖未就位的情况下，直接追求"AI原生"或"Agent Skill"的架构理想。
