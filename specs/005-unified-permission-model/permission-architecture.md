# 统一权限管理方案（全局字段字典 + 四层权限 + 授权工具白名单）

> 状态：**已确认核心模型，待确认实施范围**　｜　版本：v2（2026-06-09）
> 目标读者：业务负责人（决策）+ 执行者（实施依据）
> 本文是后续重构的唯一蓝图，按「八、实施计划」分阶段执行。
> v2 相对 v1 的变化：订正 L4 真实现状、新增「授权工具白名单」字段权限模型、明确"无权字段直接隐藏"、确认证明/补偿金两工具的改造方向。

## 一、一句话目标

把现在"散落在每张表/每个字段、忘了配就出事、每个工具各写一套"的权限，收敛成**一套全局字段字典 + 一个统一裁决引擎**：
字段建一次、分类挂一次、权限树绑一次；物理表只做"认领"；工具只声明用途，不自己判断权限。
顺带补齐报表/数据集授权（L4），把数据范围"默认放行"改成"默认拒绝"，并用"授权工具白名单"解决"HRBP 无薪酬权限却要算补偿金"这类跨工具诉求。

## 二、现状诊断（已确认的事实）

### 2.1 权限分四层，现状参差

| 层 | 控制什么 | 现状 | 问题 |
|---|---|---|---|
| **L1 功能权限** | 能进哪些菜单、点哪些按钮 | ✅ 完整（`role_menus`，V/C/U/D/E，多角色取并集） | 无 |
| **L2 数据范围** | 能看哪些行（哪些人/部门/成本中心） | ⚠️ 有引擎，接入点错位 | 逐字段标 `scope_role`，**易漏、默认放行（fail-open）** |
| **L3 字段/列权限** | 能看哪些字段、值是否脱敏 | ⚠️ 有脱敏，但模型不足 | 敏感逐表标；只有"看/不看"两态，无法表达跨工具特许 |
| **L4 资源授权** | 哪些报表/数据集能用 | 🔴 **比想象的更不完整** | 见 2.4 |

### 2.2 已确认的安全漏洞（L2，fail-open）

- 7 张业务表，**只有 1 张（实时花名册）配了 `scope_role`，其余 6 张全空**。
- 后端逻辑：表没配 `scope_role` → `return true()`（全表放行）。文件：`backend/app/permissions/scope_filter.py:231-234`、`246-249`。
- 结果：**工资表（含税前工资成本、各级部门、工号姓名）对所有非超管用户裸奔**。
- 当前实际泄露面窄（仅 1 个受限用户 rita，授权"全量数据"、无人员级标签），但**新增受限用户或新接敏感表时，越权会静默发生**。
- 旁证：数据集 ACL 是 fail-closed（没配=仅创建者），同系统两套相反默认值，说明 L2 默认值当初没设计好。

### 2.3 一个澄清：报表**没有**重复存字段定义

- 字段真身只有一处：`table_columns`。报表 `config` 只存字段 **code 引用**，运行时回查元数据。
- "字段重复"在数据层不成立。真正要解决的是：**同一业务概念（如"工号"）在不同表里是各自独立的记录**，无法统一治理与控权。

### 2.4 L4 真实现状（已订正，比 v1 更不完整）

| 对象 | 后端鉴权 | 前端配置入口 | 实际效果 |
|---|---|---|---|
| 数据集 | ✅ 有 `_can_access`（`datasets/router.py:144`） | ❌ **无**（`DatasetEdit.vue:188` 写死 `acl: []`，注释"先不在设计器里配 ACL"） | 退化成仅创建者/超管可用 |
| 报表 | ❌ **未接**（列表/详情/run/export 都没按 `report_acl` 过滤） | ❌ 无 | 仅靠 `owner_id` + `is_published` |

- 即：**后端有鉴权能力但没人能配，报表连后端校验都没接上。** L4 整体处于"半成品 + 无入口"。

### 2.5 字段分类脱敏在数据集场景**已生效**（既有能力）

- 数据集多表查询取数时，对每张表算"该用户需脱敏哪些列"并遮值。文件：`sql_builder.py:886-908`、`masker.py:90-94`。
- 即"用户A只有基础分类权限、薪酬列被脱敏成 `******`"这条链路是通的——但**脱敏是遮值，列仍显示、数据仍出库**，强度偏弱（见 2.6 的改进方向）。

### 2.6 工具权限"各写各的"，是不一致的根源（已确认）

无全局拦截层，每个工具自己决定调不调权限函数，导致策略冲突：

| 工具 | 对"基本工资"的处理 | 问题 |
|---|---|---|
| **证明开具** | 直接取 `basic_salary`/`target_bonus` 渲染，**全程不调 `get_sensitive_columns`**（`income_certificate.py:68-76`） | **漏管**：无薪酬权限者只要能开证明就能反推工资 |
| **补偿金计算** | 开头即判断 `if "基本工资" in sensitive_cols: 403`（`tools/router.py:930-932`） | **过严**：把"用于计算"当成"查看"挡死，**HRBP 现在根本算不了补偿金** |

> 同一个字段，一个完全不管、一个管得过死——这就是缺统一引擎的代价，也是本方案要解决的核心痛点之一。

## 三、目标架构（一）：全局字段字典

### 3.1 模型总览

```
①全局字段字典 global_fields（新建，一次性定义共享/管控字段）
   工号 / 一级部门 / … / 成本中心编码
   每个全局字段自带：标准名、类型、维度度量、权限角色(绑哪棵树)、字段分类(诉求2)
        ▲ 认领 (N→1)
        │
②物理列 table_columns（保留，加一列 global_field_id 指向①）
   花名册.工号 ┐
   工资表.工号 ┼─► 都认领到 全局"工号"
   分摊表.工号 ┘
        │
③权限树（保留）：org_tree、cost_center_tree
   全局"各级部门/组织节点" 绑 组织树；全局"成本中心编码" 绑 成本中心树
```

### 3.2 三条规则

1. **字段统一（诉求1）**：多个物理列**认领到同一个全局字段**（N→1，不是逐字段改名）。组织树只需绑全局"部门"一次，任何数据集只要含认领了该全局字段的列，过滤自动生效——**不用管是哪张表**。

2. **分类随字段走（诉求2）**：全局字段**新建时即挂字段分类**。物理列一旦认领，**自动归入该分类**，无需逐表标敏感。

3. **可选回退、双默认值**：
   | 场景 | 没认领全局字段时 | 默认策略 |
   |---|---|---|
   | **显示 / 报表取数** | 用物理列自己的名称 | fail-**open**（无害，兼容现有报表，不影响拉取）|
   | **权限过滤** | 声明要按某棵树管却找不到对应全局字段 | fail-**closed**（拒绝，不放行）|

### 3.3 不必全量治理（控制工作量）

- **只对"跨表共享 + 与权限/树/分类相关"的字段建全局**：工号、各级部门、成本中心编码、用工类型/主体、人员等。
- **纯表内明细**（税前工资成本、各项补贴…）仍自动发现，**永不参与权限过滤**，无需建全局。

## 四、目标架构（二）：字段权限裁决 —— 授权工具白名单模型【本次核心】

### 4.1 为什么不是"让工具自己判断"

工具自己判断权限正是 2.6 冲突的根源。正确做法：**工具只声明"我要拿这个字段干什么"，是否放行由一个全局引擎统一裁决。** 判断逻辑收归一处，新工具自动纳管。

### 4.2 最终裁决规则（已确认）

去掉"敏感分级"，只用一个维度：**字段分类 → 授权工具白名单**。判断顺序是"先看有没有分类权限，再决定白名单要不要管"：

```
用户对该字段分类有权限?
├─ 有  →  任何工具/数据集都正常显示原值、可用（白名单无关，不受任何限制）
└─ 无  →  当前工具在该分类的「授权工具白名单」里?
          ├─ 在(如 补偿金计算 / 证明开具) →  原值正常显示、可用
          │                                （这正是授权该工具使用此字段的目的）
          └─ 不在(通用数据集 / 自助报表)  →  该字段完全不可用，连聚合也拒
```

**一句话**：白名单是"**无权者的特许通道**"，不是对所有人的限制。
- 有分类权限的人 → 不经过白名单，处处可见。
- 无分类权限的人 → 只有白名单工具能用该字段，且在白名单工具内**原值可见**（因为这是授权它的目的；进了白名单还脱敏，工具就没法干活）。

### 4.3 三场景验证（全部对齐）

| 场景 | 有薪酬权限? | 工具在薪酬白名单? | 结果 |
|---|---|---|---|
| 有薪酬权限 HR，任意场景（含通用数据集分析工资） | ✅ | — | ✅ 正常显示原值 |
| HRBP（无薪酬权限）补偿金计算 / 证明开具 | ❌ | ✅ | ✅ 工资构成、补偿明细、证明金额**正常显示**、可计算 |
| HRBP 在全量数据集按成本中心**聚合**薪酬 | ❌ | ❌ | ❌ 薪酬字段不可用，**聚合也拒** |

### 4.4 通用场景下"无权字段"的处理（已确认）

- **直接隐藏**：通用数据集/自助报表里，用户无权且工具不在白名单的字段，**根本不出现**（字段选择器里选不到、结果里无此列）。比脱敏更彻底，也符合"在数据集里就不该看到该字段"的诉求。
- **脱敏 `******` 仅作兜底**：仅用于"别人建好、已分享的旧报表里残留了无权列"的情况，避免报表报错或静默泄露。不再作为主要保护手段。

### 4.5 对两个问题工具的改造方向

- **证明开具**：纳入薪酬分类白名单。改造后——有薪酬权限者照常；无权限的 HRBP 因证明开具在白名单，可开具带真实工资的证明（其用途就是印工资），并由 L1 菜单权限 + L2 数据范围兜底"谁、能给谁开"。**补上当前"完全不受管控"的旁路（改为受白名单这套统一规则约束）。**
- **补偿金计算**：纳入薪酬分类白名单。改造后——删掉 `if "基本工资" in sensitive_cols: 403` 的粗暴拦截，**HRBP 可在其数据范围内正常计算补偿金、查看工资构成与结果**。

## 五、L4 资源授权（订正后：数据集 + 报表都要补前端入口）

### 5.1 概念

L1–L3 管"字段/行/菜单"；L4 管"一个保存下来的成品对象（报表、数据集）谁能用"。类比：L1 决定能不能进"报表中台"菜单；L4 决定进去后**哪几张报表/数据集对你可见**。

### 5.2 要补的内容

1. **数据集**：后端 `_can_access` 已就绪，**补前端授权入口**（在 `DatasetEdit.vue` 把写死的 `acl: []` 换成真实可配的角色/用户授权 UI）。
2. **报表**：后端实现与数据集对齐的 `_can_access(user, report)`，并接入**列表过滤 / 详情 / 运行 `/run` / 导出 `/export.csv`** 四处；前端补授权入口。
   - 规则：创建者/超管可访问；`report_acl` 命中角色/用户可访问；已发布按既定范围；无 ACL 且未发布 → 仅创建者可见（fail-closed）。

## 六、数据模型变更

### 6.1 新建表

**`global_fields`（全局字段字典）**

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT PK | |
| code | VARCHAR(64) UNIQUE | 全局字段编码，如 `emp_id`、`dept_l1` |
| label | VARCHAR(128) | 标准显示名 |
| data_type | VARCHAR(16) | string/number/date/... |
| agg_role | VARCHAR(16) | dimension/measure |
| scope_role | VARCHAR(32) NULL | 权限角色：绑哪棵树/维度（org_node_code/cc_code/person/employment_type/employment_entity；NULL=不参与权限）|
| category_id | BIGINT FK NULL | 关联 `field_categories`（诉求2）|
| description | TEXT NULL | |
| created_at | TIMESTAMPTZ | |

**`field_category_tool_whitelist`（字段分类 → 授权工具白名单）**【本次新增核心表】

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT PK | |
| category_id | BIGINT FK | 关联 `field_categories` |
| tool_key | VARCHAR(64) | 工具标识，如 `compensation_calc`、`income_certificate`（与工具用途声明对应）|
| | | UNIQUE(category_id, tool_key) |

> 含义：某字段分类，允许哪些工具在"用户无该分类权限"时仍使用其字段。

### 6.2 改造现有表

**`table_columns` 增加一列**

| 字段 | 类型 | 说明 |
|---|---|---|
| global_field_id | BIGINT FK NULL | 认领指向 `global_fields.id`；NULL=未认领（走回退）|

- 原有 `scope_role`、`is_sensitive`、`column_label` **保留**，作为"未认领时的回退值"。
- 取值优先级：**认领了全局字段 → 用全局字段的 label/scope_role/category；未认领 → 用物理列自身值**。

### 6.3 权限引擎改造（逻辑）

- **`scope_filter.py`**：过滤列优先按"认领了某 `scope_role` 全局字段的物理列"解析；**fail-open → fail-closed**（声明按树管却解析不到列 → `false()`），另设"显式免控白名单"声明哪些表无需数据权限。
- **`masker.py` → 升级为统一字段裁决**：实现 4.2 的判断（有权限 / 无权限×白名单），输出每个字段对当前(用户,工具,资源)的三态结果：`可见原值 / 隐藏 / 脱敏兜底`。所有取数路径（data、reports、datasets、tools）统一调用它，替代各工具手写逻辑。
- **工具用途接入**：补偿金、证明开具等工具传入自己的 `tool_key`，引擎据此查白名单。通用数据集/报表传入"通用查询"标识（不在任何白名单）。

## 八、实施计划（分阶段，每阶段可独立验收）

> 原则：先止血、低风险优先；动字段模型的根放在报表体量最小的当下窗口（现 1 报表 / 2 数据集）。

### Phase A — L4 资源授权补齐（数据集前端入口 + 报表后端校验+前端入口）
- 文件：#10、#18、#19
- 验收：数据集/报表可配授权；非授权用户列表看不到、直接访问 `/run` `/export` 返回 403。

### Phase B — 数据范围 fail-open → fail-closed（安全修复）
- 文件：#8（+ 免控白名单）
- 验收：工资表对受限用户不再全表可见；正常用户访问不受影响。
- ⚠️ 上线前需逐表确认"哪些必须按树管、哪些显式免控"，防误伤。

### Phase C — 全局字段字典 + 认领映射（字段统一，诉求1）
- 文件：#1–7、#14、#16–17、#20–21
- 验收：花名册/工资表"工号"认领到同一全局字段，改全局名两处同步生效；老报表照常拉取。

### Phase D — 统一裁决引擎 + 授权工具白名单 + 分类继承（诉求2 + 核心模型收口）
- 文件：#1(白名单表)、#3、#9、#11、#12、#13、#15
- 验收：
  - 全局字段挂分类 → 物理列认领后自动归类；
  - HRBP（无薪酬权限）在补偿金/证明可正常用工资、看构成与结果；
  - HRBP 在通用数据集**看不到**成本中心薪酬字段、聚合被拒；
