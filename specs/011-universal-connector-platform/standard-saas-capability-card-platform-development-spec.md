## 0. 评审后架构决策

1. **方向确认**：UCP 按“受控企业 iPaaS”方向演进，不再限制为仅 HR Portal 内的狭义接入能力；但不建设无边界、可任意访问互联网的通用 iPaaS。
2. **MVP 优先**：首期只验证“标准 SaaS 接入后，用户按业务能力启用和编排，不手工创建原始 API 资源”的闭环。
3. **三层接入策略**：预置标准能力包 → 受控 OpenAPI/HTTP 能力卡 → 自定义连接器/SDK。三层不可混为同一种用户交互。
4. **分层对象模型**：连接器能力包、系统连接、业务对象、操作定义、系统能力卡、实例资源必须分层；`UcpResource` 仅继续承担实例资源。
5. **兼容优先**：本决定覆盖旧实施计划中“开发期不做兼容”的表述。现有 `DataSource`、`UcpResource`、流水线、飞书在线表格和生产连接不自动迁移、不受中断。
6. **单一事实来源**：共享连接器目录、`AdapterDefinition`、连接器能力包不得长期平行维护。后续以“连接器能力包”为上层注册中心，旧目录和 AdapterDefinition 逐步转为其兼容投影。

### 0.1 整体规划校准：一套入口、三类接入、两种配置对象

本规格的首要目标不是“为飞书招聘多加几个 API 页面”，而是纠正当前所有系统都落入“添加数据资源”的错误心智。以后任何新系统必须先由平台目录标识接入类型，再进入对应的配置路径：

```text
数据连接 → 接入系统 → 添加系统
                    ↓
              平台连接器目录判定
      ┌─────────────┼──────────────┐
      ↓             ↓              ↓
标准 SaaS       实例资源        受控自定义 API
系统→凭证→      系统→凭证→      系统→凭证→
业务能力→编排   具体对象→编排   管理员审批能力→编排
```

| 接入类型 | 典型系统 | 用户配置对象 | 平台一次性建设对象 | 禁止的错误交互 |
|---|---|---|---|---|
| 标准 SaaS | 飞书招聘、企业滴滴、北森已标准化业务 API | 系统、凭证、业务筛选参数、启用状态 | Connector Package、业务对象、操作定义、Scope/Schema/限流/脱敏 | 为每个原始 API 创建 `UcpResource`、填写 Adapter Code 或 URL |
| 实例资源 | 飞书在线表格/多维表格、北森报表、数据库、文件 | 企业实际表格链接、Report ID、数据库实例、文件位置 | 连接方式、链接解析器、Schema 驱动表单、适配器 | 把所有同类表格合并为一个没有具体实例的资源 |
| 受控自定义 API | 企业内部 REST、供应商特殊接口 | 经管理员审批后的 Host、认证引用、Schema、责任人 | HTTP 执行器、安全策略、OpenAPI 导入/发布流程 | 普通用户任意填写 URL/请求头/Token 后直接运行 |

**统一复用边界**：数据仓库的“入仓来源”与数据连接的“接入系统”继续独立可用，但共同消费同一个“接入类型目录、凭证策略、链接解析、字段 Schema、测试状态和脱敏规则”组件。两边不得再各自维护飞书表格、北森报表、HTTP 的下拉选项和配置表单；北森其他 API 按已标准化业务能力或受控自定义 API 规则扩展，业务落库、资产管理和 UCP 编排仍各自保持职责边界。

**标准 SaaS 的关键规则**：飞书应用拥有 Scope 不等于平台可立即显示任意 API。平台先发布一个有版本、有负责人、有输入输出 Schema 的能力包；企业管理员仅配置凭证并启用已发布能力。能力发现是“预置能力集合 × 安全验证结果”，不是自动扫描全部第三方 API，也不自动生成 `UcpResource`。
### 0.2 现有模块边界：连接负责生产、仓库负责使用、事件触发流程

本次不是新建“数据连接版数据资产”，也不要求数据仓库为本需求增加配置交互。数据连接以可复用流程组件生产数据，并把结果写入数据仓库；数据仓库保持独立，只负责接收、登记、查询、治理和使用资产。

```text
数据连接 → 流程编排（数据生产与自动化）
  北森读取组件
      → 记录循环组件
      → 按键查询组件（飞书招聘 Offer）
      → 通用合并/字段映射组件
      → 数据仓库资产写入组件
                                      ↓
数据仓库 → 已接收的数据资产（保持现有功能和页面）
  表、字段、质量、血缘、查询、报表、权限、资产目录

数据连接 → 事件处理 → 事件触发器 → 已发布流程 → 运行中心 / 死信
```

- **数据连接的责任**：配置连接、凭证、标准业务能力、流程节点、字段映射、触发方式、重试、死信和目标资产写入。北森→飞书 Offer 是第一个模板，抽象后可复用为“外部数据读取 → 按键补全 → 合并 → 资产写入”流程。
- **数据仓库的责任**：继续独立提供“数据资产 → 来源于开放 → 配置入仓来源”等现有能力，也接收数据连接流程的资产写入结果；本需求不在数据仓库新增“数据补全/富化”页面、Tab 或规则配置。数据连接写入资产后，数据仓库按既有机制登记和展示。
- **通用资产写入组件**：流程的末端节点不是专用北森或飞书逻辑，而是受控 `WarehouseAssetSink`：选择目标资产/新资产标识、字段映射、写入模式（append/upsert/replace）、主键、批次标识和权限策略。该组件是数据连接与数据仓库的唯一集成边界，不复制数据仓库功能。
- **事件中心与流程编排的边界**：事件中心是事件事实和入口治理层，不是业务步骤编辑器；流程编排是唯一执行步骤定义处。一个事件可命中多个触发器，一个触发器只能绑定一个已发布流程；同一事件过滤、重试和失败策略不得在事件中心、生命周期规则和流程画布重复配置。
- **生命周期规则的边界**：飞书→滴滴规则是流程模板/受控节点的业务参数，而不是平行事件执行引擎。现有 Webhook 到生命周期服务的直接分发仅作兼容路径；目标架构由事件触发器创建 Pipeline Run，并将 `event_id`、`trace_id`、规则版本和步骤结果统一关联。
- **UI 改造原则**：仅扩展现有 `/ucp/systems`、`/ucp/pipelines`、`/ucp/events`、`/ucp/runs` 页面及其详情/抽屉；`/warehouse/assets/:tableName` 不因本需求新增配置入口或被替换。
## 1. 背景与目标

- 背景：当前 UCP 以“系统 → 资源 → 凭证 → 流水线”为主。`UcpResource` 的粒度为一张表或一个 API，适用于飞书表格、多维表格、北森报表、文件、通用 HTTP、自建 API 等需要指定企业具体对象的场景。
- 核心问题：飞书招聘、滴滴、北森标准 API 等标准 SaaS 已具有稳定认证、Scope、接口文档与业务对象。用户接入系统并完成授权后，仍要逐个创建资源、选择 Adapter、理解资源编码，造成重复配置、错误心智和治理碎片。
- 长期目标：UCP 成为受控企业 iPaaS 底座，支持预置连接器、受控 API 能力卡、OpenAPI 导入、自定义连接器、编排、审计、安全和治理。
- MVP 目标：以飞书招聘和企业滴滴为两条业务试点，完成“接入系统 → 配置凭证 → 校验 → 展示预置业务能力 → 启用/编排 → 运行审计”的通用闭环，并分别落地“北森待入职数据资产 → 飞书 Offer 薪酬同表富化”和“飞书入离职事件 → 企业滴滴账号生命周期”。
- 非目标 / 不做范围：
  - MVP 不做 OpenAPI 导入、通用动态 HTTP 执行器、任意自定义 API、通用可配置写 API、Webhook 平台和 SDK；企业滴滴的创建/停用/删除仅作为已有生命周期规格中的封闭高风险能力，在正式接口、审批、幂等和审计条件满足后启用，不构成通用写操作入口。
  - MVP 不依赖第三方提供可枚举 Scope 的接口；以预置能力定义和安全测试为准。
  - MVP 不迁移旧资源，不删除原有数据仓库入仓来源。
  - 不在代码库配置真实生产凭证。

## 2. 目标领域模型与现状衔接

```text
连接器能力包 Connector Package（平台定义）
├─ 认证策略 / Host 白名单 / 版本
├─ 业务对象 Object（Offer、应聘者、职位）
└─ 操作定义 Operation（查询列表、查询详情、写入、事件）
        ↓
系统连接 System Connection（企业实例）
├─ 系统信息
├─ 凭证
└─ 授权与连接状态
        ↓
系统能力卡 System Capability（用户启用实例）
├─ 操作定义引用
├─ 业务参数
├─ 测试/发布/版本状态
└─ 运行审计
        ↓
流水线 Pipeline
```

同时保留实例资源路径：

```text
实例资源 Resource Instance
├─ 飞书在线表格的具体链接
├─ 飞书多维表格的具体 App/Table/View
├─ 北森具体 Report ID
└─ 自建 API / 数据库的企业具体配置
```

现状衔接原则：
- `UcpSystem` 继续承载企业接入系统。
- `UcpCredential` 继续承载密钥与环境。
- `UcpResource` 标记为实例资源 / 兼容资源，不再作为标准 SaaS 原始 API 的唯一建模方式。
- 现有 `AdapterDefinition` 和适配器注册表在 MVP 继续复用；能力包通过映射引用既有 Adapter，不立即重写执行引擎。
- 数据仓库 `DataSource` 保持独立运行；后续仅消费共享连接器能力，不成为 UCP 的上游依赖。

## 2.1 受控企业 iPaaS 边界与分级准入

“受控”不等于限制平台能力；目标是逐步支持广泛企业 API，但任何 API 必须按风险、协议成熟度和业务副作用进入对应准入路径。禁止将“用户可填写任意 URL”作为平台能力定义。

| 准入层级 | 接入对象 | 允许能力 | 准入和执行规则 |
|---|---|---|---|
| L1 预置标准能力包 | 飞书招聘、滴滴、北森等高频系统 | 预定义读操作、事件、经审批的写操作 | 平台维护 API/Scope/Schema/限流/脱敏；用户只启用业务能力 |
| L2 受控 OpenAPI 能力卡 | 有效 OpenAPI 3.x 的 JSON REST API | 白名单 HTTPS 的只读 GET/查询型 POST | 导入生成草稿；管理员确认语义、分页、敏感字段后测试发布 |
| L3 受控自定义 API | 企业内部或无规范 API | 经审批的自定义读操作 | 主机、认证、请求 Schema、响应 Schema、负责人必须登记 |
| L4 高风险操作 | 写 API、账号变更、状态变更、通知、删除 | 受控写入、Webhook、复杂认证 | 幂等、审批、试运行、发布窗口、补偿、审计、告警必需 |
| L5 自定义连接器 | 文件流、SOAP、GraphQL、双向证书、专有协议 | 平台扩展能力 | SDK/代码实现、代码审查、版本发布、契约测试 |

安全和运行时强制规则：
- Host 白名单、DNS/IP 二次校验、禁止回环/私网/metadata 地址、禁止未校验重定向。
- 凭证只能从系统凭证库注入；用户不得在请求头、参数或模板中保存 Token。
- 读操作默认可测试；写操作默认不执行真实调用，必须走预览、审批、试运行和幂等策略。
- 每次执行都执行策略校验，不能只在创建或发布时校验。
- 每个能力定义均需负责人、风险等级、数据敏感等级、限流、重试和版本；无负责人或无版本不得发布。

## 3. 用户场景与交互
### 3.1 飞书招聘 MVP

```text
数据连接 → 接入系统 → 新建飞书招聘
1. 填写系统名称、负责人、环境
2. 配置飞书 App ID / App Secret
3. 测试连接
4. 自动进入“业务能力”页
5. 启用 Offer / 应聘者 / 职位查询
6. 在流水线画布中选择已启用能力
```

能力中心界面：

```text
飞书招聘
连接状态：已连接 / 待校验 / 失败

业务能力
✓ 查询 Offer                已验证，可启用
✓ 查询应聘者                已验证，可启用
○ 查询职位                  待测试
× 查询面试安排              MVP 暂未支持

[重新校验连接] [启用能力] [查看测试记录]
```

用户可见字段只包括：业务能力名称、筛选参数、分页、输出字段、测试状态、授权/错误说明。不得要求填写 API URL、Adapter Code、Token、JSONPath。

### 3.2 实例资源交互保持不变

飞书在线表格、多维表格、北森报表和自建 HTTP 继续走“添加具体资源”。飞书表格继续使用“粘贴链接并自动解析”的简化配置，不因能力卡 MVP 改动。

### 3.3 后续 OpenAPI 交互（非 MVP）

```text
能力中心 → 添加能力 → 导入 OpenAPI
→ 选择操作 → 自动生成草稿
→ 管理员确认业务对象、动作、分页、敏感字段、风险等级
→ 测试 → 审批/发布
```

### 3.4 MVP 页面级交互契约（开发必须遵守）

#### 可打开的技术 UI 示意图（本规格的一部分）

静态蓝图文件：`ui-blueprints/ucp-existing-module-boundary-wireframes-v1.3.html`。这是当前唯一有效版本，必须用浏览器直接打开并作为开发、测试和产品验收的共同基线；它以既有模块为宿主，包含以下五页，不是仅供参考的文字描述：

| 蓝图页 | 冻结的关键决策 |
|---|---|
| 既有接入系统列表 | 标准 SaaS、实例资源、受控自定义 API 从同一入口分流；保留系统、资源、凭证和运行记录 |
| 既有系统详情 / 业务能力 Tab | 飞书招聘按 Offer/应聘者/职位展示能力卡；实例资源系统继续显示资源 Tab |
| 既有流程设计器 / 数据补全并入仓模板 | 北森读取、按投递记录 ID 查询 Offer、合并字段、`WarehouseAssetSink` 写入资产均在数据连接配置；数据仓库无需新增交互 |
| 既有事件处理与流程编排 | 事件中心 → 触发器 → 已发布流程 → 运行中心的职责、跳转和 Trace 关联明确可见 |
| 既有流程编排 / 滴滴模板 | 入职创建/恢复、离职停用/受控删除作为流程节点；映射异常禁止删除 |

页面原型中的字段、按钮、状态、跳转和禁用规则是技术交付要求；最终视觉可复用 HR Portal 的 Element Plus 和设计 Token，但不得改变上述信息架构和安全约束。

#### 页面一：接入系统列表

- 列表卡片必须区分标签：`标准能力接入`、`实例资源接入`、`受控自定义 API`；不得只显示抽象“系统类型”。
- 飞书招聘卡片主按钮为“管理业务能力”，飞书表格卡片主按钮为“管理表格资源”。同一页面不得将两者均命名为“添加资源”。
- 卡片状态展示顺序固定：连接状态 → 已启用能力/资源数 → 最近测试 → 最近运行 → 风险提示。

#### 页面二：新建飞书招聘向导

```text
步骤 1 系统信息：系统名称、负责人、环境
步骤 2 应用凭证：App ID、App Secret、凭证环境
步骤 3 校验连接：连接结果、可验证能力、错误说明
步骤 4 启用能力：Offer / 应聘者 / 职位卡片
步骤 5 完成：进入能力中心或创建第一条流水线
```

- 不显示资源编码、Adapter Code、API URL、HTTP Method、Scope 原始代码作为必填输入。
- 点击“校验连接”进入加载态；校验过程中禁用下一步和凭证保存按钮，避免并发测试。
- 凭证失败时保留已输入非敏感字段，Secret 不回显；错误必须区分“凭证无效”“权限不足”“需要测试参数”“飞书服务异常”。
- 能力未被平台支持时显示“暂未提供此能力”，不得显示为“无权限”。

#### 页面三：飞书招聘能力中心

```text
顶部：系统名称 / 连接状态 / 最近校验时间 / 重新校验连接
指标：已启用、可启用、待验证、不可用
筛选：对象（Offer/应聘者/职位）、状态、风险等级、关键词
内容：按业务对象分组的能力卡
```

能力卡必须包含：业务名称、操作类型、状态、数据敏感等级、最近测试结果、最近运行结果、启用开关、配置入口。

状态与按钮规则：

| 状态 | 主按钮 | 次操作 | 禁止操作 |
|---|---|---|---|
| 已验证可用且未启用 | 启用能力 | 测试、查看说明 | 直接进入流水线 |
| 已启用 | 配置 / 查看运行 | 测试、停用 | 删除被流水线引用的能力 |
| 待补充测试参数 | 配置参数 | 查看示例 | 标记为权限不足 |
| 权限不足 | 查看授权说明 | 重新校验 | 启用、执行 |
| 平台暂未支持 | 查看支持计划 | 提交需求 | 伪造可配置入口 |
| 第三方异常 | 重试校验 | 查看诊断 | 修改权限状态 |

#### 页面四：能力配置与测试抽屉

- 普通管理员仅见业务参数、分页、输出字段、测试样例选择；技术定义默认只读且仅向 `ucp.admin` 展示。
- “测试能力”固定展示四段结果：鉴权、授权、参数、响应结构；每段显示成功/失败、耗时、脱敏摘要、可执行下一步。
- 没有真实样例参数时，允许保存为“待验证”，但禁止启用到流水线；不得强迫用户填写 API 技术参数。
- 修改已启用能力参数时必须先提示受影响流水线数量，并提供“保存草稿 / 发布变更”两种动作；MVP 若不做版本发布，则禁止修改被流水线引用的关键参数。

#### 页面五：流水线选择器

```text
数据来源
├─ 标准业务能力
│  └─ 飞书招聘 → Offer → 查询 Offer
└─ 实例资源
   └─ 飞书在线表格 → 员工花名册
```

- 不显示原始 URL、Adapter Code、系统内部 ID。
- 只列出已启用且验证通过的能力；停用、未授权、草稿能力在搜索结果中显示但不可选，并给出原因和跳转入口。
- 选择能力后自动展示可配置业务参数及能力版本快照，不允许用户在画布内修改认证和 Host。

#### UI 验收原则

- 标准 SaaS 用户在主路径中不需要理解“资源”“适配器”“OpenAPI”“Scope 原始编码”。
- 实例资源用户仍可清晰完成表格、报表、URL 等具体对象配置。
- 所有加载、空态、无权限、第三方失败、待参数、停用、影响范围状态必须有可行动文案，不能只显示技术错误。

### 3.5 首期必须交付的两条真实业务闭环

本期以两条可验证的生产业务链路检验平台抽象；它们共享 UCP 底层能力，但不能被错误地抽象成同一种“资源配置”。

#### 闭环 A：北森待入职人员数据资产富化飞书 Offer 薪酬

```text
北森报表 API
→ 数据仓库数据资产（待入职人员业务表）
→ 资产同步成功事件 / 增量富化任务
→ 按投递记录 ID（`application_id`）调用已启用的飞书招聘 Offer 查询能力
→ 将 Offer 薪酬及富化状态写回同一张数据资产表
→ 记录逐人结果、版本、Trace 和失败待处理项
```

- **用户确认的落点**：北森待入职信息与飞书 Offer 薪酬信息保存在数据仓库中**同一张业务数据资产表**。该表按待入职人员一行一条记录管理；北森字段和飞书字段必须使用固定前缀分组（如 `beisen_*`、`feishu_offer_*`、`offer_enrichment_*`），不得以 Offer 返回值覆盖北森原始字段。
- **必须保留的可追溯性**：同表是业务查询落点，不等于删除来源和运行审计。每次北森同步、每次人员富化、请求摘要、字段版本、失败原因和重试结果仍须在同步/流水线运行记录中独立保存；需要回放时可按 `trace_id`、同步批次和投递记录 ID 定位。
- **现有代码限制**：`backend/app/ucp/datasource_bridge.py` 的 `DATASOURCE_BRIDGE_ADAPTER` 当前只复用既有 `sync_to_table` 执行“拉取并入仓”，返回行数、摘要和 datasource ID；它不会把每位待入职人员记录输出到 Pipeline 上下文。因此，不能仅把该 Bridge 串到 Offer 查询节点后就宣称完成逐人富化。
- **MVP 实现决策**：数据连接流程以北森读取组件输出待入职人员记录，按稳定投递记录 ID 去重、限流并逐人调用飞书招聘能力，经通用合并/字段映射节点后，由 `WarehouseAssetSink` 一次写入目标数据仓库资产。该“外部读取 → 按键补全 → 合并 → 资产写入”模式必须抽成通用 Pipeline 组件，后续可复用到社保、背调、组织信息等补充数据；数据仓库不新增配置页面。
- **首期必须冻结的数据合同**：北森待入职判定、业务表名和主键、应聘者 ID 字段及格式、分页和最大批量；空 ID/重复 ID/多 Offer 的选择规则；薪酬字段清单与敏感等级；无 Offer 的业务语义；写回策略（仅更新飞书前缀字段）；幂等键（资产主键 + Offer 标识/版本）；单人失败继续策略、重试、死信和人工重跑；对账指标（输入人数、成功、无 Offer、失败、重复）。
- **上线前阻断项**：在业务表、应聘者 ID 映射、薪酬字段授权范围和无 Offer 规则确认前，只能交付能力测试预览，不得把“自动富化”标记为已上线。

#### 闭环 B：飞书员工事件驱动企业滴滴账号生命周期

```text
飞书员工入职 / 离职事件
→ 统一员工事件（稳定 employee_id、事件 ID、事件时间）
→ 配置化生命周期规则
→ 企业滴滴账号能力（查询 → 创建/恢复；查询 → 停用 → 审批后删除）
→ 外部账号映射、任务、审计、告警和死信
```

- 入职默认执行“查询账号 → 不存在则创建、已停用则恢复、已启用则幂等成功”；离职默认先停用，只有企业滴滴正式接口支持、保留期/审批规则满足且没有阻断条件时才删除。不得将“离职即删除”硬编码为专用逻辑。
- 该链路复用已有的飞书事件标准化、`UcpAccountLifecycleRule`、持久化任务、`ExternalAccount` 映射和 `DidiAccountClient`；滴滴仅作为一个目标账号系统。后续门禁、VPN、邮箱、曹操等通过相同规则模型接入，而不是复制飞书 Webhook 或账号状态机。
- 历史滴滴账号批量盘点、导入及人工待处理清单不属于当前上线阻断范围；当前只保留安全账号匹配和“无法确认绝不自动删除”的保护。该延期范围以 `feishu-didi-account-lifecycle-development-spec.md` 为准。

#### 两条闭环共享与隔离边界

| 层级 | 必须共享 | 必须隔离 |
|---|---|---|
| 连接层 | Connector Package、凭证、Host/Scope 策略、能力定义、测试和审计 | 北森/飞书招聘/滴滴各自的 API 合同和业务参数 |
| 编排层 | Pipeline、事件、运行、Trace、重试、死信、监控 | 数据资产富化的批处理语义；账号生命周期的状态机和审批语义 |
| 业务数据层 | 敏感字段分级、字段映射框架、幂等和对账能力 | 数据资产表不承载账号生命周期状态；外部账号表不承载 Offer 薪酬快照 |
| 用户交互 | 系统、凭证、能力卡、测试、运行记录的一致组件 | 数据资产富化配置页面与账号生命周期规则页面按业务任务分别呈现 |

### 3.6 Phase 0 必交付物：能力发现、数据合同与 UI 蓝图

X0201 不是一份原则性说明，必须产出以下可评审文件；缺任一项不得进入代码开发：

1. **能力发现口径**：不做自动 Scope 枚举；采用“预置能力定义 + 凭证认证 + 安全只读调用验证”。状态必须区分：平台未支持、待补测试参数、验证可调用、权限不足、第三方异常、无数据。
2. **两条业务数据合同**：北森输入、资产主键和应聘者 ID 映射、飞书 Offer 输出、同表字段分组/写回规则，以及飞书员工事件、滴滴账号字段、状态机、删除前置条件；每项附脱敏样例、字段敏感级别、错误码和幂等键。
3. **五页静态 HTML 或低保真原型**：接入系统列表、新建飞书招聘向导、飞书招聘能力中心、Offer 富化规则/运行详情、飞书→滴滴生命周期规则/运行详情。原型必须覆盖加载、空态、无权限、参数不足、第三方失败、停用、重试和成功反馈。
4. **状态矩阵与验收演示脚本**：至少可演示“北森一批待入职人员入仓→逐人查询 Offer→同表写回薪酬”和“飞书入职创建/离职停用滴滴账号”，并能从每条结果追溯到 Trace、规则/能力版本及脱敏请求摘要。
5. **兼容矩阵**：明确既有 `DataSource`、`UcpResource`、飞书在线表格/多维表格、已有 Pipeline、`ExternalAccount` 和生产连接不被自动迁移或中断。

## 5. 技术设计

### 5.1 数据库 / 数据模型

**Phase 1 MVP 新增：**

| 表 / 模型 | 核心字段 | 默认值与兼容 |
|---|---|---|
| `ucp_connector_package` | code、name、connection_mode、version、status、host_allowlist | 仅新增；不影响旧 AdapterDefinition |
| `ucp_operation_definition` | package_id、object_code、operation_code、name、adapter_code、required_scopes、input_schema、output_schema、risk_level | MVP 仅只读操作；`risk_level=read_low` |
| `ucp_system_capability` | system_id、operation_id、credential_id、enabled、connection_status、verification_status、runtime_config | 默认禁用；一系统同操作唯一 |
| `ucp_capability_test_run` | capability_id、status、request_summary、response_summary、error_code、trace_id | 仅保存脱敏摘要 |

**Phase 2/3 预留：** `ucp_capability_version`、`ucp_openapi_import`、写操作审批/幂等记录。

Migration 要求：所有新增表可 downgrade；禁止因删除系统能力而级联删除历史运行记录；旧 `UcpResource` 不迁移。

### 5.2 后端接口

| 阶段 | URL | Method | 权限 | 说明 |
|---|---|---|---|---|
| MVP | `/ucp/connector-packages` | GET | `ucp.systems:V` | 获取可接入标准能力包 |
| MVP | `/ucp/systems/{id}/capabilities/discover` | POST | `ucp.systems:E` | 使用凭证做安全只读验证，返回能力状态 |
| MVP | `/ucp/systems/{id}/capabilities` | GET/POST | `ucp.systems:V/C` | 查询 / 启用预置能力 |
| MVP | `/ucp/system-capabilities/{id}/test` | POST | `ucp.systems:E` | 用临时业务参数测试并返回脱敏摘要 |
| MVP | `/ucp/system-capabilities/{id}` | GET/PUT | `ucp.systems:V/U` | 查看和更新业务参数、启用状态 |
| Phase 2 | `/ucp/connector-packages/{id}/openapi-import` | POST | `ucp.admin:C` | 导入规范，生成草稿能力 |
| Phase 2 | `/ucp/system-capabilities/{id}/publish` | POST | `ucp.approvals:C` | 发布受控能力版本 |

错误分类必须区分：`CREDENTIAL_INVALID`、`AUTHORIZATION_UNVERIFIED`、`PERMISSION_DENIED`、`REQUIRES_TEST_PARAMETERS`、`THIRD_PARTY_UNAVAILABLE`、`OPERATION_NOT_SUPPORTED`。禁止将无数据或缺测试参数误报为缺 Scope。

### 5.3 业务逻辑

- 能力状态分三层，不能混用：
  - 授权状态：未验证、可调用、权限不足、未知；
  - 测试状态：未测试、通过、参数不足、第三方失败；
  - 发布状态：未启用、已启用、停用、草稿、待审批、已发布。
- MVP 能力由飞书招聘能力包预定义路径、认证、分页、响应处理和 Adapter 映射；用户只配置业务参数。
- MVP 复用既有适配器执行入口。若现有 Offer Adapter 不能满足列表/详情场景，允许补充飞书招聘专用 Adapter；不以 Generic HTTP 重写为前提。
- Pipeline 节点新增 `system_capability_id`，执行时快照 operation/version/参数，保证可追溯；旧 `resource_id` 节点继续运行。
- Phase 2 才实现 Generic HTTP：凭证注入、Schema 校验、Host 白名单、SSRF 防护、分页、响应提取、脱敏和审计。

### 5.4 前端与 UI/交互

MVP 页面调整：
- 新建系统第一步增加“接入模式”展示，但对预置飞书招聘自动选择“标准能力接入”，不让普通用户选择技术模式。
- 标准能力接入向导为：系统信息 → 凭证 → 校验连接 → 启用业务能力 → 完成。
- `SystemsTabView` 中，标准能力系统的资源 Tab 改为“业务能力”；实例资源系统仍显示“资源”。
- 能力中心按对象分组：Offer、应聘者、职位；卡片显示状态、最近测试、启用开关、配置入口。
- 流水线资源选择器增加“标准能力”分组，使用“系统 → 对象 → 操作”三级选择；旧资源选择器保留“实例资源”分组。
- 空态：未配置凭证、无可用能力、未支持能力、无权限、无测试样本均有清晰行动按钮。

### 5.5 权限、安全与外部系统

- MVP 仅支持只读飞书招聘能力，禁止写入、删除、状态变更。
- 真实凭证只能存放在 `UcpCredential` 加密字段，测试结果和日志不得回显 Token、App Secret 或完整 PII。
- 能力测试只允许预定义 Host 和预定义操作，按平台限流；任何参数不得拼接到未经校验的 URL。
- Phase 2 Generic HTTP 必须复用/强化现有 SSRF 防护、域名/IP 白名单、审计、Trace、死信与告警能力。
- OpenAPI 文档不应默认保存为“加密原文”；应按来源、敏感性、版本、哈希和访问权限管理，只有含凭证或内部敏感信息时才加密保存。

## 8. 验收标准

### MVP 业务验收
- 普通管理员不填写 URL、HTTP Method、Adapter Code，即可接入飞书招聘并启用 Offer、应聘者、职位查询能力。
- 从新建系统到启用首个 Offer 查询能力，标准演示环境不超过 10 分钟。
- 流水线可以选择已启用能力，运行失败显示业务能力名称和可追溯 Trace。

### MVP 技术与安全验收
- 现有 DataSource、飞书表格、UcpResource、旧 Pipeline 不受影响。
- 凭证和测试响应无明文 Secret、Token、完整 PII。
- 未验证、未启用、停用能力不能被流水线执行。
- 能力发现不能将无数据、缺参数错误判定为权限不足。

### 平台演进验收
- Phase 2 才允许管理员基于有效 OpenAPI 生成受控只读能力卡。
- Phase 3 才允许写操作或复杂认证，并且必须经过审批与运行治理。

## 9. 风险与兼容性

| 风险 | 等级 | 影响 | 应对方案 |
|---|---|---|---|
| MVP 范围膨胀为完整 iPaaS | 高 | 延迟首个业务价值 | 固定飞书招聘只读试点；OpenAPI/通用执行器后置 |
| Scope 与真实可调用混淆 | 高 | 错误能力状态 | 授权、测试、发布三层状态分离 |
| 新能力模型破坏旧资源 | 高 | 生产流水线中断 | 双路径并存，不自动迁移，完整回归 |
| 多个连接器注册中心并存 | 高 | 配置漂移、维护成本高 | Phase 0 明确上层注册中心和兼容投影 |
| 动态 HTTP 引入 SSRF / 数据泄露 | 高 | 安全事故 | Phase 2 才启用，白名单、DNS/IP 校验、审计、只读默认 |
| OpenAPI 质量差 | 中 | 草稿不可运行 | 限制规范范围、人工确认、测试后发布 |
| 写操作误执行 | 高 | 业务副作用 | 后置到 Phase 3，审批、幂等、试运行、回滚 |
