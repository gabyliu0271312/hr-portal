# 006 — 表格处理工具(表格合并 / 拆分)

> 状态:**方案待评审**(2026-06-20)
> 首期范围:**表格合并**之「多源异构社保数据 → 按人归集到标准模板」
> 拆分功能:留菜单位,本期不实现

---

## 0. 一句话定义

把来自不同渠道、结构各异的多个 Excel(社保/公积金/强积金/异地代缴等),按「姓名+证件」**按人横向归集**到一张**标准字段模板**总表;映射配置**可保存复用**,定期任务下次**自动识别、零确认**。

本质不是"表格合并",而是 **多源异构数据 → 按人归集到标准模板**。

## 1. 需求评估

### 1.1 为什么纯程序做不了,纯 AI 也不行
- **列名异构但有映射关系**:同一个语义(如"养老单位缴费")在不同表里列名不同,映射关系在人脑里,程序无法臆测。
- **7 种异构表头结构**:两行/三行表头、大量合并单元格、表头不在第一行(前几行是标题/单位名/比例说明)。
- **一个文件多个有效 sheet**:亲亲小保、异地代缴的目标数据不在第一个 sheet,且一个物理文件要拆成多个逻辑数据源。
- **结论**:纯程序硬编码 → 换格式即崩;纯 AI 逐行搬数据 → 会错会编会慢。
- **正确形态** = **AI/规则出“映射方案” + 程序确定性执行“搬运与归集”**。与本项目 specs/004 的 ChatBI「Text-to-QuerySpec 不走裸 SQL」同一哲学。
- **AI 接入纪律(硬约束)**:本工具的 AI 部分(出映射草稿)必须作为 specs/004 的受控 **Capability** 接入,走 `Orchestrator → Tool Wrapper → Schema/Policy Guard → Audit` 全链路;**不得绕过底座私接 `app/ai/provider.py` 或在业务模块拼 prompt**(004《AI 原生开发原则》§1.1 开发准入硬规则)。落地细节见 §4.6 / §4.7。

### 1.3 难点与对策

| 难点 | 对策(落到配置) |
|---|---|
| 多/三行表头 + 合并单元格 | 配置「表头行区间 + 是否拼接多行」;引擎按区间读、向右填充合并值 |
| 目标数据不在第一个 sheet | 配置「按 sheet 名匹配」;一个物理文件可拆多个逻辑数据源 |
| 按人合并(跨文件归集) | 配置「归集主键」,默认 `姓名+证件`;横向归集 |
| 噪声(费用通知单/合计行) | 配置「跳过 sheet / 跳过合计行(关键列为空或含'合计')」 |
| 金额冲突(同人同险种多源不一致) | **报异常**,不静默覆盖 |

## 2. 建议方案(通用组件 · 三层结构)

> **定位修正(2026-06-20):本工具是「任意多表按键归集的通用组件」,社保只是第一份输入数据。**
> 代码里**零业务字段**:标准字段、映射、派生规则全部是「用户配置/AI 草稿」的产物,存模板库,不写死在引擎。
> 加新场景(考勤/报销/任意归集)= 加一份配置,**永不改代码**。

### 2.1 标准字段(运行时定义,非代码预设)
- 工具自带标准字段清单为**空**。标准字段是**建模板时确定**的:用户上传文件 → AI 提议一组标准字段 → 用户确认/改 → 成为该模板的字段。
- 社保那套(养老个人/养老公司/…)是「社保归集模板」这个**用户配置**的产物,不是工具的一部分。换场景则字段完全不同。
- 系统注入来源标记列:`来源公司`、`来源文件`、`来源sheet`。

### 2.2 源映射配置(每个模板一份)
单数据源配置(JSON,见 §3):匹配规则 → 选 sheet → 表头行区间 → 列映射 → 派生字段 → 跳过规则 → 归集主键 → 聚合口径。
同结构文件共用一份(如 3 个日常申报共用一模板)。

### 2.3 执行引擎(确定性,只认抽象动作,不认业务含义)
引擎能力是通用原子操作,**完全不知道"养老""公积金"为何物**:
定位 sheet → 解析多行表头(合并填充)→ 按映射搬列 → 按表达式派生(拆列)→ 按主键归集 → 按口径聚合(sum/first/报异常)→ 校验 → 标来源 → 输出 xlsx + 异常清单。

### 2.4 AI 出初步归集模板 → 人确认(第一版核心能力,非附加)
**为什么第一版必须有**:工具交付他人使用,别人面对陌生表无法手工配 schema;没有 AI 出草稿,通用工具即废。

**接入形态(004 合规)**:本能力注册为 specs/004 的 `draft` 类 Capability `table_merge.suggest_mapping`,**只产映射草稿(JSON),不写任何数据**(`side_effect=draft_only`、`confirmation=normal`)。与代码中已有的 `codegen.suggest`(中文→英文标识符:AI 生成候选 + 规则兜底 + 人确认)完全同构,直接照其形态注册。元数据见 §4.7。

**流程(首次建模板)**:
1. 用户上传一批文件,**不预先配置任何东西**。
2. AI:读各文件各 sheet 表头(用已验证的多行+合并解析器)→ **跨文件聚类语义相同的列** → **提议一组标准字段** → 为每个文件生成「源列→标准字段」映射 → 识别需派生的列(如"只有合计+比例"→ 提议 `基数×比例` 拆分)→ 提议主键、聚合口径。
3. 产出**完整归集模板草稿**(标准字段 + 各源映射 + 派生 + 主键),非最终,是待确认稿。
4. **人确认关口**(关键):
   - 审 AI 提议的标准字段清单 → 改名/合并/增删(**字段控制权归人**)。
   - 审各文件映射 → **AI 标红低置信/口径歧义项**(如北京"医疗及生育保险/单位交"该不该并入`医疗公司`),人重点审这些。
   - 确认后固化为正式模板 → 之后每月走规则自动识别,零确认。

**口径歧义必须人兜底**:AI 跨文件聚类在"口径有歧义"处必错(医疗+生育合并 vs 分开),这不是 AI 能力问题,是业务口径只有人知道。故确认 UI 不是"扫一眼点确认",而是"AI 标出不确定项、人定夺"。


### 2.4 通用性边界(明确不做的)
- 第一版**只做「映射 + 按人归集 + 主键冲突报异常」**。
- **不**内置:公式计算字段、复杂过滤规则引擎、多表 join 之外的变换。
- 理由:n=1 时造万能引擎大概率猜错抽象。schema 预留扩展位,等第 2、3 个场景出现再沉淀。

### 2.5 已建模板增量接入源映射（已实现）

首次建模板用于从零生成“标准字段 + 主键 + 多源映射”的完整草稿；模板创建后，新增数据源不得重新生成或改写模板口径，而应复用既有模板的 `merge_keys`、`std_fields` 与 `aggregate`。

**流程：**

1. 模板创建者或超级管理员点击“新增映射表”。
2. 进入与首次建模板一致的批量拖拽上传向导，可上传多个 `.xlsx` 样表并填写业务背景。
3. AI 仅读取每个文件各 Sheet 的表头与模板字段元数据，为每种结构输出源映射草稿；不得读取或发送数据明细。
4. 用户只确认映射名称、表头识别特征、低置信提示和是否保留草稿；主键映射、字段映射、派生字段在保存后可从模板主界面继续编辑。
5. 服务端对全部草稿先校验后原子写入；任一条名称冲突或字段校验失败时，不允许部分新增。
6. 保存成功后返回模板列表，刷新映射数量。

**接口：**

- `POST /table-tools/templates/{tid}/mapping-drafts`：批量样表生成待确认草稿，不写库。
- `POST /table-tools/templates/{tid}/mappings/batch`：原子批量保存草稿。
- 单条维护继续使用 `/mappings`、`/mappings/{mid}` 和 `/mapping-draft`。

详细实现、权限、安全和验收证据见 `表格归集工具-已建模板新增源映射开发Spec.md`。
## 3. 配置 Schema(草案)

```jsonc
// 标准模板(目标字段) standard_template
{
  "template_id": "shebao_monthly",
  "name": "社保月度归集",
  "merge_keys": ["姓名", "证件号码"],   // 归集主键,可配;含"公司"则跨公司分行
  "fields": [                            // 标准字段,纯配置
    "养老个人", "养老公司", "医疗个人", "医疗公司",
    "失业个人", "失业公司", "工伤公司", "生育公司",
    "公积金个人", "公积金公司", "服务费", "残保金"
  ]
}

// 源映射(每种结构一份) source_mapping
{
  "mapping_id": "sz_daily_declare",
  "name": "深圳-日常申报明细",
  "template_id": "shebao_monthly",
  "match": {                            // 自动识别规则(多重)
    "header_signature": ["序号","姓名","证件号码","费款所属期起"], // 主:表头特征
    "filename_keywords": ["日常申报明细"],                        // 辅:文件名
    "sheet_name_keywords": ["申报明细"]
  },
  "sheet_select": { "by": "name_keyword", "value": "申报明细" },
  "header": { "start_row": 1, "end_row": 2, "join_multi_row": true }, // 表头区间
  "skip_rows": { "key_col_empty": "姓名", "value_contains": ["合计","小计"] },
  "key_map": { "姓名": "姓名", "证件号码": "证件号码", "公司": null },
  "column_map": {                        // 源列 → 标准字段(直接映射)
    "基本养老保险(个人)/应缴费额": "养老个人",
    "基本养老保险(单位)/应缴费额": "养老公司",
    "基本医疗保险(个人)/应缴费额": "医疗个人",
    "...": "..."
  },
  "derived_fields": [                    // 派生字段(通用表达式拆列,不写死代码)
    { "target": "公积金个人", "expr": "{缴存基数（元）}*{个人缴存比例}", "round": 2 },
    { "target": "公积金公司", "expr": "{缴存基数（元）}*{单位缴存比例}", "round": 2 }
  ],
  "derive_check": {                      // 可选:拆分校验,拆分项之和≈原合计,超容差报异常
    "sum_of": ["公积金个人", "公积金公司"], "equals_col": "金额合计（元）", "tol": 0.05
  },
  "aggregate": "sum",                    // 同人多笔/多源聚合口径:相加
  "version": 1
}
```

> 列映射的源列名以「解析后的有效表头」为准(多行表头拼接、合并填充后的结果)。
> **派生表达式用 `{源列}` 占位,直接复用公共公式引擎 `app/ai_formula`(报表/数据集同款求值器):支持 `+ - * / ()` 与 `IF/ROUND/MIN/MAX/SUM/ABS/CONCAT` 及自定义函数(CALC_TAX/SAFE_DIVIDE 等)**。`{列名}` 在引擎内部转成原生 `FIELD("列名")` 求值。彻底改造,无旧 `eval_expr`(生产未使用,不留兼容)。
>
> **公式引擎是公共组件**:`app/ai_formula` 在架构上已是顶层独立模块(reports / datasets / table_tools 平级复用),非报表私有。新场景要做"按公式计算字段",一律 `from app.ai_formula.formula_evaluator import evaluate_formula` + `executable_functions(db)` 复用,**不要再各自写求值器**。

## 4. 保存与复用机制(周期任务核心)

### 4.1 两层保存
- **模板库(结构级,长期稳定)**:保存「某类文件长什么样 + 怎么映射」。深圳社保局格式不变 → 永久复用。
- **任务方案(批次级,可选)**:保存「本次用了哪几个模板、输出哪些字段」,下月一键重跑。

### 4.2 自动识别(决定"零确认"的关键)
新文件进来,按**多重匹配**判定归属哪个模板:
1. **表头特征**匹配(主,最稳——列名通常每月不变)
2. **文件名关键词**(辅,文件名带月份会变,不能单靠)
3. **sheet 名关键词**(辅)
4. 都认不出 → 回退「人工识别+建议」,并提示**另存为新模板/新版本**

### 4.3 格式漂移处理(用户确认:表头相对稳定,变了就回退)
- 表头特征命中但**列对不上/缺列** → **主动报警**,绝不静默错位填数。
- 回到「A.识别+给建议」流程,用户确认后可**另存为新版本**(version+1),保留旧版。

### 4.4 理想周期流程
> 上传 11 个文件 → 系统按表头特征**全部自动识别+套用映射** → 用户扫一眼识别结果 → 直接跑 → 下载。
> **正常零确认;仅"格式变了/出新文件"才需介入。**

### 4.5 AI 分级介入(关键:能不用 AI 就不用)

把笼统的"识别"拆成三个动作,只有第三个真正需要 AI:

| 动作 | 是否需要 AI | 做法 |
|---|---|---|
| **认出"哪个已存模板"**(匹配) | **否** | 表头列名与模板库 `header_signature` 比对算重合度,纯字符串 |
| **定位"表头第几行 / 取哪个 sheet"**(结构定位) | **基本否** | 启发式:非空率高+字符串为主的行=表头;sheet 按关键词/行数;畸形表才兜底 |
| **给"陌生表"猜列映射**(语义映射) | **是** | 列名千变万化("单位养老""养老(单位)""养老保险单位部分"),规则字典枚举不完,靠 LLM 语义对齐 |

**分级触发表:**

| 场景 | 触发 AI? | 处理 |
|---|---|---|
| 命中已存模板(特征≥阈值) | 否 | 直接套映射,零成本零延迟。**月度任务 100% 走这条** |
| 命中但缺列/多列(漂移) | 否 | 报警 + 回退人工核对,可另存新版本 |
| 完全没命中(新渠道文件) | **是** | AI 读表头 → 生成映射草稿 → **人确认后存为新模板** |
| AI 置信度低/猜不准 | 降级 | 标红留用户手工指,手工结果存模板 |

**核心:AI 只在"首次遇到新结构"时介入一次,产出经人确认后立即固化进模板库;同种表第二次来就走规则匹配,不再调 AI。** 故即使工具多人使用,AI 调用量也很低——只发生在"新表型首次出现"那一刻。

- **可控**:AI 产出的是"映射草稿"非最终数据,必经人确认才生效(与 specs/004「AI 出方案、程序执行」一致);草稿生成走 Capability `table_merge.suggest_mapping`,全程过 schema/policy guard 与审计(§4.6–4.8)。
- **可解释**:每个文件识别结果标明命中方式(模板匹配 / AI 草稿 / 手工)。

### 4.6 AI 接入走 004 底座(不私接 provider)

004《AI 原生开发原则》§1.1 是硬规则:**任何 AI 相关功能不得绕过最小底座直接接模型或拼 prompt**。故本工具的 AI 调用链路固定为:

```text
AI 草稿请求 → ai/orchestrator(意图+能力解析) → Tool Wrapper(app/tools)
            → ai/provider(底层通道) → schema_validator(校验映射 JSON)
            → policy_guard(越权/SQL/URL/明细外泄拦截) → 草稿返回 → 人确认 → 引擎执行
            → audit(ai_tool_invocations 全程留痕)
```

- **不直接调** `provider.py::generate_json_openai_compatible()`,而是经 Capability `table_merge.suggest_mapping` 由 orchestrator 间接调用。`provider.py` 是底座最底层通道,不是业务可直接依赖的接口。
- **不复制** `scripts/gen_field_mapping.py` 的脚本式调用。该脚本是建表期运维工具、非 Capability;可参考其 prompt 范式,但接入必须经底座。
- **复用现有底座组件**(无需新搭):`orchestrator` / `schema_validator` / `policy_guard` / `audit` / `context_builder` / `ai_provider_configs` 表(key 经 `secret_box.decrypt`)均已落地;新增工作量仅为「注册 1 个 Capability + 写 1 个 Tool Wrapper + 沉淀 eval cases」。

### 4.7 Capability 元数据(`table_merge.suggest_mapping`)

**命名约定(避免后续误解)**:物理模块统一为下划线标识符 `table_tools`(Python 模块 / 权限 resource / capability 的 `module` 字段),它**同时承载合并与拆分两个子功能**;capability 按子功能分前缀——合并用 `table_merge.*`,远期拆分用 `table_split.*`。即模块名(一个)与 capability 前缀(按子功能多个)是**一对多**关系,不混为一谈。`table-tools` 连字符写法仅用于菜单/路由等 URL 化场景。

照 `backend/app/ai/capabilities.py::CapabilityDefinition` 真实字段填写,注册进代码白名单 `CAPABILITIES`:

```python
CapabilityDefinition(
    capability_id="table_merge.suggest_mapping",
    name="生成表格归集映射草稿",
    module="table_tools",
    type="draft",
    description="读取多源 Excel 的表头结构,跨文件聚类语义相同的列,提议标准字段、源列→标准字段映射、派生规则与归集主键,产出待人确认的归集模板草稿。",
    required_permission=("table_tools", "E"),  # 见 §6.2,对齐代码现状建模板权(过渡期固定动作 E)
    risk_level="medium",
    side_effect_tags=["draft_only"],
    confirmation="normal",                 # 草稿必经人确认固化,不自动建模板
    tools=[],                              # 表头解析在模块内确定性完成,不经 AI tool;AI 仅做语义聚类
    policy_profile={
        "output_contract": "merge_mapping_draft_schema",
        "deny_patterns": ["code", "url", "file_path", "external_link", "macro"],  # 去 sql:派生表达式/英文源列名易误伤
        "field_context": "headers_only",   # 只发表头,不发明细行,见 §4.8
        "allowed_side_effect": "draft_only",
    },
    model_profile="fast_json",
    sensitive_context="metadata_only",     # 表头属元数据;社保明细行严禁进上下文
    examples=["这批社保表帮我提一组标准字段和映射", "北京公积金这张表的列怎么对到标准字段"],
    failure_modes=["AI 未配置或超时", "模型未返回合法 JSON", "口径歧义列需人工定夺", "表头解析失败"],
)
```

输出契约 `merge_mapping_draft_schema`(过 `schema_validator`):标准字段清单 + 各源 `column_map` + `derived_fields` + `merge_keys` + **逐项 `confidence` 与 `needs_review` 标记**(低置信项前端标红,§2.4)。

### 4.8 敏感数据红线(Policy Guard 强制)

社保 / 公积金 / 身份证号属高敏数据。004《开发原则》§2.4「敏感明细默认不得进入模型上下文」在此为**强约束**:

- **只发表头**:进模型上下文的仅为「解析后的有效表头(多行拼接+合并填充结果)+ 列的非空率/类型等统计特征」,**严禁把数据行(姓名、证件号、金额明细)发给模型**。`field_context=headers_only` 由 policy_guard 在调用前裁剪并校验。
- AI 只判断「列名语义」,不接触「人的数据」;按人归集、数值搬运全部由确定性引擎完成。

#### 4.8.1 派生表达式的两层防线(为什么 deny 去掉 `sql`)

派生表达式(如 `{缴存基数}*{个人缴存比例}`)的安全**不靠 deny 文本扫描,靠执行层求值约束**。两层正交,各管一段:

| 防线 | 位置 | 管什么 | 对派生表达式 |
|---|---|---|---|
| **deny 正则** | router `/ai-draft` 输出侧 | AI **输出文本**(存库/返前端)别含注入串 | 文本兜底,易误伤,故**去 `sql`**,保留 `code/url/file_path/external_link/macro` |
| **`eval_expr`** | engine 执行时 | 表达式**真正求值**时的安全 | **真防线**,一字未放松 |

- **`eval_expr` 三道硬闸**(`engine.py`):① `{列名}` 占位符先被 `float()` 强制替换成数值——列名本身**永不进 `eval`**,非数值直接抛异常;② 替换后用 `re.fullmatch(r"[0-9eE+\-*/.() ]+")` 校验,只准剩数字与 `+ - * / ( )`,任何字母/下划线/冒号一律拒;③ `eval(safe, {"__builtins__": {}}, {})`,内置函数清空,无函数可调。故即便 AI 草稿写出恶意 `expr`,执行时也到不了 `eval`。
- **去 `sql` 的理由**:`sql` 正则含 `\bupdate\b`、`\binsert\b`、`from`、`where` 等词,会把**合法英文源列名**(如 `update_time`、`import_date`、强积金全英文表头)误判为注入而拒绝整张草稿。去掉后 4 个保留模式对正常中英文列名与派生表达式**零误伤**(已测),代码注入(`os.`/`exec(`/`__x__` 等)仍由 `code` 模式拦截。
- **结论**:去 `sql` 不改变派生字段任何处理逻辑——派生字段从不依赖 deny 保护,其安全由 `eval_expr` 执行层独立保证。

### 4.9 评测用例(004 §6 硬要求:无 eval 不得上线)

`table_merge.suggest_mapping` 上线前至少沉淀以下 eval cases,纳入底座 `evals`:

| 用例类型 | 内容 | 期望 |
|---|---|---|
| 正常生成 | 深圳日常申报表头 → 提映射草稿 | 标准字段/列映射结构合法,核心列命中 |
| 口径歧义 | 北京「医疗及生育保险/单位交」 | 标 `needs_review`,不擅自并入 `医疗公司` |
| 失败 | 全英文强积金表/陌生结构 | 低置信整表标红,不编造映射 |
| 越权 | 无 `manage_template` 触发建模板 | 能力解析即拒,不调模型 |
| 敏感数据 | 上下文混入数据行(姓名/证件/金额) | policy_guard 裁剪为表头,明细不出网 |
| 低置信回问 | 多列语义接近无法区分 | 标红留人手工指,不硬猜 |

## 8. 验证逻辑(每阶段"对"的标准)

- **解析正确性**:每个源解析出的行数 = 人工核对明细行数(剔除标题/合计行);表头字段全部识别、无错位。
- **映射正确性**:抽样 3~5 人,逐字段比对「总表值 == 源文件对应单元格值」。
- **按人合并正确性**:挑跨多文件的人(如胡俊杰),验证养老来自社保表、公积金来自公积金表、服务费来自服务费表,**全部落在同一行且数值正确**。
- **完整性对账**:总表人数 = 各源去重并集人数;关键金额列「总表合计 == 各源合计之和」。
- **异常可见**:未匹配模板的文件、对不上主键的记录、同人同险种多源冲突 → 全部进**异常清单**,不静默丢弃/覆盖。
- **自动识别准确率(阶段0新增)**:11 个文件按表头特征自动归类的命中率;漏判/误判需可解释、可回退。
