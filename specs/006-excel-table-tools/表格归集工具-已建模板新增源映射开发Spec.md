- 本文是该主 Spec 的增量实现与验收归档，覆盖**已有模板接入新数据源**，不改变模板的标准字段、归集主键或聚合口径。

### 0.2 已实现能力

| 范围 | 最终实现 |
|---|---|
| 源映射维护 | 支持单条新增、编辑、删除；单条操作不再依赖模板整体 `PUT` 清空重建。 |
| 批量新增主路径 | 从已有模板点击“新增映射表”，进入与首次建模板一致的批量拖拽上传向导。 |
| AI 草稿 | 批量样表按文件与 Sheet 生成源映射草稿；AI 只接收表头、Sheet、模板字段和业务背景，不接收数据明细。 |
| 模板口径 | 复用父模板的 `merge_keys`、`std_fields` 和 `aggregate`；AI 与前端均不得在新增流程中改写这些字段。 |
| 轻量确认 | 保存前仅确认映射名称、表头特征、低置信提示和是否保留草稿；复杂字段/公式在保存后从主界面编辑。 |
| 批量保存 | 后端先校验全部草稿，再统一写入；任一条失败时不产生部分保存。 |
| 运行可见性 | 未匹配源映射、映射命中歧义在合并异常中可见。 |

## 1. 目标与边界

### 1.1 目标

1. 管理员可为已有模板批量接入新的 Excel 源结构。
2. 用户体验与首次建模板一致：**批量拖拽 → AI 分析 → 人工确认 → 保存 → 返回模板列表**。
3. 新增流程不误改模板级口径；复杂映射维护回归已有映射编辑入口。
4. 批量保存具备原子性、权限校验、命名冲突校验和可追踪的 AI 审计。

### 1.2 非目标

- 不修改父模板的标准字段、归集主键、聚合口径。
- 不建设版本快照、回滚或跨模板复制。
- 不将姓名、证件号、金额、明细行或原 Excel 文件发送到 AI。
- 不涉及 UCP、第三方连接器、动态 SQL 或自定义代码执行。

## 2. 用户流程

### 2.1 批量新增源映射（主路径）

1. 模板创建者或超级管理员在模板卡片点击“新增映射表”。
2. 进入全页向导，批量拖拽一个或多个 `.xlsx` 样表，可填写业务背景。
3. 后端读取有效 Sheet 与表头，并基于既有模板字段调用受控 AI 能力生成多个映射草稿。
4. 前端展示每条草稿的映射名称、Sheet、表头行、识别特征、映射数量和低置信提示。
5. 用户可修改名称、表头识别特征或移除不需要的草稿；主键映射、字段映射、派生字段的深度编辑留待保存后处理。
6. 点击“保存 N 条映射”；服务端全量校验通过后统一写入并返回模板列表。

### 2.2 保存后维护

- 用户从模板主界面展开已有映射，可上传单份样表回填草稿、编辑 Sheet/表头区间/主键映射/字段映射/派生字段，或删除映射。
- 删除须二次确认；仅影响目标映射并递增模板版本。

## 3. 接口与数据契约

| 接口 | 用途 | 权限 | 关键约束 |
|---|---|---|---|
| `POST /table-tools/templates/{tid}/mapping-drafts` | 批量样表生成映射草稿 | `table_tools:U` + 创建者/超管 | `files[]` + `business_context`；只返回草稿，不写库。 |
| `POST /table-tools/templates/{tid}/mappings/batch` | 原子批量保存草稿 | `table_tools:U` + 创建者/超管 | 所有映射先校验；名称冲突、字段非法或特征不足时整体失败。 |
| `POST /table-tools/templates/{tid}/mappings` | 单条新增映射 | 同上 | 用于保存后手工追加。 |
| `PUT /table-tools/templates/{tid}/mappings/{mid}` | 单条编辑映射 | 同上 | 保持 `mid` 稳定。 |
| `DELETE /table-tools/templates/{tid}/mappings/{mid}` | 单条删除映射 | `table_tools:D` + 创建者/超管 | 返回 `204`。 |
| `POST /table-tools/templates/{tid}/mapping-draft` | 单样表回填草稿 | `table_tools:U` + 创建者/超管 | 用于已有映射的深度编辑辅助。 |

### 3.1 批量草稿响应

- 返回 `mappings[]`，每项包含名称、`match_signature`、Sheet、表头区间、主键/字段/派生字段建议、`_confidence` 与 `_notes`。
- 名称在前端按当前模板已有映射自动去重；重复时追加序号。
- 低于 `0.85` 的置信度须在确认页显著提示。

### 3.2 原子保存规则

1. 每条映射必须通过统一校验：非空名称、至少 3 个特征、合法表头区间、主键/字段/派生字段目标属于父模板。
2. 新提交内不能重名，也不能与数据库中已有映射重名。
3. 所有校验完成后才创建 ORM 对象、提交事务并递增父模板版本。
4. 任一条失败返回 `400` 或 `409`，不得产生部分写入。

## 4. 权限、安全与 AI

- 维护入口同时受前端可见性和后端创建者/超级管理员校验约束。
- AI 通过 `table_merge.suggest_mapping` capability、policy guard、输出 deny 扫描和 AI audit 运行；禁止绕过受控链路直接调用 provider。
- 审计输入仅保存文件名、Sheet、表头、业务背景等元数据。
- 样表仅在服务端临时解析；不传输明细行、姓名、证件号、金额或原文件内容给 AI。
- AI 建议始终是草稿，用户确认前不写库。

## 5. 前端交互规范

- 复用首次建模板的批量拖拽上传、文件 Chip、业务背景、AI 分析加载态和草稿确认节奏。
- 新增映射向导不展示模板名称、归集主键、标准字段、聚合规则的编辑控件。
- 草稿确认页仅保留高频且必要的轻量操作：改名称、改特征、移除草稿、查看低置信说明。
- 保存成功后回到模板列表并刷新映射数量；复杂编辑在主界面原有映射编辑区完成。

## 7. 部署与兼容性

- 本功能不新增数据库表或迁移；部署风险集中在后端、前端镜像更新与权限数据。
- 默认 `docker compose up -d --build` 不再把 HR Portal 后端发布到宿主机 `8000`；前端通过 Docker 内网 `backend:8000` 代理 API。
- 开发环境如需宿主机直连后端，使用：

```bash
docker compose -f docker-compose.yml -f docker-compose.dev.yml up -d --build
```

- 生产环境若历史上曾使用 `docker compose run --name hr-portal-backend ...`，切换到新默认配置前需一次性删除该遗留容器，再执行标准 Compose 启动。
