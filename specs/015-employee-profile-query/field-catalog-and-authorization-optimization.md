# 员工档案字段目录与授权解耦优化

> 文档类型：功能优化原子级开发任务文档  
> 关联 Capability：`employee.profile.query`  
> 所属原需求：`specs/015-employee-profile-query/spec.md`  
> 优化状态：待开发  
> 编写日期：2026-07-21
> 最近修订：2026-07-22（评审优化版）

## 1. 背景与目标

- 背景：
  - 原 `015` 员工基础工作信息查询采用代码枚举和固定物理列映射维护字段白名单。字段是否能被 AI 解析、查询和展示，与字段是否为敏感字段混在同一实现中。
  - 当前统一字段权限的正确规则是：字段未归入敏感分类时默认可见；字段归入敏感分类后，才由角色可见字段分类和用户额外可见字段分类决定是否可见。
  - 现有实现使公司名称、BU 等未写入枚举的普通业务字段无法被查询，后续每增加一个名册字段都需要修改后端枚举、映射、Extractor 提示、Schema、前端排序和测试，形成逐字段补丁。
  - 当前员工实时花名册存在多个业务字段，并且字段分类只负责敏感字段裁决。字段目录应成为技术字段与业务字段的统一边界，不能把未标敏感等同于无法查询。
- 目标：
  1. 将员工档案从代码固定字段枚举升级为基于员工实时花名册的服务端字段目录，新增业务字段无需改 AI 路由或字段枚举。
  2. 明确字段可见性规则：业务字段未标敏感时默认可见；标敏感后由现有字段分类、角色可见分类、用户额外可见分类及既有分类工具白名单语义统一裁决，不新增旁路。
  3. 在系统设置 → 字段管理中维护员工档案字段的显示名称和能力范围；`员工档案可查询` 仅决定字段能否进入该能力目录，不替代敏感字段授权。
  4. 未明确指定字段时，默认员工卡片候选固定为五项基础信息；默认项、明确指定字段与追问追加字段均须同时满足“员工档案可查询”、统一字段权限、scope、有效性和非空值条件，失败时省略且不补位。追问由模型语义决定追加既有字段或仅展示本轮指定字段。
  5. 保持 `employee.profile.query` 的功能权限、数据范围、候选受控 action、审计、限速、Web 与飞书统一结果协议不变。
  6. 本期仅使 `emp_realtime_roster` 接入员工档案查询；字段目录、展示配置和治理检查按表设计可复用，但不因此扩展为任意表或数据集的通用 AI 查询入口。
- 非目标 / 不做范围：
  - 不允许用户、前端或 LLM 传入 SQL、表名、任意物理列名、表达式、筛选条件或批量查询条件。
  - 不将内部技术列、同步列、主键、哈希列和审计列纳入员工档案字段目录。
  - 不改变字段分类、角色配置、用户管理的数据模型和既有授权语义；不以员工档案配置替代敏感字段分类。
  - 不自动将字段值、员工查询结果、权限结果、scope、selection handle 或审计数据发送给 LLM。
  - 不修改候选选择的短期 opaque handle、跨会话校验和单次消费语义。
  - 不因表出现在字段目录中自动授予 AI 查询能力；其他表或数据集必须独立注册 Capability 并完成权限、scope、身份字段和结果协议核验。

### 1.1 与原 `015` 文档的关系

| 项目 | 原 `015` 方案 | 本优化方案 | 关系 |
|---|---|---|---|
| 字段来源 | 代码固定字段白名单 | 服务端动态字段目录与配置覆盖 | 替换字段来源实现 |
| 字段权限 | 统一字段权限裁决 | 保持不变，作为唯一可见性授权依据 | 复用 |
| 默认卡片 | 固定默认字段集合 | 固定五项候选与顺序由代码维护，并按能力范围和权限交集过滤 | 收敛并保留 |
| 用户追加字段 | 固定枚举内字段 | 仅目录中 `员工档案可查询` 已开启的非技术业务字段，按权限裁决 | 扩展 |
| Extractor | 固定字段代码 | 读取受控低敏字段目录上下文 | 调整 |
| 候选、scope、审计、限速、飞书 | 原 `015` 设计 | 保持原设计 | 不变 |

本文件优先替代原 `spec.md` 中关于固定字段白名单、固定 `EmployeeProfileFieldCode` 枚举和静态物理列映射的描述。其他安全边界继续以原 `015` 文档为准。

### 1.2 假设与待确认事项

- 假设：员工实时花名册的业务字段均以稳定实体列形式存在，可由动态表反射获取。
- 假设：字段分类的敏感标记、角色/用户可见分类和字段分类工具白名单共同构成唯一字段授权来源；员工档案必须复用既有 `resolve_field_access()`，不得自行简化或重写其语义。
- 待确认：`id_number`、`bank_card_number`、`mobile_phone`、`corporate_email` 等个人敏感信息必须在上线前完成敏感分类治理；在未被归类前按本期确认规则默认可见，系统仅告警不静默改变授权。
- 待确认：字段中文显示名称优先读取既有字段管理元数据；若该元数据不存在，管理员需在员工档案展示配置中补充显示名称。

### 1.3 本期边界与后续扩展契约

- **本期查询边界：** `employee.profile.query` 仅绑定 `emp_realtime_roster`。员工工号、姓名匹配、候选选择、组织 scope 和卡片结果均为该实体表专属契约；不得将其 Handler 用于其他表或数据集。
- **可复用底座：** `FieldCatalogService` 可按 `table_name` 建立字段目录、展示配置、技术列排除、显示名称、缓存和治理检查，但目录只提供元数据，不是授权或查询注册表。
- **后续接入门槛：** 其他实体表或数据集必须在独立需求中注册 `capability_id`、受控数据源、稳定业务主键、标题字段、精确查询方式、scope 策略、字段授权 `tool_key`、支持类型、候选策略和统一结果 Schema；未注册时服务端必须拒绝查询。
- **数据集特别约束：** 数据集可能有聚合、派生字段、来源视图和不同的行级权限，不能复用员工“按姓名/工号查询单人”的 Handler，也不得仅依赖 `table_name` 开放。

## 2. 用户场景

### 2.1 系统管理员：配置员工档案能力范围

- 入口：系统设置 → 字段管理 → 选择数据表 `员工实时花名册`。
- 操作：查看系统识别出的业务字段，维护显示名称和 `员工档案可查询` 开关。
- 系统反馈：默认五项候选和顺序由代码固定；字段仅在开关开启且通过统一权限和 scope 裁决后出现在默认卡片或追问结果中。员工姓名和工号固定作为卡片标题，不占五项正文名额。
- 失败表现：字段不存在、技术字段、重复排序、默认字段不足或超过五个时禁止保存并指出字段。

### 2.2 HR/HRBP：查询默认员工档案

- 入口：Portal 全局 AI 助手或原 `015` 已允许的飞书私聊入口。
- 操作：输入员工姓名或工号，不明确请求字段。
- 系统反馈：返回卡片标题 `工号 -- 姓名`；正文按管理员配置的五项默认字段从上至下展示。
- 成功结果：未标敏感字段直接返回；已标敏感且无可见分类权限的字段不展示。
- 失败 / 空态 / 无权限表现：沿用原 `015` 的中性无匹配、候选、403、429 和 scope 失败表现。

### 2.3 HR/HRBP：连续追加员工字段

- 入口：同一员工档案会话的后续消息。
- 操作：例如增加公司名称、补充 BU、查看工作地点或增加直属上级。
- 系统反馈：系统根据受控字段目录解析字段代码，继承当前员工上下文，按当前字段权限裁决后更新结果卡片。
- 成功结果：默认五项保持原顺序；新增字段按用户本轮与历史追加顺序展示在默认五项之后。
- 失败 / 空态 / 无权限表现：字段不存在或属于技术字段时提示不支持该员工档案字段；字段已标敏感但无权时不回显字段值、不说明具体权限原因；无新增可见字段时不重复发送旧卡片。

### 2.4 系统管理员：敏感字段与角色授权

- 入口一：系统设置 → 字段分类管理，将实时花名册字段分配到分类，并决定分类是否敏感。
- 入口二：系统设置 → 角色配置 → 可见字段分类；用户管理中的额外可见字段分类用于叠加授权。
- 操作：将基本工资归入薪酬敏感分类，但不勾选 HRBP 的薪酬分类。
- 系统反馈：HRBP 查询员工档案时，基本工资不会出现在默认或追加字段结果；管理员或被授权角色可见。
- 边界：未加入敏感分类的字段默认可见，不要求在角色配置中额外勾选。

## 4. 技术设计

### 4.1 数据库 / 数据模型

新增 `employee_profile_field_settings`，仅保存员工档案展示配置，不保存字段授权：

| 字段 | 类型 | 规则 |
|---|---|---|
| `id` | bigint | 主键 |
| `table_name` | varchar(128) | 固定为 `emp_realtime_roster`，与 `column_name` 唯一 |
| `column_name` | varchar(128) | 必须是当前反射表中的非技术实体列 |
| `field_code` | varchar(128) | 稳定 API 代码；新记录默认等于 `column_name`，保留旧代码兼容映射 |
| `display_name` | varchar(64) | 管理员维护的中文展示名称，不能为空 |
| `is_queryable` | boolean | 默认 `false`；仅能力范围开关，开启后字段才可作为默认候选或追问追加候选 |
| `created_at` / `updated_at` | timestamptz | 审计时间 |
| `created_by` / `updated_by` | bigint | 管理员审计字段 |

- 索引与约束：
  - 唯一约束 `table_name + column_name`。
  - 唯一约束 `table_name + field_code`。
  - 默认卡片排序在 `is_default_card=true` 时不可重复。
  - 数据库无法保证恰好五项时，由事务内服务层校验；保存失败必须整体回滚。
- 技术字段排除清单：`id`、`pk_hash`、`synced_at` 以及服务端定义的保留技术前缀列；该清单只用于防止内部实现信息暴露，不构成业务字段授权。
- 兼容旧数据：migration 读取旧固定字段映射并预置已有默认五项及其显示名称；保留旧 `business_unit → bu` 等别名映射，用于恢复已存会话中的字段代码。
- Migration：新增 upgrade 与完整 downgrade。downgrade 删除设置表和索引，不改动名册、字段分类、角色授权或用户授权数据。

### 4.2 后端接口

#### 4.2.1 员工档案展示配置接口

| URL | Method | 权限 | Request / Response | 状态与错误 |
|---|---|---|---|---|
| `/api/v1/admin/employee-profile-fields` | GET | 复用字段管理查看权限 | 返回反射业务字段、配置、敏感分类摘要和默认卡片标记 | `200`；表未反射为 `409` |
| `/api/v1/admin/employee-profile-fields` | PUT | 复用字段管理维护权限 | `{ fields: [{ column_name, field_code, display_name, is_queryable }] }` | `200`；字段不存在、技术字段、代码重复为 `422`；无权 `403` |
| `/api/v1/admin/employee-profile-fields/governance-check` | GET | 复用字段管理查看权限 | 返回未配置显示名称、业务字段无敏感分类、敏感字段未分配角色等治理告警 | `200`；仅告警，不改变授权 |

接口禁止接收敏感字段可见性布尔值、角色 ID、用户 ID 或 SQL。敏感分类仍由既有字段分类接口维护，角色和用户可见分类仍由既有角色配置、用户管理接口维护。

#### 4.2.2 既有 AI Chat 接口兼容

- 继续使用 `POST /api/v1/ai/chat`，不新增业务顶层字段。
- `employee_profile_result.data.fields` 保持数组结构，字段项为 `{ code, label, value }`。
- `code` 从固定 Enum 调整为受限字符串：必须来自服务端字段目录或兼容别名，符合字段代码正则，禁止客户端和 LLM绕过目录自造代码。
- `label` 只能由服务端目录返回，前端不得推导或覆盖。
- `value` 只能来自 scope 下推 SQL 与字段权限裁决后的结果，`hide` 字段不得出现在数组中。
- 状态码、候选 action、403、410、429 和审计 Envelope 沿用原 `015` 约束。

### 4.3 业务逻辑

1. **构建字段目录**
   - 后端从已反射的 `emp_realtime_roster` 获取实体列，剔除技术字段。
   - 将 `employee_profile_field_settings` 与实体列左连接，得到 `FieldCatalogItem { field_code, column_name, display_name, default_card, default_order, append_order }`。
   - 对没有配置记录的业务字段生成只读目录项：`field_code=column_name`，显示名称按字段管理元数据或列名回退规则生成；管理员可在字段管理页面补充更友好的名称。
   - 目录不包含字段权限结论和员工字段值。

2. **解析字段请求**
   - Intent 分类仍只接收低敏路由目录和用户消息。
   - Target Capability Gate 成功后，员工档案 Extractor 可接收 `FieldCatalogContext`，内容仅含 `field_code`、`display_name` 和可解析别名，不含敏感标记、角色、用户权限、scope、员工资料或查询结果。
   - Extractor 仅输出目录内 `field_code`；服务端对字段代码去重、限制单轮最多 20 项、拒绝技术字段和不存在字段。
   - 模型解析失败时不按关键词猜测字段；返回中性提示。别名兼容仅用于历史稳定代码和管理员维护的显式别名。

3. **默认字段与追加字段**
   - 默认候选固定为 `department`、`hire_date`、`employee_type`、`standard_position`、`position_level`，顺序由代码维护；候选字段只有在 `is_queryable=true` 且通过统一字段权限、scope、有效性和非空值过滤后才展示。
   - 明确请求字段时：仅接受 `is_queryable=true` 的目录字段；查询通过过滤的默认候选与当前会话已追加字段的并集，再追加本轮新增字段。
   - 会话只保存查询类型、查询值和字段代码列表，不保存员工字段值。
   - 已存在字段不重复展示；无新增有效字段时不重放旧卡片。

4. **scope 与字段权限裁决**
   - SQL 必须继续通过 `build_scope_filter()` 在查询前下推对象范围。
   - 对目录字段的物理列调用 `resolve_field_access(user, emp_realtime_roster, db, tool_key="employee.profile.query")`。
   - 敏感分类列按现有 resolver 的角色、用户额外分类和字段分类工具白名单语义裁决；不得以目录或展示配置覆盖其结果。
   - 不使用 AI 字段目录或展示配置授予、拒绝或覆盖字段权限。

5. **结果与排序**
   - 标题固定使用 `employee_no -- 姓名`；姓名优先 `full_name`，为空时仅按 `chinese_name`、`english_name` 回退，标题无法安全生成时按原中性降级规则处理。
   - 卡片正文先按默认五项 `default_display_order` 展示，再按会话字段追加顺序展示新增字段；`append_display_order` 仅用于非会话场景的稳定兜底。
   - 前端只渲染后端 `fields` 数组，不能按对象属性、物理列或本地字段表推断结果。

6. **复用与禁止重复建设**
   - 复用 `app.permissions.masker.resolve_field_access`、`build_scope_filter`、`AiConversation`、统一 AI Envelope、受控 action 和审计投影。
   - 不新增员工字段专属角色表、用户授权表、脱敏器、scope 过滤器、聊天接口或候选 action。

### 4.3.1 强制安全、兼容与数据契约（本节优先于本文件其他泛化表述）

1. **目录与查询分离**
   - 字段目录存在不等于字段可被查询，更不等于其他表/数据集可被查询。员工 Capability 只允许使用已注册的 `emp_realtime_roster`。
   - 目录、请求校验和 SQL 投影三层均须拒绝技术列、主键/哈希/同步/审计列、未知列、失效列和不支持类型；查询仅投影标题字段与已授权请求字段，禁止 `SELECT *`、字符串拼列和任意物理列输入。
   - 首期仅允许短文本、数值、布尔、日期/时间和受控枚举；JSON、数组、富文本、二进制、大对象和超长文本默认不进入目录。服务端统一格式化、限制单字段与整卡片长度，并省略空值。

2. **字段代码生命周期**
   - `field_code` 是不可变 API 标识；旧代码到当前物理列的完整映射必须在 migration 中显式维护，至少包括 `business_unit → bu`、`organization_name → department`、`position_name → position` 及所有现网已发布代码。
   - 物理列重命名不得改变已发布 `field_code`；列删除时字段应标记失效，旧会话与受控 action 仅安全省略并返回中性提示，旧代码不得重新绑定至不同业务含义的列。
   - 配置更新采用版本号或 `ETag` 乐观锁；保存、反射刷新、列删除和配置更新必须使跨实例目录缓存失效。

3. **标题、默认五项与会话**
   - 标题固定使用 `employee_no -- 姓名`；姓名优先 `full_name`，为空时仅按 `chinese_name`、`english_name` 回退，标题无法安全生成时按原中性降级规则处理。
   - 默认配置必须恰好五项；若其中字段因权限、空值、失效或删除被过滤，实际卡片可以少于五项，**禁止用其他字段补位**。
   - 会话和 handle 只保存规范字段代码、顺序及目录/配置版本，不保存字段值、完整目录、scope、权限结论或原始消息。

4. **统一字段权限与 LLM 边界**
   - 字段权限唯一调用 `resolve_field_access(user, "emp_realtime_roster", db, tool_key="employee.profile.query")`；必须保留角色、用户额外分类授权和字段分类工具白名单的既有语义，目录与展示配置不得覆盖裁决。
   - Extractor 仅接收经安全筛选的非敏感 `FieldCatalogContext`。敏感字段名称、别名、分类、用户权限和字段值不得进入 LLM；敏感字段仅可由服务端确定性兼容别名解析后再走当前权限裁决。
   - 管理员维护的显示名称须校验长度、字符集、控制字符和提示词分隔符；不得自动从显示名称生成别名，避免提示词注入和解析歧义。

5. **管理端与渠道约束**
   - 配置接口不得接收角色 ID、用户 ID、敏感可见性开关、SQL、表名或物理列名；敏感分类摘要必须按既有元数据查看权限裁剪。
   - 前端与飞书只渲染后端 `fields` 数组和服务端标题，不得自行依据物理列、本地字段表或分类名称推导结果。

### 4.5 权限、安全与外部系统

- 功能权限：继续要求 `employee.profile:V` 与原受控验证 Gate。
- 对象范围：继续通过 `build_scope_filter` 在 SQL 查询前强制注入。
- 字段权限：字段分类是唯一授权来源。未敏感字段默认可见；敏感字段必须满足分类可见授权；用户额外可见分类与角色可见分类取并集。
- 技术字段安全：技术字段被目录构建器硬排除，不因未敏感而可查询。
- LLM 安全：仅发送无员工数据的 `FieldCatalogContext`；不发送权限结果、敏感标记、scope、查询结果、候选或 handle。
- 审计：仍仅记录字段代码、数量、状态、scope 标记和 failure stage；不记录字段值、原始消息、目录完整快照或权限结果。
- 外部系统：不新增外部系统。飞书复用原 `015` 与 `004` 通道；Web 和飞书使用同一字段目录和服务端裁决结果。

| 未分类个人字段默认可见 | 高 | 可能按既定默认可见规则暴露不应公开的信息 | 上线前运行治理预检，补全敏感分类；发布审批必须附分类矩阵证据 |
| 动态字段目录被当作任意列查询 | 高 | 可能读取技术列或扩大数据面 | 仅允许已反射的名册实体业务列，硬排除技术列，服务端二次校验 |
| LLM 伪造字段代码 | 高 | 尝试绕过字段目录 | Extractor 输出必须与服务端目录匹配；未知代码拒绝；前端不可追加字段代码 |
| 显示名称缺失或错误 | 中 | 卡片可读性下降、解析歧义 | 配置页强校验默认字段名称，治理预检提示未配置字段 |
| 原固定 Enum 协议变化 | 中 | 前后端、飞书或旧会话不兼容 | 保持数组字段对象，保留旧代码别名，前后端同批发布并覆盖旧会话测试 |
| 默认卡片配置错误 | 中 | 卡片展示超出或少于五项 | 事务校验恰好五项、排序唯一、保存前端禁用与后端 `422` 双校验 |
| 字段目录缓存过期 | 中 | 新字段或配置不能及时生效 | 配置保存后主动失效；名册反射刷新后重建；提供管理端刷新机制 |
| 审计记录目录或字段值 | 高 | 人员信息和治理信息泄露 | 审计只记录字段代码和计数；禁止保存字段值、完整目录、原始消息和权限结论 |
| 分类工具白名单语义被忽略 | 高 | 员工档案与现有统一字段权限不一致，导致越权或错误拒绝 | 固定使用 `tool_key="employee.profile.query"`，覆盖白名单、角色和用户授权组合测试 |
| 目录存在即开放其他表/数据集 | 高 | 任意表查询、scope 错配或数据集聚合语义被绕过 | 目录与 Capability 注册分离；未注册数据源 fail-closed；后续接入必须独立立项 |
| 敏感字段元数据进入 LLM | 高 | 字段名称、分类或权限语义泄露，并可能受提示词注入影响 | LLM 仅接收安全非敏感目录；敏感字段走服务端确定性解析；显示名称严格校验 |
| 旧代码或物理列变化错映射 | 高 | 旧会话查询错误字段或泄露不相干业务数据 | 不可变 `field_code`、完整映射、失效标记和中性降级；禁止旧代码重绑定 |
| 默认卡片与权限过滤冲突 | 中 | 实际少于五项造成产品预期偏差 | 明确“配置恒五项、实际可少于五项、不补位”，并覆盖 UI/E2E |
| 不支持类型或超长值展示 | 高 | 数据泄露、性能下降或渠道渲染异常 | 类型白名单、服务端格式化和长度限制；未知类型 fail-closed |
| 并发修改或缓存过期 | 中 | 管理员覆盖彼此修改或新配置不生效 | 乐观锁/ETag、跨实例失效、反射刷新和冲突提示 |
