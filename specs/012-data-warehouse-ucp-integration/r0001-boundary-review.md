# R0001 三期启动前边界评估记录

> 评估日期：2026-07-06
> 评估依据：spec.md 第 14 章（分层职责）、ucp-coordination.md（UCP 协同边界）、附件《数据仓库.txt》8.4
> 评估结论：R01-R07 全部能力均属于"已入仓数据加工"，不涉及跨系统连接/凭证/Pipeline/BI 可视化，应由数据仓库承建。

---

## 核心边界判断准则

| 层级 | 负责 | 数据仓库能做吗？ |
|------|------|------------------|
| **UCP / 专业 ETL** | 外部连接、凭证、跨系统 Pipeline、重试/死信、限流 | ❌ 不碰 |
| **数据仓库** | 已入仓数据沿 ODS→DWD→DWS→ADS 的加工、计算、治理、发布 | ✅ 本期焦点 |
| **帆软 / BI** | 报表设计、图表、大屏、自助分析 | ❌ 不内建 |

---

## 逐项评估

### R01：ODS → DWD 标准化与清洗

**能力清单**：字段重命名、类型转换、枚举映射、单位转换、拆分合并、去重、空值处理、格式标准化、模板管理、预览、DWD 视图生成。

| 问题 | 结论 |
|------|------|
| 为什么在数据仓库做？ | 输入是已入仓 ODS 表，输出是仓内 DWD 视图/DataSet。全程不碰外部系统，属于仓内数据加工核心能力。 |
| 为什么不在 UCP 做？ | UCP 的职责是"把外部数据搬进来"，不是"把数据洗干净"。字段级标准化规则（如员工状态码映射、日期格式统一）属于数据口径治理，UCP 的 Adapter/Pipeline 模型不适合承载这类细粒度规则。且 UCP 不可用时标准化能力必须仍可用（Q0003 降级基线）。 |
| 为什么不在帆软做？ | 帆软是消费端。如果在帆软做清洗，每个报表各自维护一套规则，口径不一致、无法复用、无法血缘追踪。标准化必须在数据进入消费层之前完成。 |
| ODS 原始数据不能被覆盖 | 规则执行写入派生/物化结果，ODS 原始数据保留不变。 |
| 不开放 SQL/脚本 | 规则只通过受控表单配置 8 类 `rule_type`，无任意代码输入。 |

**结论：数据仓库负责。**

---

### R02：分层产物物化与刷新支撑

**能力清单**：数据集物化（virtual→materialized）、构建运行记录、分层校验（ODS→DWD→DWS→ADS）、增量/全量刷新策略。

| 问题 | 结论 |
|------|------|
| 为什么在数据仓库做？ | 物化对象是仓内 `DataSet`（`build_mode=virtual/materialized`），刷新依据是仓内时间字段或 `SyncRun` 摘要。全程仓内闭环。 |
| 为什么不在 UCP 做？ | UCP 的 Pipeline 物化面向的是"从外部系统拉取后落地"，数据仓库的物化面向的是"仓内模型从视图转为物理表"。两者物化对象、触发条件、分层校验规则完全不同。R00 明确禁止物化接口触发 DataSource/UCP 实时外部拉取。 |
| 为什么不在帆软做？ | 帆软的数据集物化是 BI 内部机制，不能替代数据仓库的分层产物治理。数据仓库物化结果需要血缘、影响分析、权限传播，帆软不具备这些能力。 |
| 分层校验 | 执行前必须校验输入层→输出层顺序（ODS→DWD→DWS→ADS），非法跳转返回 400。 |

**结论：数据仓库负责。**

---

### R03：DWD → DWS 指标聚合与维度增强

**能力清单**：指标计算/重算、维度定义管理（层级树+字段绑定）、DWS 聚合定义（group_by/聚合函数/时间粒度）、DWS 视图生成。

| 问题 | 结论 |
|------|------|
| 为什么在数据仓库做？ | 输入是仓内 DWD/DataSet/维度表，输出是 DWS 聚合视图。指标口径、维度层级、聚合逻辑都属于数据仓库语义层，是仓内核心加工链路的中间环节。 |
| 为什么不在 UCP 做？ | UCP 不做数据聚合和指标计算。跨系统取数后如果需要聚合，也应在入仓后由数据仓库完成——否则口径分散在 UCP Pipeline 中无法统一治理。 |
| 为什么不在帆软做？ | 帆软可以消费 DWS 结果做报表，但不能作为统一指标口径的 owner。如果指标聚合分散在各个 BI 报表里，不同报表对"在职员工数"的定义可能不一致。数据仓库负责统一口径，BI 只负责展示。 |
| 不替代 BI 语义层 | DWS 提供公共汇总服务，不做一次性报表口径。 |

**结论：数据仓库负责。**

---

### R04：快照与拉链

**能力清单**：员工/组织等对象快照生成（周期+保留策略）、SCD 拉链（business_key/effective_from/effective_to/current_flag）。

| 问题 | 结论 |
|------|------|
| 为什么在数据仓库做？ | 快照源为仓内表/模型，不需要访问外部系统。快照和拉链是数据仓库的经典能力（缓慢变化维度），与分层流转紧密相关。 |
| 为什么不在 UCP 做？ | UCP 可以做"从外部系统定期全量/增量拉取"，但拉取后是否保留历史快照、如何管理 SCD 拉链，是数据仓库的分内之事。UCP 只管搬运，不管历史版本管理。 |
| 为什么不在帆软做？ | 帆软没有 SCD 拉链管理能力。BI 工具可以做时间序列分析，但不负责生成和管理历史快照数据。 |
| 保留策略 | 快照不可无限堆积，必须配置保留策略（如保留最近 12 个月）。 |

**结论：数据仓库负责。**

---

### R05：仓内调度、重跑与审计

**能力清单**：为 `dataset_build`/`metric_run`/`quality_run`/`snapshot_run` 四类仓内任务注册 handler、定时配置 UI、失败重跑 + 审计日志。

| 问题 | 结论 |
|------|------|
| 为什么在数据仓库做？ | 调度的对象全部是仓内任务（构建、指标计算、质量检查、快照生成）。项目已有 `app/scheduler/` 通用调度组件（`scheduled_jobs`/`job_runs` + `JOB_HANDLERS` 注册机制），直接复用即可。 |
| 为什么不在 UCP 做？ | R00 明确禁止仓内调度直接复用/耦合 UCP Pipeline 表或触发 UCP Pipeline 执行。UCP 调度面向跨系统 Pipeline（拉取→转换→写入），数据仓库调度面向仓内加工任务（构建→聚合→快照），两者调度对象和触发条件不同。如果混在一起，UCP 不可用时仓内任务也全部瘫痪。 |
| 为什么不在帆软做？ | 帆软调度面向报表刷新和数据抽取，不管理仓内模型构建、指标计算等内部加工任务。 |
| 必须复用现有组件 | `app/scheduler/` 已实现通用调度，禁止另建"仓内调度中心"。 |

**结论：数据仓库负责（复用 scheduler）。**

---

### R06：数据开放增强

**能力清单**：数据开放 API 契约设计（将已发布数据集/指标开放给 BI/API/订阅）、数据开放状态 UI。

| 问题 | 结论 |
|------|------|
| 为什么在数据仓库做？ | "开放"的是仓内已发布的 ADS/指标资产。数据仓库需要定义哪些资产可开放、开放粒度、权限要求——这是资产治理的最后一环，不是新建外部 API 网关。 |
| 为什么不在 UCP 做？ | UCP 负责跨系统推送/API 的执行层面（凭证、限流、重试）。数据仓库负责"声明哪些资产可开放 + 权限 + 脱敏提示"。两者是分工关系：仓库声明、UCP 执行（未来协作）。但 R06 的开放 API 契约设计是仓库侧的定义工作。 |
| 为什么不在帆软做？ | 帆软是消费方之一（BI 分析、报表），不是开放能力的管理者。数据仓库作为资产 owner，定义哪些字段可以给 BI、哪些不能。 |
| 安全红线 | 不默认开放所有资产；不绕过审批直接生成外部 API；不开放敏感字段明文导出。 |

**结论：数据仓库负责（契约定义 + 状态管理），UCP 负责执行层。**

---

### R07：DWS → ADS 组装与消费资产发布

**能力清单**：ADS 组装定义（DWS+维度→ADS 宽表）、组装向导 UI、发布 API（发布为数据资产/视图/API候选/PushTarget候选）、发布 UI。

| 问题 | 结论 |
|------|------|
| 为什么在数据仓库做？ | 输入是仓内 DWS/维度，输出是仓内 ADS 消费资产。这是数据仓库分层链路的最后一步（DWS→ADS），本质是"把公共汇总服务组装成面向具体场景的消费产品"，属于仓内加工能力。 |
| 为什么不在 UCP 做？ | UCP 不负责数据组装和语义映射。ADS 组装需要维度关联、字段选择、预置过滤、脱敏、权限继承——这些都是仓内语义操作，不是跨系统搬运。 |
| 为什么不在帆软做？ | 帆软可以消费 ADS 资产作为报表数据源，但不应在 BI 层做 DWS→ADS 的组装。如果每个 BI 报表各自组装，消费口径又不一致。统一在数据仓库组装好，BI 直接用。 |
| 不新增 DM 分层 | `DM` 用主题域/消费域标签表达，不写入 `warehouse_layer`；不跳过 DWS 直接拼装不可治理的报表宽表。 |

**结论：数据仓库负责。**

## 总结：能力下沉判定矩阵

| 能力 | 输入源 | 输出目标 | 是否外部连接 | UCP 适合？ | BI 适合？ | 归属 |
|------|--------|----------|-------------|-----------|----------|------|
| ODS→DWD 标准化 | ODS 仓内表 | DWD 视图 | ❌ | ❌ UCP 不加工已入仓数据 | ❌ BI 是消费端 | **仓库** |
| DWD 物化/刷新 | DWD 视图 | 物化表 | ❌ | ❌ UCP 不管仓内物化 | ❌ BI 不管存储层 | **仓库** |
| DWS 聚合/维度 | DWD+维度表 | DWS 视图 | ❌ | ❌ UCP 不做聚合 | ❌ BI 管展示不管口径 | **仓库** |
| 快照/拉链 | 仓内表/模型 | 快照/拉链表 | ❌ | ❌ UCP 不管理历史版本 | ❌ BI 无 SCD 能力 | **仓库** |
| 仓内调度/重跑 | 仓内任务 | 调度/审计记录 | ❌ | ❌ 禁止耦合 UCP Pipeline | ❌ BI 不管仓内调度 | **仓库** |
| 数据开放契约 | ADS/指标资产 | 开放状态/告示 | ❌ | 🔗 未来协作（UCP 执行） | ❌ BI 是消费方之一 | **仓库** |
| ADS 组装/发布 | DWS+维度 | ADS 消费资产 | ❌ | ❌ UCP 不做数据组装 | ❌ BI 不做上游组装 | **仓库** |

## 风险评估

- **低风险**：R01/R02/R03/R04/R07（纯仓内加工，边界清晰）。
- **中风险**：R05（需确保 handler 注册不冲突）；R06（未来需与 UCP 执行层对接，本期只做契约定义）。
- **不纳入三期**：跨系统 Pipeline（UCP）、BI 自助分析（帆软）、任意 SQL/脚本（安全原因）、复杂维度建模设计器（四期复评）。

## 批准

- [x] 所有 R01-R07 能力均有"为什么不在 UCP/帆软做"的明确理由
- [x] 没有复杂 ETL/Pipeline 被默认搬进数据仓库
- [x] 评审通过，可进入 R0002
