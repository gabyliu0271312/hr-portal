## 第二轮讨论：DataSet 视图 vs 物理表

### 关键发现

用户提出三个深层问题后，追踪代码链路确认：

**1. DataSet 是什么？**

DataSet 是**逻辑视图**（`build_mode = "virtual"`，DB 默认值），不是物理表。它在查询时动态应用输出字段定义（包括清洗规则），查的是 ODS 物理表。

**2. 两个独立产物**

| | execute_full（执行） | generate_dwd_view（发布为DWD视图） |
|---|---|---|
| 产物 | 物理 DWD 表 | DataSet 记录（逻辑视图） |
| 数据从哪来 | ODS 全量读取 → 写入 DWD | 实时查 ODS 表 + 动态转换 |
| 性能 | 查询快（预计算） | 查询慢（实时计算） |
| 新鲜度 | 两次执行之间过时 | 始终最新 |

**3. DataSet 创建时机**

DataSet 在**资产创建时**（`POST /admin/tables`）就由 `ensure_single_table_dataset` 自动创建，不是清洗后才建。命名规则：

```python
SINGLE_TABLE_DATASET_PREFIX = "单表数据集 · "
# 实际产物： "单表数据集 · ods_employee"
```

**4. DataSet 指向哪张物理表？**

当前 `generate_dwd_view` 创建的 DataSet 指向 **ODS 源表**（`DataSetTable.table_name = asset_code`，即 ODS 表名），同时设置 `warehouse_layer = "DWD"`。

### 数仓专家评估（已确认）

**DataSet 应指向 ODS 表。** 原因：

- ODS 通过 DataSource 同步保持近实时更新
- 清洗规则是字段级转换（重命名、类型转换、枚举映射），查询时动态计算性能损失极小
- DataSet 提供新鲜度，物理 DWD 表提供查询性能，两者互补
- 如果 DataSet 指向 DWD 物理表，两次执行之间下游看到的是过时数据

## 第四轮讨论：warehouse_layer 归属（未解决）

### 两难

**场景 A（清洗）**：DataSet 指向 ODS 表，但清洗规则应用后输出质量是 DWD。`warehouse_layer = "DWD"` 语义上是"输出层"。

## 最终结论：表、数据集、视图的概念收敛

### 1. 产品层只保留“表”和“数据集”两个核心概念

系统不应在产品层同时暴露“表、视图、数据集”三个平级概念，否则会增加业务用户理解成本，也会导致资产层级和报表引用逻辑混乱。

最终口径如下：

| 概念 | 定义 | 用户是否感知 |
|---|---|---|
| 表 | 底层物理存储对象，承载实际数据 | 是 |
| 数据集 | 面向清洗、建模、报表等功能的数据访问入口 | 是 |
| 视图 | 数据集的一种内部技术实现方式 | 否 |

因此，系统对用户只呈现：

```text
表：数据实际存储在哪里
数据集：业务功能应该通过什么入口使用数据
```

“视图”不再作为独立产品概念出现，只作为数据集背后的技术实现方式，例如虚拟查询、SQL View 或物化结果。

---

## 最终结论：ODS 拉取时创建的数据集归属 ODS 层

### 2. 资产层级取决于当前承载的数据质量，而不是未来用途

系统中所有资产都需要归属到数仓层级时，应遵循以下原则：

> 资产归属层级，取决于它当前承载的数据质量层级，而不是取决于它未来可能被用于生成哪个层级。

因此，拉取 ODS 表时自动创建的数据集，应归属于 ODS 层，而不是 DWD 层。

示例：

```text
ods_employee 表        → ODS 层
ds_ods_employee 数据集 → ODS 层
```

原因：

1. 该数据集底层仍然指向 ODS 原始表；
2. 数据尚未经过标准化清洗；
3. 字段口径、枚举、格式、缺失值处理都还不稳定；
4. 它只是 ODS 表的逻辑访问入口，不代表 DWD 清洗产物。

所以，ODS 拉取后的自动数据集应定义为：

> ODS 源表数据集，用于字段管理、原始数据预览、清洗规则配置和清洗输入。

它不是正式报表消费入口。

---

## 最终结论：DWD 层应先形成物理表，再形成 DWD 数据集

### 3. 清洗后的 DWD 产物应以物理表为基础

清洗执行后，系统应先生成 DWD 物理表，再基于 DWD 物理表创建 DWD 数据集。

推荐链路：

```text
ODS 表
  ↓ 清洗规则
DWD 物理表
  ↓ 数据集包装
DWD 数据集
```

示例：

```text
ods_employee
  ↓
dwd_employee
  ↓
ds_dwd_employee
```

DWD 物理表承载清洗后的稳定明细数据，DWD 数据集作为报表、建模和下游分析的统一消费入口。

不建议将仍然指向 ODS 表的数据集，仅通过修改 layer 字段包装成 DWD 数据集。这样会制造“假 DWD 数据集”，导致资产目录、报表口径和后续治理继续混乱。

---

## 最终结论：报表模块只能引用 DWD / DWS 数据集

### 4. 报表不直接引用表，也不引用 ODS 数据集

报表模块应统一引用数据集，而不是直接引用物理表。

但报表可选范围应限制为：

```text
DWD 数据集
DWS 数据集
```

不应默认允许选择：

```text
ODS 数据集
```

原因：

1. ODS 是原始数据层，字段和口径未经清洗；
2. 报表如果直接引用 ODS 数据集，会绕过数仓分层；
3. 报表需要的是稳定业务口径，而不是原始落地数据；
4. 字段展示、权限、指标口径、格式化规则应由 DWD/DWS 数据集承载。

最终报表链路应为：

```text
报表 → DWD/DWS 数据集 → DWD/DWS 物理表
```

而不是：

```text
报表 → ODS 数据集 → ODS 表
```

## 新增规则：新生成资产的处理方式

### 5. 新拉取 ODS 表时的规则

从本结论生效后，新拉取 ODS 表时：

```text
创建 ODS 物理表（ods_ 前缀）
自动创建 DWD 物理表（dwd_ 前缀，同结构，字段先保持一致）
自动创建 DWD 数据集（ds_dwd_ 前缀，指向 DWD 物理表）
DWD 数据集可作为报表和建模数据源
```

设计理由：用户拉取数据后立即能用 DWD 数据集建报表，清洗变成可选增强而非阻塞门槛。数据量小，同结构拷贝的存储成本可忽略。

示例：

```text
ods_employee              → ODS 物理表，清洗输入
dwd_employee              → DWD 物理表，自动生成（同结构）
ds_dwd_employee           → DWD 数据集，可直接用于报表/建模
```

### 6. 清洗执行后的规则

执行清洗后：

```text
更新 DWD 物理表（应用清洗规则，替换自动生成的同结构表）
更新 DWD 数据集字段（同步清洗后的字段定义）
```

DWD 数据集自始至终是同一份，清洗前后只是数据质量提升，不改变数据集本身的存在和引用关系。

示例：

```text
清洗前：dwd_employee = ods_employee 的同结构拷贝
清洗后：dwd_employee = 应用了字段重命名、类型转换、枚举映射后的标准化表
         ds_dwd_employee 自动同步清洗后的字段定义
```

### 6a. 资产命名规范

新建资产必须遵循以下命名规则，系统在注册时校验：

| 层级 | 物理表前缀 | 数据集前缀 | 示例 |
|------|-----------|-----------|------|
| ODS | `ods_` | —（不创建用户可见 ODS 数据集） | `ods_employee` |
| DWD | `dwd_` | `ds_dwd_` | `dwd_employee` → `ds_dwd_employee` |
| DWS | `dws_` | `ds_dws_` | `dws_headcount_summary` → `ds_dws_headcount_summary` |

规则：
- 新建 ODS 表时，表名必须以 `ods_` 开头，否则拒绝注册
- DWD/DWS 表名由系统自动推导（`ods_xxx` → `dwd_xxx`），用户不可覆盖
- 数据集名由系统自动推导（`dwd_xxx` → `ds_dwd_xxx`），用户不可覆盖
- 历史无前缀表（如 `emp_monthly_salary`）视为 ODS，不改名，仅新资产强制前缀

## 历史报表引用迁移方案

### 7. 迁移背景

当前生产环境已经存在部分报表引用 ODS 层数据集的情况。这些历史引用不能简单删除，也不建议通过原地修改 layer 的方式把 ODS 数据集伪装成 DWD 数据集。

正确处理方式是：

> 对历史报表引用做迁移，而不是原地改层级。

---

### 8. 迁移原则

历史迁移遵循以下原则：

1. 保证已有报表继续可用；
2. 不制造“底层仍是 ODS、层级却标 DWD”的假 DWD 数据集；
3. 报表最终切换到 DWD/DWS 数据集；
4. 原 ODS 数据集保留，但不再允许新报表选择；
5. 新旧规则切换后，新增报表必须走 DWD/DWS 数据集。

---

### 9. 迁移步骤

#### 第一步：识别历史依赖

扫描当前报表配置，找出所有引用 ODS 数据集的报表。

迁移前示例：

```text
报表 A → ds_ods_employee → ods_employee
```

#### 第二步：补建 DWD 物理表

对被历史报表引用的 ODS 数据集，补建对应的 DWD 物理表。

如果历史数据集没有明确清洗规则，先采用同结构迁移：

```text
ods_employee → dwd_employee
```

即字段结构先保持一致，确保报表迁移不中断；后续再逐步补充标准化清洗规则。

#### 第三步：创建 DWD 数据集

基于补建的 DWD 物理表创建 DWD 数据集：

```text
dwd_employee → ds_dwd_employee
```

该数据集归属 DWD 层，并允许被报表模块选择。

#### 第四步：迁移报表引用

将历史报表的数据源从 ODS 数据集切换为 DWD 数据集。

迁移前：

```text
报表 A → ds_ods_employee
```

迁移后：

```text
报表 A → ds_dwd_employee
```

#### 第五步：保留原 ODS 数据集但关闭报表入口

原 ODS 数据集继续保留，用于清洗输入、字段管理和原始数据预览。

```text
ds_ods_employee
layer = ODS
reportable = false
```

它不再出现在新建报表的数据源选择列表中。

## 最终目标状态

### 10. 统一后的资产关系

```text
拉取 ODS（自动完成）：
ods_employee
  ↓ 自动同结构拷贝
dwd_employee
  ↓ 自动创建数据集
ds_dwd_employee
layer = DWD
用途 = 报表 / 建模 / 下游分析
立即可用

清洗（可选增强）：
ods_employee → 清洗规则 → 更新 dwd_employee → 同步 ds_dwd_employee 字段

报表消费（始终不变）：
报表 → ds_dwd_employee → dwd_employee
```

### 11. 一句话结论

> 系统产品层只保留”表”和”数据集”两个概念。拉取 ODS 表时自动创建 DWD 物理表（同结构）和 DWD 数据集（ds_dwd_ 前缀），立即可用于报表和建模。清洗是可选增强——执行后更新 DWD 表内容和数据集字段定义，数据集 ID 不变，已有报表引用不中断。新建资产强制命名规范：ods_ / dwd_ / ds_dwd_。历史上引用无前缀表的数据集已通过补建 DWD 表 + 迁移引用完成平滑过渡。

## 补充结论：数据建模阶段的数据源范围

### 12. 当前代码现状

结合当前代码确认，数据建模阶段的数据源选择目前没有严格按数仓层级过滤：

- 前端建模页面 `WarehouseModelingQuick.vue` / `WarehouseModelingVisual.vue` 调用 `listAssets({ page_size: 200 })` 获取资产列表，未传 `warehouse_layer` 过滤条件；
- 后端 `list_assets` 接口本身支持按 `warehouse_layer` 查询，但建模页面当前未使用；
- 后端 `create_model` / `publish_model` 当前没有限制建模输入必须来自 DWD/DWS，也没有明确限制是否只能选择单表数据集；
- Visual 建模支持多表关系，Quick 建模也存在主表 + 关联表的组合关系。

这意味着当前系统实际允许建模阶段选择到 ODS 层资产，存在绕过清洗层的风险。

---

### 13. 建模输入应选择 DWD 单表数据集

结合数仓分层原则，数据建模阶段不应直接选择 ODS 数据集。

最终规则建议：

> 数据建模阶段的输入，应默认选择 DWD 层的单表数据集，不允许选择 ODS 数据集。

原因：

1. ODS 数据集是原始数据入口，字段和口径未经清洗；
2. 建模是面向业务主题整合，输入应是已经完成标准化的明细数据；
3. 如果建模直接引用 ODS，会绕过 DWD 清洗层，使数仓分层失效；
4. DWD 单表数据集已经完成字段命名、类型、枚举、口径标准化，更适合作为模型输入；
5. 建模模块的职责是将多个标准明细数据集组合为更高层的 DWS/ADS 消费资产，而不是处理原始数据清洗问题。

推荐链路：

```text
ODS 表 / ODS 数据集
  ↓ 清洗
DWD 表 / DWD 单表数据集
  ↓ 建模
DWS 模型表 / DWS 数据集
  ↓ 报表
```

---

### 14. “单表数据集”的含义

这里的“单表数据集”不是说建模只能选择一个数据集，而是说：

> 建模输入的每一个候选数据集，都应是基于单张 DWD 物理表形成的数据集。

也就是说，建模可以选择多个 DWD 单表数据集进行关联建模，例如：

```text
ds_dwd_employee
ds_dwd_department
ds_dwd_position
  ↓ 建模关联
 ds_dws_headcount_model
```

但不应选择：

```text
ds_ods_employee
```

也不建议在建模输入阶段再选择已经是多表模型结果的数据集，避免模型套模型导致血缘复杂、性能不可控、口径难追踪。

因此，建模输入过滤规则建议为：

```text
warehouse_layer = DWD
asset_type = dataset
build_scope = single_table
reportable 可不限，但 modeling_available = true
```

如果当前系统尚无 `build_scope` / `modeling_available` 字段，可先通过命名、来源关系或 dataset table 数量做过渡判断。

## 补充结论：资产字段管理与数据清洗职责边界

### 15. 当前代码现状

当前系统在“资产管理-字段”模块中已经承载了较多字段配置能力：

- 字段显示名 `column_label`；
- 维度/度量角色 `agg_role`；
- 主键、敏感字段、可见性、复制上月；
- 权限角色、排序、字段描述；
- 后端接口还支持数据类型、枚举选项、计算字段、公式表达式等字段属性。

同时，数据清洗/标准化规则中也存在字段级转换能力，例如：

- 字段重命名；
- 类型转换；
- 枚举映射；
- 输出字段生成。

因此当前存在职责重叠：

```text
资产字段管理：也能改字段展示名、类型、枚举、维度/度量
数据清洗模块：也需要定义字段标准名、类型转换、枚举映射、输出字段
```

---

### 16. 资产管理-字段模块与数据清洗模块可以共存

资产管理-字段模块和清洗模块可以同时存在、都可修改，两者作用对象不同，不冲突：

| 模块 | 作用对象 | 影响范围 |
|---|---|---|
| 资产管理-字段（ODS） | ODS 表/数据集的字段元数据 | ODS 层的展示 |
| 数据清洗模块 | ODS → DWD 的转换规则 | DWD 产物的字段定义 |

清洗模块定义的是"从 A 变成 B"的规则，资产字段模块定义的是"当前这层怎么展示"。如果报表引用 DWD 数据集，它看到的是清洗规则的输出，ODS 资产字段配置不影响它。

因此，清洗模块应新增字段标准化能力（字段重命名、类型转换、枚举映射、输出字段选择/排序），但不需要先从 ODS 资产字段模块移除已有能力。过渡期内两边共存，文档标注清楚：**清洗模块 = DWD 输出的权威来源，资产字段 = ODS 层的辅助展示**。

---

### 17. 维度/度量配置：DWD/DWS 优先，但不强制移除 ODS 已有配置

维度/度量属于分析语义，理想位置是 DWD/DWS 数据集字段，不应在 ODS 原始字段上新增。但 P4 阶段不强制移除 ODS 字段上已有的维度/度量配置，允许两处暂时共存。

规则：

> 新建维度/度量配置应在 DWD/DWS 数据集字段上进行。ODS 字段上已有的历史维度/度量配置暂不删除，但当 DWD/DWS 和 ODS 两处同时配置时，DWD/DWS 数据集字段的配置优先。

原因：

1. ODS 字段尚未标准化，不适合新增分析语义；
2. 但直接移除已有配置会打断现有用户工作流；
3. 共存 + 优先级规则既保证新链路正确，又不破坏历史兼容；
4. 后续根据使用情况逐步引导迁移，最终收敛到 DWD/DWS。

共存规则：

```text
新配置：只在 DWD/DWS 数据集字段上做
旧配置：ODS 字段上已有的保留不动
冲突时：DWD/DWS 优先
```

---

### 18. 资产管理-字段模块：维持现���职责，暂不做减法

资产管理-字段模块现有职责暂不削减。过渡期内，ODS 资产字段页面仍可编辑字段展示名、维度/度量等配置。但文档和 UI 应标注：清洗模块是 DWD 输出的权威来源。

后续（P4-03 完成后）再根据用户使用情况，逐步引导迁移，最终收敛为：

- ODS 字段模块：原始字段查看、来源说明、敏感标记、基础治理
- DWD/DWS 数据集字段：字段展示名、维度/度量、分析语义

不建议在 ODS 字段模块上继续新增维度/度量配置——新增应引导到 DWD/DWS 数据集字段。

---

### 19. 过渡期模块职责分工

| 模块 | 主要职责 | 过渡期可保留 | 最终不应承担 |
|---|---|---|---|
| 资产管理-字段（ODS） | 原始字段查看、来源说明、敏感标记、基础治理 | 字段展示名、维度/度量（旧配置保留） | 新增维度/度量、新增报表口径 |
| 数据清洗模块 | ODS → DWD 的字段重命名、类型转换、枚举映射、输出字段定义 | — | 多表建模、报表指标分析 |
| DWD 数据集字段配置 | 字段展示名、维度/度量、可报表字段、分析语义 | — | 原始字段清洗规则 |
| 数据建模模块 | 多个 DWD 单表数据集关联生成 DWS/DWS 数据集 | — | 直接清洗 ODS、直接选择 ODS |
| 报表模块 | 消费 DWD/DWS 数据集 | — | 直接消费 ODS 数据集或物理表 |

### 20. 一句话补充结论

> 数据建模阶段应选择 DWD 层的单表数据集作为输入，不应选择 ODS 数据集；多个 DWD 单表数据集可以在建模阶段组合生成更高层的数据集。资产管理-字段模块与数据清洗模块可以共存——清洗模块新增字段标准化能力（纯增量），资产字段模块暂不做减法。维度/度量新配置应在 DWD/DWS 数据集字段上进行，ODS 字段上已有配置暂不删除，冲突时 DWD/DWS 优先。后续根据使用情况逐步引导迁移。

## 最终修订口径：拉取 ODS 时自动创建 DWD 表 + DWD 数据集

> 本节为对前文”拉取 ODS 后只创建 ODS 表”方案的修订。后续以本节口径为准。

### 21. 修订原因

经过进一步讨论和实际迁移验证，确认以下事实：

1. 数据量小（最大表 6MB），同结构 DWD 拷贝的存储成本可忽略；
2. 用户拉取数据后期望立即能用，而非必须完成清洗才能建报表；
3. 清洗应作为可选增强，不应成为阻塞门槛；
4. DWD 数据集自始至终是同一份，清洗只提升数据质量，不改变引用关系。

因此，最终修订为：

> 新拉取 ODS 表后，自动创建 DWD 物理表（同结构）和 DWD 数据集。清洗执行时更新 DWD 表内容，不创建新的数据集。

### 22. 修订后的资产生成链路

```text
1. 拉取源数据
   → 创建 ODS 物理表（ods_ 前缀）
   → 自动创建 DWD 物理表（dwd_ 前缀，同结构）
   → 自动创建 DWD 数据集（ds_dwd_ 前缀，指向 DWD 物理表）

2. 下游消费
   → 报表 / 建模可直接使用 DWD 数据集

3. 配置清洗规则（可选）
   → 基于 ODS 表字段配置 ODS → DWD 清洗规则

4. 执行清洗（可选）
   → 更新 DWD 物理表（应用清洗规则）
   → 更新 DWD 数据集字段定义
```

示例：

```text
ods_employee
  ↓ 自动生成
dwd_employee              → 同结构拷贝
ds_dwd_employee           → DWD 数据集，立即可用于报表

  ↓ 配置清洗规则（可选）
dwd_employee              → 字段重命名、类型转换、枚举映射
ds_dwd_employee           → 字段定义自动同步
```

### 23. 数据集概念收敛后的定义

数据集是面向下游消费的数据访问入口，从 DWD 层开始产生：

| 层级 | 是否默认创建数据集 | 说明 |
|---|---|---|
| ODS | 否 | 原始表资产即可承担预览、字段查看、清洗输入 |
| DWD | 是 | 自动随 ODS 拉取创建，清洗前后同一份数据集 |
| DWS | 是 | 建模产物，汇总模型和主题分析的消费入口 |
| ADS | 可选 | 面向具体应用或看板的消费入口 |

### 24. 对历史数据的处理

生产环境中已存在的无前缀表和旧数据集，仍按历史迁移方案处理（已完成）。

迁移原则：
> 补建 DWD 表、创建 DWD 数据集，迁移报表引用，删除旧数据集。

### 25. 新旧规则边界

**新增资产**（从新规则生效后）：

```text
拉取 ODS 表 → ODS 表 + DWD 表 + DWD 数据集（自动）
报表/建模 → 只选择 DWD/DWS 数据集
```

**历史资产**（已完成迁移）：

```text
无前缀表 → 视为 ODS，已补建 dwd_ 表 + ds_dwd_ 数据集
旧数据集 → 已删除，报表已切换到新 DWD 数据集
```

### 26. 更新后的一句话最终结论

> 新架构下，拉取 ODS 表后自动创建 DWD 物理表（同结构）和 DWD 数据集（ds_dwd_ 前缀），立即可用于报表和建模。清洗是可选增强——执行清洗后更新 DWD 表内容和数据集字段定义，不改变数据集引用关系。新建资产强制命名规范：ODS 表 ods_ 前缀，DWD 表 dwd_ 前缀，DWD 数据集 ds_dwd_ 前缀。

# 开发计划：历史迁移优先，随后收口新规则

> 本开发计划参考 `atomic-tasks.md` 的原子任务写法补充。执行顺序特意调整为：先处理生产环境历史报表引用问题，再改新规则。原因是当前工作发生在支线开发中，先把生产历史问题平滑迁移掉，才能降低后续合并主线时对现有报表的影响。

## P0. 开发顺序原则

### P0-01 为什么先做历史迁移

本次方案不先改新建 ODS 资产逻辑，而是先做历史迁移，原因如下：

1. 生产环境已经存在报表引用 ODS 数据集；
2. 如果先禁止 ODS 数据集进入报表/建模，可能导致历史报表不可用；
3. 当前在支线开发，先补齐历史迁移能力，可以在不影响主线现有使用的前提下完成数据修正；
4. 历史迁移完成后，再收口新规则，风险更低；
5. 合并主线时，已有报表已经迁移到 DWD 数据集，不会因为新规则变更而中断。

实施状态：

```text
✅ 第一阶段：P1 历史迁移（已完成 — 生产和开发环境均已迁移完成）
⬜ 第二阶段：P2 新增资产规则收口（拉取 ODS → 自动创建 DWD 表 + DWD 数据集）
⬜ 第三阶段：P3 报表与建模数据源限制（只选 DWD/DWS 数据集）
⬜ 第四阶段（独立排期）：P4 清洗模块字段标准化能力建设
  ↑ P2-P3 为可交付里程碑；P4 解耦，共存策略
```

---

## P4. 清洗模块字段标准化能力建设 ✅（已完成 2026-07-08）

> P4 与 P1-P3 解耦。P1-P3（历史迁移 + 数据源收口）可独立交付验收。P4 是增量增强阶段，不在 P1-P3 的完成路径上。

### 设计原则：共存而非硬迁移

资产管理-字段模块和清洗模块可以同时存在、都可修改，两者作用对象不同，不冲突：

| | 资产管理-字段（ODS） | 数据清洗模块 |
|---|---|---|
| 作用对象 | ODS 表/数据集的字段元数据 | ODS → DWD 的转换规则 |
| 影响范围 | ODS 层的展示 | DWD 产物的字段定义 |

清洗模块定义的是”从 A 变成 B”的规则，资产字段模块定义的是”当前这层怎么展示”。如果报表引用 DWD 数据集，它看到的是清洗规则的输出，ODS 资产字段配置不影响它。

P4 的策略是**在清洗模块新增字段标准化能力（纯增量）**，不要求先从 ODS 资产字段模块移除已有能力。过渡期内两边都可修改，文档标注清楚：**清洗模块 = DWD 输出的权威来源，资产字段 = ODS 层的辅助展示**。

后续（P4-03 之后）再逐步引导用户从资产字段模块迁移到清洗模块，最终在 DWD/DWS 数据集字段上承接维度/度量。
