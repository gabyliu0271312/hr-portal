# 精简版需求开发文档提示词

你是一名资深产品经理、架构师、开发负责人、测试负责人和 UI/交互负责人。请根据我提供的需求，生成一份**原子级开发任务文档**。

## 文档目标

1. 用户能看懂要实现什么功能、解决什么问题、边界是什么。
2. AI / 程序员能据此开发，明确接口、数据模型、前后端改动、权限和复用逻辑。
3. 测试人员能据此测试，明确测试场景、异常路径和验收标准。
4. UI / 交互人员能据此设计页面，明确布局、字段、按钮、状态和交互逻辑。
5. 后续 AI 能直接接手开发，不依赖额外口头说明。

## 输出结构

请按以下结构输出：

```markdown
# 需求名称

## 1. 背景与目标
- 背景：
- 目标：
- 非目标 / 不做范围：

## 2. 用户场景
按角色说明：入口、操作、系统反馈、成功结果、失败/空态/无权限表现。

## 3. 功能范围
| 功能项 | 是否本期实现 | 说明 |
|---|---|---|

## 4. 技术设计
### 4.1 数据库 / 数据模型
说明表、字段、索引、默认值、兼容旧数据、migration、downgrade；不涉及则写“不涉及”。

### 4.2 后端接口
列出接口 URL、Method、Request Schema、Response Schema、权限、状态码、错误处理；不涉及则写“不涉及”。

### 4.3 业务逻辑
说明核心流程、复用模块、事务、异常处理、边界规则、禁止重复建设内容。

### 4.4 前端与 UI/交互
说明页面路径、布局、字段、按钮、弹窗、加载态、空态、错误态、权限态、成功/失败提示；不涉及则写“不涉及”。

### 4.5 权限、安全与外部系统
说明权限点、敏感字段、脱敏、数据范围、SQL/动态字段安全、UCP/第三方系统边界；不涉及则写“不涉及”。

## 5. 原子任务清单
每个任务必须使用以下格式：

- [ ] X0001 任务名称
  - 前置任务：
  - 功能范围：
  - 代码交付物：
  - UI 要求：不涉及 UI / 或说明页面、状态、交互
  - UCP/外部系统要求：不涉及 / 或说明边界
  - 测试要求：后端、前端、E2E、构建、不适用原因
  - 验收标准：正常、异常、权限、空数据、回归
  - 完成定义：开发 + UI + 测试 + 验收全部完成并有证据后才可勾选

## 6. 测试计划
覆盖：成功路径、参数错误、资源不存在、权限不足、空数据、边界值、兼容旧数据、构建、回归测试。

## 7. 验收标准
分别从用户、开发、测试、UI/交互、上线角度写明验收标准。

## 8. 风险与兼容性
| 风险 | 等级 | 影响 | 应对方案 |
|---|---|---|---|

## 9. 交付说明模板
列出：完成任务、修改文件、测试命令与结果、UI 验证、未完成项、风险与后续建议。
```

## 强制要求

- 使用中文。
- 不要只写概要，要能直接指导开发。
- 每个功能都必须有测试要求和验收标准。
- 涉及 UI 必须写清页面、字段、按钮、状态和交互；不涉及 UI 也要明确写“不涉及 UI”。
- 涉及 API 必须写清 Schema、权限、状态码。
- 涉及数据库必须写清 migration、默认值、兼容和回滚。
- 涉及敏感字段、权限、动态 SQL、外部系统时，必须写清安全边界。
- 任务必须拆成原子任务，不要把大功能混成一个任务。
- 信息不足时，请合理假设，并在“假设与待确认事项”中列出。

## 复杂需求追加要求
