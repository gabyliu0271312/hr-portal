# 表格处理工具(表格合并/拆分)— specs/006-excel-table-tools

**本质**:不是"表格合并",而是「多源异构数据 → 按键(姓名+证件)横向归集到标准字段模板」。**核心定位(用户校正,2026-06-20):是「任意多表按键归集的通用组件」,社保只是第一份输入;代码零业务字段,标准字段/映射/派生全是用户配置或AI草稿的产物,加场景=加配置不改代码。** 首期只做合并,拆分留菜单位。

**第一版必须含(用户拍板)**:AI 出初步归集模板→人确认。AI 跨文件聚类列→提议标准字段+各源映射+派生+主键置信度,人审(改字段/审低置信与口径歧义项)后固化模板;之后每月走规则识别零确认。理由:工具交付他人用,别人面对陌生表无法手工配schema,没AI草稿则废。口径歧义(如北京"医疗及生育合并"vs深圳分开)AI必错,必须人兜底。

**Demo已通过(阶段0,见 spec §3.1)**:11文件→识别17/17 100%(纯规则无AI)→归集542人→公积金按`基数×比例`通用表达式拆个人/公司、拆分校验全过→异常0。demo脚本 specs/006/demo_merge.py(注:demo里STD_FIELDS和MAPPINGS写在.py仅为验证引擎,工程化要改成运行时从模板库读)。

**为什么不能纯程序**:列名异构有映射关系、7 种异构表头(两/三行+合并单元格+表头不在第1行)、一文件多 sheet。正确形态 = **AI/规则出映射方案 + 程序确定性执行**(同 specs/004 ChatBI 哲学)。

**真实数据**:`D:\乐逗\Desktop\社保处理文件` 11 个文件 = 7 种结构。只有「日常申报明细×3(A组)」「单笔缴存清单×3(B组)」能同构堆叠,其余(香港强积金/异地代缴社保·公积金·服务费/北京分公司社保·公积金)各自孤立。详见 spec §1.2。

**关键设计**:
- 三层:标准字段字典(纯配置不预设)→ 源映射配置(每模板一份,JSON)→ 确定性引擎
- 归集主键 `merge_keys` 做成配置项(默认 姓名+证件;含"公司"则跨公司分行)
- 金额冲突→报异常,不静默覆盖
- 配置可保存复用(两层:模板库结构级 + 任务方案批次级);周期任务正常**零确认**,靠表头特征自动识别

**AI 分级介入(§4.5)**:① 认模板/定位表头行=规则,不用 AI;② 陌生表语义列映射=用 AI,**仅"无命中模板"时触发一次,人确认后固化进模板库**,第二次同结构走规则。**②第一版必做**(工具交付他人用,别人拿陌生表,没 AI 草稿配不来)。

**AI 基建已就绪,无需新搭**:`app/ai/provider.py::generate_json_openai_compatible()`(OpenAI兼容+强制JSON+畸形修复);`ai_provider_configs` 表存 base_url/api_key_encrypted/model_fast_json(secret_box.decrypt);范式参考 `scripts/gen_field_mapping.py::ai_translate()`。

**授权**:menu_actions 动作字典;`view`(用工具)给一般HR,`manage_template`(改字段/映射)独立给管理员;不做数据范围(用户自己上传文件天然受控)。新菜单走 [[hr_portal_add_menu_sop]]。

**实施**:阶段0 Demo(11文件验证自动识别+按人归集,只读不改,跑通才进工程)→ 1 后端 excel-tools 模块+模板库+②AI草稿 → 2 前端三步向导 → 3 菜单+授权。

**节奏纪律**(用户要求):先文档→后 demo→demo 跑通再落工程。当前:文档已成(spec.md),进入 Demo 阶段。
