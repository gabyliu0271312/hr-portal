L3 字段权限（字段分类 + 角色授权）有两条非直觉但**符合设计**的语义，被问"是不是 bug"时直接引用，不必重查代码。判据见 `permission-model.md` §4.1（已回写）。

## 1. 「敏感」是字段分类管控的总开关

裁决引擎入口 `_table_sensitive_category_map`（`backend/app/permissions/masker.py:114-138`）**只捞 `is_sensitive=True` 分类下的字段**。推论：
- 非敏感分类的字段**完全不进 L3 管控**，对所有有数据访问权的人默认可见（仅受 L2 行级范围约束）。
- 建了非敏感分类、挂了字段、给角色配了可见分类 → 对该分类字段**零效果**，字段照样所有人可见（典型现象：奖金分类未标敏感，未授权用户仍能看到目标奖金）。
- **要让分类授权真正生效，必须勾「敏感」。** 字段分类管理功能 = 敏感字段管控功能。

## 2. 报表含无权敏感字段 → 整张报表 403（不是脱敏/隐藏单列）

报表的列是 admin 建表时**显式指定**的，运行走 `sql_builder.py:1126-1137` 的 `_reject_hidden_ref` 逐列校验：命中无权敏感字段直接抛 `403 无权使用字段: <alias>.<col>`，**整张报表不可用**。
- 与"数据视图自动选列静默丢列"（`sql_builder.py:1138-1144`）不同——报表不能擅删 admin 定的列，故 fail-loud 整张拒绝（方案文档阶段 4d 有意为之：无权字段不进 SELECT、聚合也拒）。
- **连带坑**：admin 把含敏感字段的报表授权给某角色，若该角色无对应敏感分类权限，**该授权实际无效**（用户打开即 403）。

## 待观察项（用户已拍板"先不改"，2026-06-24）

若 admin 频繁踩"授权了却打不开"，优先在**报表授权 UI 加提示**（"本报表含敏感字段 X，被授权角色无对应分类权限将无法查看"），而非改裁决逻辑。
